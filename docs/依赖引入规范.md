# 项目依赖引入规范

## 1. 规范目的

本规范用于约束项目中的外部依赖引入方式，避免在重写式重构过程中重新堆积历史依赖、破坏模块边界或引入不必要复杂度。

本规范服务于以下目标：

- 保持依赖最小化
- 保持模块边界清晰
- 保持依赖归属明确
- 支持渐进式补齐
- 支持长期维护与替换

## 2. 当前总策略

当前项目处于重写式重构阶段，依赖策略采用：

- 先清理到最小可接受基线
- 再按真实需求逐步补齐
- 每次只补一类依赖
- 每次补依赖都要说明归属、用途、边界和替代成本

当前默认基线为：

- Spring Boot 体系依赖
- Lombok

除非有明确实现需求和模块归属，否则不得新增其它外部依赖。

## 3. 总原则

### 3.1 先证明需要，再引入依赖

引入依赖前必须先回答：

- 当前缺的能力是什么
- 这项能力为什么不能先用 JDK / Spring Boot 现有能力实现
- 这项依赖应该放在哪个模块
- 它是固定基础设施，还是可替换外部服务能力

### 3.2 一项依赖只解决一类问题

禁止为了一个很小的问题引入“功能过大”的库。

优先选择：

- JDK 原生能力
- Spring Boot 已提供能力
- 项目内可复用能力

谨慎选择：

- 功能集过大的综合框架
- 会强绑定实现方式的框架
- 容易侵入业务层的依赖

### 3.3 依赖归属先于依赖可用性

不能因为“某模块现在能用”就把依赖放进去。

依赖必须先判断归属，再决定放入哪个模块。

### 3.4 默认不恢复历史依赖

旧项目中出现过，不代表新结构下仍然合理。

历史依赖只有在当前设计下仍然必要、且归属明确时，才允许重新引入。

## 4. 模块归属规则

### 4.1 `infrastructure-basic`

允许放入：

- 全局固定基础设施能力依赖
- 日志
- JSON / 序列化基础能力
- 时间、ID、异常、基础工具等通用支持能力
- 与具体外部服务无关的 Spring Boot 基础扩展

不应放入：

- 数据库驱动
- 缓存客户端
- 邮件客户端
- 对象存储客户端
- MQ 客户端
- 任何可替换外部服务依赖

判断标准：

- 如果未来大概率不会按服务替换，且属于全局通用基础能力，可考虑放入 `infrastructure-basic`

### 4.2 `infrastructure-service`

允许放入：

- 与外部服务交互的依赖
- 可替换实现依赖
- 中间件客户端
- 驱动、SDK、适配器类依赖

规则：

- `*-api` 不应引入具体中间件依赖
- `*-impl` 才能引入具体服务实现依赖

判断标准：

- 只要它明显绑定某类外部系统，就应优先进入 `infrastructure-service/*-impl`

### 4.3 `chat-domain`

允许放入：

- Spring Boot Web / Validation 等直接支撑业务域骨架的依赖
- 真正服务于核心业务表达的最小依赖

谨慎放入：

- 容易把业务逻辑绑死在某个技术框架上的库

禁止放入：

- 具体数据库实现依赖
- 具体缓存实现依赖
- 具体邮件实现依赖
- 具体对象存储实现依赖
- 任何 `*-impl` 才应持有的依赖

### 4.4 `application-starter`

允许放入：

- 启动与装配所需依赖
- 运行时选择具体实现所需依赖

不应放入：

- 核心业务规则专属依赖
- 与启动无关的实现性工具依赖

## 5. POM 放置规则

### 5.1 根 `pom.xml`

适合放：

- 统一版本管理
- 所有模块共用且方向明确的依赖版本
- 插件版本
- 依赖管理规则

不适合放：

- 只属于某个具体模块的实现依赖

### 5.2 子模块 `pom.xml`

适合放：

- 该模块实际使用的直接依赖
- 模块专属依赖

规则：

- 依赖加在“真正使用它的模块”
- 不因为转递方便就随意加在上游模块

### 5.3 `dependencyManagement` 与 `dependencies`

规则：

- 版本统一管理优先放 `dependencyManagement`
- 实际使用位置必须在对应模块的 `dependencies`
- 不允许只在根工程声明版本就默认视为该模块可以使用

## 6. 渐进式补齐流程

每次新增依赖都按以下顺序执行：

1. 明确当前缺失能力
2. 明确依赖应归属的模块
3. 只引入满足当前最小需求的依赖
4. 补充必要注释或文档说明
5. 做最小验证
6. 记录是否为后续可替换点

禁止一次性引入一组“以后可能会用到”的依赖。

## 7. 引入依赖前的检查清单

新增依赖前，必须至少检查以下问题：

- 是否能先用 JDK 或 Spring Boot 现有能力解决
- 是否真的已有代码要使用该依赖
- 是否已经明确模块归属
- 是否会破坏既定模块边界
- 是否会把 `chat-domain` 绑定到具体实现
- 是否属于短期临时方案
- 是否需要同步新增配置
- 是否需要同步新增注释或文档

## 8. 引入依赖后的补充要求

新增依赖后，视情况同步补充：

- 对应模块注释
- 配置说明
- 架构文档
- 迁移说明
- 删除旧依赖计划

如果该依赖属于可替换外部服务能力，还应明确：

- 未来 API / Impl 分层位置
- 替换点在哪一层
- 启动层如何装配

## 9. 禁止事项

禁止以下行为：

- 因为以前项目用过就直接恢复依赖
- 因为“以后可能会用”而预先引入依赖
- 把具体实现依赖直接放进 `chat-domain`
- 把外部服务依赖放进 `infrastructure-basic`
- 把只在一个模块使用的依赖塞进所有模块共享位置
- 引入依赖但不说明用途和归属
- 引入依赖后不清理旧方案残留配置

## 10. 常见判断示例

### 10.1 日志相关

- 应放入 `infrastructure-basic`
- 原因：属于全局固定基础设施能力

### 10.2 Jackson 全局序列化能力

- 应放入 `infrastructure-basic`
- 原因：属于全局统一规则，不绑定具体外部服务

### 10.3 MySQL / Redis / MinIO / 邮件客户端

- 应放入 `infrastructure-service/*-impl`
- 原因：属于具体外部服务接入能力，应保留替换空间

### 10.4 业务控制器直接需要的 Spring MVC / Validation

- 可放入 `chat-domain`
- 原因：属于当前业务域骨架的直接支撑依赖

### 10.5 历史编排框架类依赖

- 默认不恢复
- 只有在当前架构下确认仍然需要时才允许重新引入

## 11. 最终判断标准

一个依赖引入是否合格，以以下标准判断：

- 当前是否真的需要
- 模块归属是否正确
- 是否保持了架构边界
- 是否仍然符合最小依赖原则
- 未来是否容易替换或移除
