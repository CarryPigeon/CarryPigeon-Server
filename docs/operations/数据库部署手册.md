# 数据库部署手册

## 1. 文档目的

本文面向 CarryPigeon Backend 的开源使用者，说明当前仓库对外保留的数据库初始化方式、结构脚本入口和测试数据脚本用法。

当前数据库事实来源：

- 运行配置：`application-starter/src/main/resources/application.yaml`
- 结构 SQL：`docs/sql/00-all-in-one.sql`
- 领域 SQL：`docs/sql/01-auth.sql` 到 `docs/sql/05-notification.sql`
- 测试数据 SQL：`docs/sql/10-test-data.sql`

当前仓库已经不再保留 `db/migration` 版本迁移目录，数据库初始化以 `docs/sql/` 下的最终 SQL 为准。

## 2. 当前数据库方案

当前项目使用：

- MySQL 8.x
- Spring Boot DataSource
- 手工执行的整库 SQL 或领域 SQL

推荐基线与仓库默认值：

- 推荐镜像：`mysql:8.4`
- 默认数据库名：`carrypigeon`
- 默认字符集：`utf8mb4`
- 默认排序规则：`utf8mb4_unicode_ci`
- 推荐时区：`Asia/Shanghai` 或 `+08:00`

项目默认 JDBC 地址：

```text
jdbc:mysql://127.0.0.1:3306/carrypigeon?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai
```

## 3. SQL 入口说明

当前对外提供以下 SQL 文件：

- `docs/sql/00-all-in-one.sql`
  - 一次性初始化整库结构的最短路径
- `docs/sql/01-auth.sql`
  - 认证与 refresh session 结构
- `docs/sql/02-user.sql`
  - 用户资料结构
- `docs/sql/03-channel.sql`
  - 频道、成员、邀请、治理、已读、置顶结构与系统种子频道
- `docs/sql/04-message.sql`
  - 消息与 mention 结构
- `docs/sql/05-notification.sql`
  - 通知偏好结构
- `docs/sql/10-test-data.sql`
  - 本地开发和联调用测试数据

推荐规则如下：

1. 新库初始化时，优先执行 `00-all-in-one.sql`
2. 如果你想按领域阅读和分步执行，再按 `01` 到 `05` 的顺序执行
3. 如果你还需要现成账号、频道、消息和通知数据，再执行 `10-test-data.sql`

## 4. 前置条件

在执行数据库初始化前，应确保：

1. MySQL 实例可连接
2. 目标数据库已经创建，或当前账号有创建数据库权限
3. 当前账号具备建表、建索引、建外键和写入测试数据的权限
4. 应用运行参数或环境变量已经提供正确的 `MYSQL_USERNAME` 和 `MYSQL_PASSWORD`

若使用仓库自带 Docker Compose，默认外部依赖启动命令为：

```bash
bash bin/linux/docker-up.sh
```

## 5. 数据库连接配置

当前仓库中与数据库直接相关的默认配置如下：

```yaml
spring:
  datasource:
    url: jdbc:mysql://127.0.0.1:3306/carrypigeon?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai
    username: ${MYSQL_USERNAME:}
    password: ${MYSQL_PASSWORD:}
    driver-class-name: com.mysql.cj.jdbc.Driver
```

含义如下：

- 应用负责连接已经准备好的 MySQL schema
- 仓库默认不再在启动时自动扫描迁移目录
- 数据库结构初始化应在应用启动前完成

## 6. 部署步骤

### 6.1 使用仓库默认 Docker MySQL

1. 启动外部依赖：

```bash
bash bin/linux/docker-up.sh
```

3. 初始化数据库结构：

```bash
mysql -h 127.0.0.1 -P 3306 -u"${MYSQL_USERNAME}" -p"${MYSQL_PASSWORD}" carrypigeon < docs/sql/00-all-in-one.sql
```

4. 如需测试数据，再执行：

```bash
mysql -h 127.0.0.1 -P 3306 -u"${MYSQL_USERNAME}" -p"${MYSQL_PASSWORD}" carrypigeon < docs/sql/10-test-data.sql
```

5. 启动应用，命令请参考 `docs/operations/启动脚本索引.md` 或 `docs/operations/部署手册.md`

### 6.2 使用你自己的 MySQL

1. 创建数据库，例如：

```sql
CREATE DATABASE carrypigeon
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;
```

2. 使用具备建表与写入权限的账号执行结构脚本
3. 按需执行测试数据脚本
4. 覆盖应用中的数据库连接配置
5. 启动应用

### 6.3 按领域分步执行

如果你希望按领域理解和初始化数据库，可按以下顺序执行：

1. `docs/sql/01-auth.sql`
2. `docs/sql/02-user.sql`
3. `docs/sql/03-channel.sql`
4. `docs/sql/04-message.sql`
5. `docs/sql/05-notification.sql`
6. 可选：`docs/sql/10-test-data.sql`

该路径只适用于空库初始化。

## 8. 初始化后的关键事实

### 8.1 固定种子频道

结构脚本会写入两条基础频道数据：

- `public`
  - 固定 `id = 1`
  - `type = 'public'`
  - `is_default = true`
- `system`
  - 固定 `id = 2`
  - `type = 'system'`
  - `is_default = false`

### 8.2 关键完整性约束

当前结构已覆盖以下关键数据库约束：

- `auth_refresh_session.account_id -> auth_account.id`
- `user_profile.account_id -> auth_account.id`
- `chat_channel_member.channel_id -> chat_channel.id`
- `chat_channel_member.account_id -> auth_account.id`
- `chat_message.channel_id -> chat_channel.id`
- `chat_message.sender_id -> auth_account.id`
- `chat_channel_read_state.channel_id -> chat_channel.id`
- `chat_channel_read_state.account_id -> auth_account.id`
- `chat_notification_server_preference.account_id -> auth_account.id`
- `chat_notification_channel_preference.account_id -> auth_account.id`
- `chat_notification_channel_preference.channel_id -> chat_channel.id`

此外，结构脚本会写入一条固定 `type = 'system'` 的频道记录；当前未建立数据库级唯一约束来硬性防止多条 `type = 'system'` 记录。

### 8.3 查询导向索引

当前结构已包含主要查询索引，例如：

- `idx_chat_message_channel_message`
- `idx_chat_message_channel_sender_message_id`
- `idx_chat_message_channel_type_message_id`
- `idx_chat_channel_read_state_account_id`
- `idx_chat_channel_pin_channel_id_pinned_at`
- `idx_chat_mention_target_account_read_mention_id`

## 9. 测试数据说明

`docs/sql/10-test-data.sql` 会写入以下内容：

- 3 个测试账号
- 2 个额外测试频道
- 成员关系、邀请、封禁、审计日志
- 频道消息、mention、置顶、已读状态
- 服务级与频道级通知偏好

测试账号如下：

- `carry-owner` / `carrypigeon123`
- `carry-admin` / `carry-admin-123`
- `carry-member` / `carry-member-123`

该脚本设计为可重复执行：

- 会先清理自身使用的固定测试主键
- 不会删除 `public` 与 `system` 两条种子频道

## 10. 部署后校验

初始化完成后，建议至少执行以下检查。

### 10.1 检查核心表

```sql
SHOW TABLES;
```

### 10.2 检查固定频道与测试频道

```sql
SELECT id, name, type, is_default
FROM chat_channel
ORDER BY id;
```

若同时执行了 `10-test-data.sql`，预期至少能看到：

- `public`
- `system`
- `project-alpha`
- `frontend-lab`

### 10.3 检查 system channel 唯一性

```sql
SELECT COUNT(*)
FROM chat_channel
WHERE type = 'system';
```

预期结果：

```text
1
```

### 10.4 检查测试账号

```sql
SELECT id, username
FROM auth_account
WHERE id IN (1001, 1002, 1003)
ORDER BY id;
```

## 11. 维护建议

1. 当前仓库以 `docs/sql/` 下的最终 SQL 为准，不要再依赖已移除的 `db/migration` 目录
2. 结构发生变化时，应同步更新对应领域 SQL 和整库 SQL
3. 测试数据脚本只用于本地开发、演示和联调，不建议直接用于生产环境
4. 如果你的分支增加了新的表或索引，应同步更新本手册与 `docs/sql/README.md`
