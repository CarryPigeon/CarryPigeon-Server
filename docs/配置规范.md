# 项目配置规范

## 1. 文档目的

本文用于统一项目中的配置设计、配置落点与配置命名规则。

本文只固化当前重写阶段已经明确的配置约束，不扩展未确认的配置体系设计。

## 2. 总原则

- 配置只表达运行时差异，不表达业务流程
- 配置归属先于配置可用性
- 最小配置优先，不预埋“以后可能会用”的配置项
- 固定基础设施配置归 `infrastructure-basic`
- 可替换外部服务配置归对应 `infrastructure-service/*-impl`
- 最终运行配置统一由 `application-starter` 承接

## 3. 配置文件落点规则

### 3.1 `application-starter`

`application-starter/src/main/resources/application.yaml` 是当前项目默认配置入口。

适合放入：

- Spring Boot 启动配置
- 当前运行时应用名
- 已接入基础设施的最小必需配置
- 各模块最终装配后实际生效的运行配置

不适合放入：

- 还未接入的未来配置
- 纯注释性示例配置
- 某个模块内部未启用的占位配置

### 3.2 `infrastructure-basic`

`infrastructure-basic` 可以定义：

- 配置前缀常量
- `@ConfigurationProperties` 配置类
- 配置校验工具
- 自动配置逻辑

但不负责承载最终运行环境下的完整 `application.yaml`。

### 3.3 `infrastructure-service`

后续外部服务配置应放入对应服务实现模块的配置类中，例如：

- 数据库配置归数据库 `*-impl`
- 缓存配置归缓存 `*-impl`
- 邮件配置归邮件 `*-impl`

这些配置最终仍由 `application-starter` 的运行配置文件提供值。

## 4. 配置前缀规则

### 4.1 项目前缀

项目自定义配置统一使用：

- `cp`

当前已存在的前缀常量位于：

- `team.carrypigeon.backend.infrastructure.basic.config.ConfigurationPrefix`

### 4.2 分层前缀

推荐按“项目前缀 + 基建归属 + 能力名”组织配置前缀，例如：

- `cp.infrastructure.id`

原则：

- 前缀必须体现归属
- 前缀不允许过短到失去语义
- 前缀不允许跨模块混用同一语义空间

### 4.3 命名示例

当前正确示例：

- `cp.infrastructure.id.worker-id`
- `cp.infrastructure.id.datacenter-id`

## 5. 配置项命名规则

- 配置项统一使用小写加中划线
- 配置名应表达业务或基础设施语义，不使用随意缩写
- 布尔配置应能直接表达开关语义
- 数值配置应在注释或配置类中说明取值约束

推荐：

- `worker-id`
- `datacenter-id`

不推荐：

- `workerId`
- `dcid`
- `id1`

## 6. 配置类规则

### 6.1 适用场景

只要某组配置具有稳定语义、需要统一校验或会被多处使用，就应定义配置类。

当前示例：

- `SnowflakeIdProperties`

### 6.2 设计规则

- 使用 `@ConfigurationProperties` 绑定稳定配置组
- 配置类名应直接表达语义
- 配置类应只承载配置语义，不承载业务逻辑
- 配置校验应尽量前置到绑定阶段

### 6.3 校验规则

- 能在配置绑定阶段校验的，就不要拖到运行时深处才失败
- 校验信息必须可读，能直接指出字段与约束
- 通用基础校验可复用 `ConfigurationSupport`

当前示例：

- `workerId` 与 `datacenterId` 必须在 `0..31`

## 7. 默认值规则

- 默认值只保留当前最小可运行所必需的部分
- 默认值必须合理、可解释、可替换
- 默认值不能掩盖明显配置错误

当前示例：

- `worker-id: 1`
- `datacenter-id: 1`

约束：

- 单机开发可使用默认值
- 多节点部署时必须显式区分节点配置

## 8. 注释规则

配置文件中的注释至少应说明以下信息中的必要部分：

- 当前配置项的职责
- 当前配置项的边界
- 当前配置项的取值约束
- 当前配置项在多环境或多节点下的注意事项

要求：

- 注释应直接服务理解与维护
- 不写空话式注释
- 注释内容与配置语义必须一致

## 9. 当前最小配置基线

当前项目保留的最小配置如下：

```yaml
spring:
  application:
    name: CarryPigeonBackend

cp:
  infrastructure:
    id:
      worker-id: 1
      datacenter-id: 1
```

说明：

- `spring.application.name` 为运行时应用名
- `cp.infrastructure.id.*` 为全局雪花 ID 配置

## 10. 禁止事项

- 禁止在配置文件中提前堆积未来能力的占位配置
- 禁止把外部服务实现配置塞进 `infrastructure-basic`
- 禁止在业务代码中硬编码应由配置提供的运行时参数
- 禁止同一语义出现多套命名方式
- 禁止配置类承载业务处理逻辑
- 禁止只加配置项、不补注释与约束说明

## 11. 后续扩展规则

后续新增配置时，必须先回答：

1. 这项配置属于哪个模块
2. 这项配置是固定基础设施还是可替换外部服务
3. 这项配置是否真的已经被当前代码使用
4. 这项配置是否需要配置类、默认值与校验
5. 这项配置是否需要同步补充文档与注释

后续数据库、缓存、邮件等配置，应进入对应的 `infrastructure-service/*-impl` 配置类，而不是直接堆入全局基础设施配置空间。
