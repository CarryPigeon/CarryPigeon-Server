# 项目测试规范

## 1. 文档目的

本文用于统一项目中的测试分层、测试命名、测试注释与测试边界。

本文只固化当前重写阶段已经明确的测试约束，不扩展未确认的复杂测试体系。

## 2. 总原则

- 测试服务于稳定重构，不服务于形式化堆数量
- 测试优先覆盖稳定契约，不优先覆盖易碎实现细节
- 测试边界必须和模块边界一致
- 每个测试都应回答“当前在验证什么契约”
- 测试代码同样必须可读、可维护、可长期迭代

## 3. 测试目录规则

- 测试代码统一放在各模块自己的 `src/test/java`
- 测试文件应与被测包结构尽量保持邻近
- 不允许把一个模块的测试放到另一个模块中

当前示例：

- `infrastructure-basic/src/test/java/team/carrypigeon/backend/infrastructure/basic/...`

## 4. 测试命名规则

### 4.1 测试类命名

测试类统一使用：

- `<Name>Tests`

当前示例：

- `JsonsTests`
- `InfrastructureBasicsTests`
- `SnowflakeIdGeneratorTests`

### 4.2 测试方法命名

测试方法统一使用：

- `methodName_condition_expected()`

要求：

- 方法名必须能直接看出被测行为、条件和预期
- 禁止使用无语义命名，例如 `test1`、`shouldWork`

当前示例：

- `toJson_map_returnsJsonString()`
- `fromJson_invalidJson_throwsInfrastructureException()`

## 5. 测试分层规则

### 5.1 `infrastructure-basic`

`infrastructure-basic` 重点验证固定基础设施契约。

至少应覆盖：

- 配置绑定与配置校验
- ID 生成能力
- JSON 序列化与反序列化
- 时间能力
- 日志上下文工具
- 基础异常与错误码
- 对外门面聚合行为

原则：

- 重点测“项目封装后的契约”
- 不对第三方框架内部实现做重复性测试

### 5.2 `chat-domain`

`chat-domain` 重点验证业务语义与业务边界。

优先覆盖：

- 应用层用例编排
- 领域规则
- 仓储抽象交互
- 协议入口的基本输入输出约束
- 业务异常与响应码映射

原则：

- 测业务语义，不测无意义框架细节
- 测成功路径，也测失败路径

### 5.3 `application-starter`

`application-starter` 重点验证启动与装配。

适合覆盖：

- Spring 容器能否启动
- 自动配置是否生效
- 关键基础 Bean 是否可装配

不适合覆盖：

- 大量业务规则细节
- 重复验证下游模块已经覆盖的契约

### 5.4 `infrastructure-service`

后续外部服务模块建立后：

- `*-api` 优先验证契约稳定性
- `*-impl` 优先验证具体实现行为、失败处理与适配边界

## 6. 当前阶段推荐测试类型

当前重写阶段优先三类测试：

- 单元测试
- 模块内契约测试
- 最小启动测试

当前不建议过早铺开：

- 大量端到端测试
- 大量依赖真实外部服务的集成测试
- 为未来功能预写的大量占位测试

## 7. 断言规则

- 断言必须直接对应测试目标
- 一个测试只聚焦一个主要契约
- 失败断言必须验证失败语义，而不只是“抛了异常”

推荐断言：

- 返回值是否符合预期
- 错误码是否正确
- 异常类型是否正确
- 原始 `cause` 是否保留
- 响应码是否符合统一约定

不推荐：

- 只断言方法执行过
- 只断言对象非空但不验证语义

## 8. 响应码测试规则

业务测试必须覆盖成功与失败路径。

当前对外响应约定中，至少应覆盖以下 `CPResponse.code`：

- `100`
- `200`
- `300`
- `404`
- `500`

要求：

- 成功场景验证成功响应码
- 参数错误场景验证参数类失败码
- 权限失败场景验证权限类失败码
- 资源不存在场景验证不存在类失败码
- 系统异常场景验证内部错误码

## 9. 异常测试规则

异常相关测试至少应验证：

- 抛出的异常类型
- 错误码是否正确
- 错误消息是否可读
- 原始 `cause` 是否保留

当前基础设施异常测试应重点覆盖：

- `InfrastructureException`
- `InfrastructureErrorCode`

## 10. 测试数据规则

- 测试数据应最小化，只保留当前场景需要的输入
- 测试数据命名应直接表达语义
- 不使用含糊的魔法值堆砌测试场景
- 能直接内联表达的简单数据，不必额外引入复杂夹具

推荐：

- `Map.of("name", "carry")`
- `workerId = 1`

不推荐：

- 大量无语义字符串
- 与当前断言无关的大对象初始化

## 11. Mock 与替身规则

- 只有在隔离边界、降低外部依赖时才使用 Mock
- Mock 应服务于验证当前模块契约，不应用来掩盖设计问题
- 能使用简单假对象或最小真实对象时，优先不用复杂 Mock

原则：

- `chat-domain` 测仓储抽象与外部服务抽象时可以使用替身
- `infrastructure-basic` 多数场景优先使用真实基础对象

## 12. 注释规则

测试代码必须遵守 `docs/注释规范.md`。

测试类必须说明：

- 被测对象
- 验证的核心契约
- 不验证什么

测试方法必须说明：

- 当前测的是什么
- 输入条件是什么
- 期望结果是什么

禁止：

- 只有测试代码没有注释说明
- 只有“测试某方法”这类无信息注释

## 13. 禁止事项

- 禁止为追求覆盖率编写无语义测试
- 禁止把多个核心场景硬塞进一个测试方法
- 禁止只测成功不测失败
- 禁止测试方法命名含糊
- 禁止测试代码不遵守注释规范
- 禁止把第三方框架内部行为当作项目核心契约反复测试

## 14. 当前阶段结论

当前项目测试规范的目标不是一次性铺满所有层，而是先守住重写阶段最关键的稳定面：

- 固定基础设施契约稳定
- 业务核心路径可验证
- 响应码与异常语义不漂移
- 启动装配至少可最小验证

后续随着 `chat-domain` 和 `infrastructure-service` 逐步落地，再按本文规则继续增量补齐测试即可。
