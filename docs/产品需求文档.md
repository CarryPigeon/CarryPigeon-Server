# CarryPigeon 产品需求文档

## 1. 文档目的

本文用于定义 CarryPigeon 的正式产品需求边界，作为后续模块设计、接口设计、实现排期和验收的产品依据。

本文重点解决以下问题：

- CarryPigeon 到底是什么产品
- 当前阶段只做什么，不做什么
- 服务端需要提供哪些稳定能力
- 为什么需要插件、JWT 鉴权、通用消息模型和多服务端协议边界
- V0、V1、V2 的演进目标分别是什么

## 2. 产品定义

### 2.1 产品名称

- 产品名：CarryPigeon
- 产品形态：面向特定场景的分布式聊天服务平台

### 2.2 产品定位

CarryPigeon 不是一个以通用社交为目标、直接与 QQ、微信竞争的即时通讯产品。

CarryPigeon 的目标是提供一套面向垂直场景的聊天服务底座，让服务提供者无需从 0 开发聊天内核，也无需大规模改造现有开源 IM 项目，就能快速构建特定场景下的小型聊天服务。

CarryPigeon 的核心价值不在于“让更多人聊天”，而在于：

- 让特定场景快速拥有聊天能力
- 让聊天流与业务流自然融合
- 让场景能力通过插件扩展，而不是侵入聊天内核
- 让部署、鉴权、消息、权限、扩展和接入成本尽量可控

一句话定义：

> CarryPigeon 是一个面向垂直场景、支持插件化扩展的分布式聊天服务平台。

## 3. 产品目标与非目标

### 3.1 当前目标

当前阶段的核心目标是交付一个稳定、可扩展、明确边界的服务端产品基础，使其可以承载小型、特定场景的聊天服务。

当前目标包括：

- 提供服务端注册、登录和鉴权主链路
- 提供群聊频道、小型群聊和实时消息能力
- 提供可扩展的通用消息模型
- 提供插件接口和事件边界
- 提供通过 URL 添加服务端所需的公开源信息
- 为后续 MC 场景插件等垂直能力预留清晰接口

### 3.2 明确非目标

当前阶段明确不做：

- 通用大型社交平台
- 1 对 1 私聊
- 客户端 UI 实现
- 客户端插件系统
- 跨服务端联邦和默认互通
- 全局账号体系
- 音视频通话
- 插件市场
- 完整商业化能力
- 端到端加密
- 企业级 SSO
- 大规模分布式调度和多数据中心容灾

## 5. 典型应用场景

### 5.1 MC 服务器社区

这是 V1 优先验证的目标场景。

目标：

- 将游戏内状态、游戏角色和群聊行为融合

典型能力：

- 展示服务器状态和在线玩家信息
- 游戏消息与聊天群消息桥接
- 保留真实映射身份，而不是简单机器人代发
- 区分聊天管理员身份与服务器 OP 身份
- 通过按钮或结构化消息承载管理动作

说明：

- 当前阶段先建立插件接口和消息模型
- 真正的 MC 插件功能放在后续阶段实现，不进入当前核心后端主链路

### 5.2 诗词歌赋或长文讨论社区

目标：

- 让高质量内容在群聊中被沉淀、检索和互动

潜在能力：

- 长文格式化预览
- 内容归档与检索
- 评论、评分、收藏和专题整理

### 5.3 学校或班级管理场景

目标：

- 把群聊与文件、通知、班级管理动作结合

潜在能力：

- 老师文件自动分类
- 作业、试卷和教案整理
- 聊天内触发远程操作

### 5.4 开发团队协作群

目标：

- 把项目状态和研发事件整合到聊天流中

潜在能力：

- GitHub、CI、发布状态集成
- 项目进度提醒
- 结构化状态卡片

## 6. 产品基本原则

1. 平台优先
- 核心服务端提供通用聊天底座，不做单一场景定制品。

2. 插件优先
- 场景能力应尽量通过插件实现，而不是直接写入聊天内核。

3. 服务端边界清晰
- 当前阶段只做服务端；客户端实现与客户端插件不纳入当前开发范围。

4. 默认隔离
- 服务端之间默认不共享账号、权限、插件和数据。

5. 默认安全
- 鉴权、权限、审计、限流和插件能力约束必须前置考虑。

6. 小型特定服务
- 单个服务实例面向小型、明确场景用户群，而不是超大规模全民社交网络。

## 7. 服务端范围边界

### 7.1 当前服务端负责

- 服务端实例公开信息
- 用户注册、登录、JWT 鉴权与刷新
- 群聊频道与小型群聊
- 实时消息收发
- 历史消息查询
- 在线状态基础维护
- 通用消息模型
- 插件接口、事件、权限声明和运行边界

### 7.2 当前服务端不负责

- 客户端本地 token 保存
- 客户端通知聚合
- 客户端多服务端界面管理
- 客户端插件渲染
- 跨服务端账号打通
- 默认跨服务端消息同步

## 8. 多服务端协议边界

### 8.1 核心原则

一个客户端可以同时连接多个独立服务端，但 CarryPigeon 当前只负责服务端侧边界定义。

服务端必须保证：

- 本服务端身份只在本服务端内有效
- 本服务端签发的 token 只在本服务端有效
- 本服务端的消息、权限、插件和数据不默认跨服务端共享
- 所有协议返回内容都能明确服务端来源

### 8.2 URL 优先添加服务端

服务端添加方式优先使用 URL。

标准流程：

1. 客户端输入服务端 URL
2. 客户端读取 `/.well-known/carrypigeon-server`
3. 服务端返回公开源信息
4. 用户决定是否注册或登录该服务端

### 8.3 公开源信息接口

服务端必须提供：

- `GET /.well-known/carrypigeon-server`

接口要求：

- 匿名可读
- 可通过限流保护
- 只暴露公开摘要，不泄露敏感信息

建议字段：

- `server_id`
- `server_name`
- `description`
- `icon_url`
- `public_base_url`
- `realtime_endpoint`
- `register_enabled`
- `login_methods`
- `server_version`
- `public_plugins`
- `public_capabilities`

明确不公开：

- 管理员列表
- 密钥和私有配置
- 内部网络地址
- 详细运行状态
- 成员数与在线数

## 9. MVP 范围

### 9.1 MVP 总目标

MVP 的目标是交付一个可运行、可验证、可继续扩展的最小服务端产品闭环。

最小闭环定义：

- 服务端可以启动并接受连接
- 匿名访问者可以读取服务端公开源信息
- 用户可以注册、登录并通过 JWT 完成鉴权
- 用户可以加入默认群聊或小型群聊频道并发送消息
- 实时通道可以推送消息
- HTTP 接口可以查询基础数据
- 消息存储采用通用字段，支持后续文件消息和插件化消息扩展
- 插件系统至少具备接口、事件和权限声明边界

### 9.2 MVP 非目标

- 私聊
- 客户端开发
- 客户端插件体系
- 跨服务端联邦
- 全局账号
- 插件市场
- 外部插件动态加载
- 音视频通话
- 高级搜索
- 完整后台管理系统

说明：

- 强化分布式部署和运维能力

## 11. 聊天核心能力

### 11.1 会话与频道模型

当前阶段只支持群聊频道与小型群聊。

频道类型：

- `public`
- `private`
- `system`

约束：

- 不支持 1 对 1 私聊
- 小型群聊通过 `private channel` 表达

### 11.1.1 当前频道治理规则

当前已稳定的最小治理规则：

- `public` 频道保留较保守的公共讨论语义，不强制引入完整 owner/admin 治理流程。
- `private` 频道作为小型群聊的主要治理载体。
- `system` 频道不进入普通用户群治理流程。

`private` 频道当前固定角色：

- `OWNER`
- `ADMIN`
- `MEMBER`

当前固定边界：

- 不引入可配置 ACL / RBAC 矩阵。
- 频道活跃成员通过 `ChannelMember` 表达。
- invite / ban / audit 通过独立记录表达，不与活跃成员投影混为单一状态机。
- 创建 `private` 频道的用户成为该频道初始 `OWNER`。
- `OWNER` 可以提升 / 降级管理员、转移所有权。
- `OWNER / ADMIN` 可以对 `MEMBER` 执行禁言、移除、封禁等治理动作。
- `ADMIN` 不能治理 `OWNER`。

### 11.2 核心对象

服务端核心对象包括：

- `ServerInstance`
- `User`
- `AccountCredential`
- `Channel`
- `ChannelMember`
- `Conversation`
- `Message`
- `RealtimeSession`

### 11.3 通用消息模型

消息必须使用通用字段，而不是为不同消息类型拆出完全独立的数据结构。

关键字段：

- `message_id`
- `server_id`
- `conversation_id`
- `channel_id`
- `sender_id`
- `message_type`
- `content`
- `payload`
- `metadata`
- `created_at`
- `status`

字段定位：

- `message_type`：表示消息总类型
- `content`：承载简单文本或摘要
- `payload`：承载结构化消息主体
- `metadata`：承载来源、展示、追踪和插件扩展信息

设计原因：

- 文件消息进入 V1
- 插件化自定义消息是核心扩展能力
- 数据库字段不应被单一消息形态绑定

### 11.4 消息类型

当前建议支持：

- `text`
- `file`
- `system`
- `plugin`
- `custom`

### 11.5 历史与实时一致性

必须满足：

- 实时推送中的消息 ID 与历史查询中的消息 ID 一致
- 历史消息可按频道分页查询
- 分页优先使用 cursor，而不是 offset 深分页

### 11.6 消息撤回规则

当前已稳定的消息撤回规则：

- 消息撤回采用软撤回，而不是硬删除。
- 被撤回消息必须保留原 `message_id`，以保持历史、分页与实时更新的一致性。
- 被撤回消息的对外展示应使用稳定占位文案，而不是继续暴露原正文。
- 被撤回消息不再参与关键词搜索命中。
- 撤回动作属于频道治理行为，必须有审计留痕。
- 当前默认规则下，发送者可以撤回自己的消息；`private` 频道内具有治理权限的角色可以按既定权限矩阵撤回他人消息。
- 实时链路需要向在线成员下发消息更新事件，使撤回状态在历史与实时之间保持一致。

## 12. 注册、登录与鉴权

### 12.1 基本要求

开发期也必须提供完整注册、登录和鉴权主链路，不允许仅用简化 dev token 替代正式方案。

### 12.2 注册方式

当前阶段采用：

- 用户名 + 密码注册

当前阶段不采用：

- 邮箱验证码注册
- 手机验证码注册
- 第三方 OAuth 登录

### 12.3 token 方案

当前正式结论：

- 使用 JWT
- 使用 `access token + refresh token`
- token 只在当前服务端内有效
- token 不能跨服务端复用

建议生命周期：

- `access token`：30 分钟
- `refresh token`：14 天

这样既能控制泄露风险，也能避免客户端频繁重新登录。

### 12.4 密码安全策略

当前安全要求：

- 密码只保存摘要，不保存明文
- 应采用专业密码哈希方案

当前推荐方案：

- 首选 `Argon2id`

说明：

- `Argon2id` 是当前更现代的密码哈希方案，抗 GPU 暴力破解能力更强，更适合新的认证系统
- 若实现阶段依赖审批或运行环境受限，再明确是否采用兼容替代方案

### 12.5 鉴权边界

匿名可访问：

- 服务端公开源信息接口
- 注册接口
- 登录接口
- refresh 接口

需要鉴权：

- 业务 HTTP 接口
- 实时消息收发
- 历史消息查询
- 频道管理动作

### 12.6 实时连接鉴权

实时连接必须显式完成 token 认证。

约束：

- 未认证连接不能发送业务消息
- 认证成功后需绑定用户身份
- 心跳和断线清理需要纳入实时会话管理

## 13. 插件系统边界

### 13.1 插件系统目标

插件系统的目标是让聊天服务提供者在不修改聊天内核的前提下扩展垂直业务能力。

核心目标：

- 支持垂直场景扩展
- 保护聊天内核稳定
- 控制插件权限
- 记录插件行为
- 为后续 MC 插件等场景插件提供运行基础

### 13.2 插件分层

1. 内核能力层
- 不是插件，承载用户、频道、消息、权限、实时连接与鉴权等基础能力。

2. 平台通用插件层
- 提供跨场景可复用能力，如通知、搜索、归档、提醒。

3. 场景业务插件层
- 面向某一垂直场景提供深度业务扩展，如 MC 插件、班级管理插件等。

### 13.3 当前阶段插件范围

V0 只要求：

- 插件描述模型
- 插件事件模型
- 插件权限声明
- 插件运行边界
- 插件消息输出边界

当前阶段不要求：

- 实际动态加载外部插件包
- 插件市场
- 客户端插件
- 完整场景插件实现
- 插件业务功能落地

### 13.4 插件权限原则

- 默认无权限
- 显式声明
- 管理员授权
- 最小权限
- 可审计

权限示例：

- `message.read`
- `message.send`
- `message.bridge`
- `channel.read`
- `channel.manage`
- `user.read`
- `user.bind_external_identity`
- `file.read`
- `file.write`
- `external.http.call`

说明：

- 危险权限必须额外确认
- 外部 HTTP 访问不能默认开放

### 13.5 插件事件与主链路关系

要求：

- 插件事件默认异步处理
- 插件异常不能影响消息主链路
- 插件处理结果应记录日志

### 13.6 MC 插件边界

MC 插件属于后续场景插件，不进入当前核心后端必做主链路。

当前阶段只要求为它预留：

- 外部身份绑定接口边界
- 插件消息桥接边界
- 结构化消息承载能力
- 场景权限与平台权限分离模型

## 14. 文件消息与存储能力

### 14.1 文件消息范围

文件消息进入 V1。

V1 需要支持：

- 文件上传
- 文件消息
- 文件下载
- 文件元信息保存

### 14.2 文件存储方案

当前正式结论：

- V1 首个文件存储实现采用 MinIO

说明：

- MinIO 已在当前仓库外部服务规划内
- V1 只要求文件能力可用，不要求复杂分类、预览和内容处理

## 15. 公开能力与匿名访问保护

### 15.1 匿名可访问内容

- 服务端公开源信息
- 公开插件摘要
- 基础服务发现信息

### 15.2 匿名访问保护

当前正式结论：

- 接受匿名公开源信息访问
- 对匿名公开接口实施基础限流

目的：

- 兼顾服务发现与基础防滥用

## 16. 验收标准

### 16.1 V0 验收

- 单服务端可启动
- 匿名用户可读取服务端公开源信息
- 注册、登录、JWT 鉴权和 refresh 可用
- HTTP 与实时通道可同时工作
- 用户可以进入默认群聊频道并发送文本消息
- 消息可持久化并按频道查询
- 实时推送与历史消息 ID 一致
- 插件接口、事件和权限声明边界存在
- 插件异常不影响消息主链路

### 16.2 V1 验收

- 支持小型群聊的基础管理
- 支持文件消息与 MinIO 存储
- 支持更完整的权限和审计能力
- 支持插件启用、禁用和基础配置
- 为 MC 场景插件落地提供稳定接口基础

## 17. 当前结论

CarryPigeon 当前的正确方向不是做另一个通用聊天软件，而是做一套：

- 面向垂直场景的聊天服务底座
- 强调服务端边界与多服务端隔离的聊天平台
- 以插件接口而非内核侵入承载场景扩展的基础设施

在正式实现顺序上，应优先稳定：

- JWT 注册登录与刷新链路
- 群聊频道与小型群聊模型
- 通用消息模型
- 实时连接与消息主链路
- 插件接口与事件边界
