# 项目包结构规范

## 1. 规范目标

本规范用于统一项目包结构，确保：

- 业务域边界清晰
- 模块依赖方向稳定
- 包职责明确
- 人和 AI 都能快速判断类该放哪

本规范以整洁架构为原则，但 Spring Boot、Lombok 等能力视为项目内部能力，不做机械隔离。

## 2. 总原则

- 先按模块分，再按职责分
- 在 `chat-domain` 中先按 `features` 分业务域，再按层分
- 包结构优先表达业务边界和职责，不优先表达技术细节
- `shared` 只放稳定共享内容，不作为杂物区
- `application-starter` 只做启动与装配

## 3. 顶层包规范

统一使用以下顶层包前缀：

- `team.carrypigeon.backend.starter`
- `team.carrypigeon.backend.chat.domain`
- `team.carrypigeon.backend.infrastructure.basic`
- `team.carrypigeon.backend.infrastructure.service`

## 4. `chat-domain` 包结构

`chat-domain` 采用“按 feature 分域，再按职责分层”的结构。

推荐结构：

```text
team.carrypigeon.backend.chat.domain
├── features
│   ├── auth
│   │   ├── application
│   │   │   ├── command
│   │   │   ├── query
│   │   │   ├── service
│   │   │   ├── assembler
│   │   │   └── dto
│   │   ├── domain
│   │   │   ├── model
│   │   │   ├── service
│   │   │   ├── repository
│   │   │   ├── rule
│   │   │   └── event
│   │   ├── controller
│   │   │   ├── http
│   │   │   ├── ws
│   │   │   ├── dto
│   │   │   └── advice
│   │   ├── support
│   │   └── config
│   ├── channel
│   ├── message
│   ├── file
│   ├── user
│   └── server
├── shared
│   ├── domain
│   ├── application
│   ├── controller
│   └── support
└── config
```

### 4.1 为什么先有 `features`

原因：

- 防止 `chat-domain` 退化成超大单域
- 让业务边界比技术分层更明显
- 更适合未来按业务域拆分和迁移
- 更适合先定位业务域，再定位具体职责

### 4.2 `feature` 内部职责

- `application`：用例编排层
- `domain`：核心业务语义层
- `controller`：协议接入层
- `support`：仅该 feature 内部共享的辅助能力
- `config`：该 feature 的局部配置

## 5. `chat-domain` 细分规则

### 5.1 `application`

推荐结构：

```text
application
├── command
├── query
├── service
├── assembler
└── dto
```

规则：

- `command` / `query` 区分写操作和读操作
- `service` 作为应用层入口
- `assembler` 负责对象转换
- `dto` 为应用层内部对象，不等同于协议 DTO
- 可以合理使用 Spring `@Service`

### 5.2 `domain`

推荐结构：

```text
domain
├── model
├── service
├── repository
├── rule
└── event
```

规则：

- `model` 放实体、值对象、聚合内对象
- `service` 放领域服务
- `repository` 放业务语义上的仓储抽象
- `rule` 放业务规则、判断、策略
- `event` 放领域事件

特别约束：

- `repository` 保留在 `domain` 层
- `repository` 的实现不在 `chat-domain`
- `repository` 通过 `infrastructure-service` 下的数据库服务完成读写
- `domain` 不得依赖任何 `*-impl`

### 5.3 `controller`

推荐结构：

```text
controller
├── http
├── ws
├── dto
└── advice
```

规则：

- `http` 放 HTTP 控制器
- `ws` 放 WebSocket 入口
- `dto` 放协议层请求响应对象
- `advice` 放异常处理与统一绑定处理
- `controller` 只负责协议接入，不承载核心业务规则

### 5.4 `support` 与 `config`

- `support` 只放该 feature 内部共享辅助能力
- `config` 只放该 feature 的局部配置与装配
- 不允许把 `support` 或 `config` 变成兜底杂物区

## 6. `shared` 包规范

允许保留 `shared`，但只放跨 feature 且语义稳定的共享内容。

推荐结构：

```text
shared
├── domain
├── application
├── controller
└── support
```

禁止放入：

- 单一 feature 的临时代码
- 因为“不知道放哪”而塞进去的代码

## 7. `infrastructure-basic` 包结构

推荐结构：

```text
team.carrypigeon.backend.infrastructure.basic
├── config
├── json
├── logging
├── exception
├── time
├── id
└── support
```

规则：

- `config`：基础自动配置
- `json`：全局序列化能力
- `logging`：日志上下文与日志工具
- `exception`：基础异常能力
- `time`：统一时间能力
- `id`：统一 ID 能力
- `support`：无法再细分的基础支持能力

限制：

- 不放业务对象
- 不放外部服务客户端
- 不放数据库、缓存、邮件、对象存储等实现

## 8. `infrastructure-service` 包结构与契约

`infrastructure-service` 以服务维度拆分，而不是按技术组件平铺。

推荐结构：

```text
infrastructure-service
├── database-api
├── database-impl
├── cache-api
├── cache-impl
├── mail-api
├── mail-impl
└── ...
```

### 8.1 `*-api` 契约

推荐结构：

```text
team.carrypigeon.backend.infrastructure.service.{service}.api
├── service
├── model
└── exception
```

职责：

- `service`：抽象接口
- `model`：服务契约对象
- `exception`：服务抽象层异常语义

限制：

- 不引入具体中间件依赖
- 不写实现逻辑
- 可被 `chat-domain` 依赖
- 可被所属 `*-impl` 依赖

### 8.2 `*-impl` 契约

推荐结构：

```text
team.carrypigeon.backend.infrastructure.service.{service}.impl
├── config
├── service
├── client
├── convert
└── support
```

职责：

- `config`：装配配置
- `service`：接口实现
- `client`：第三方客户端封装
- `convert`：契约对象与外部对象转换
- `support`：实现内部辅助能力

限制：

- 可以依赖所属 `*-api`
- 可以依赖 `infrastructure-basic`
- 可以依赖具体外部中间件
- 不承载业务规则
- 不得被 `chat-domain` 直接依赖

### 8.3 `repository` 与数据库服务的关系

这是本项目的重要边界：

- `chat-domain.features.xxx.domain.repository` 表达业务语义上的仓储抽象
- `infrastructure-service.database-api` 表达数据库服务能力抽象
- `infrastructure-service.database-impl` 提供数据库实际接入实现

规则：

- `repository` 不放在 `infrastructure-service`
- `chat-domain` 不能直接依赖数据库实现
- 数据读写通过数据库服务能力完成

## 9. `application-starter` 包结构

`application-starter` 保持极小。

推荐结构：

```text
team.carrypigeon.backend.starter
├── config
├── bootstrap
└── support
```

规则：

- `config`：Bean 装配、条件装配、实现选择
- `bootstrap`：启动初始化逻辑
- `support`：仅启动层需要的辅助能力
- 不放业务 controller、业务 service、领域规则

## 10. 依赖方向

- `controller` 可以依赖 `application`
- `application` 可以依赖 `domain`
- `domain` 可以依赖 `infrastructure-service/*-api`
- `chat-domain` 不得依赖任何 `*-impl`
- `*-api` 可被 `chat-domain` 依赖
- `*-impl` 由 `application-starter` 装配
- `infrastructure-basic` 不依赖 `chat-domain`

## 11. 命名规范

- 包名全部小写
- 包名用职责名，不用模糊词作为主分层名

避免以下主包名：

- `common`
- `util`
- `manager`
- `helper`
- `misc`

类命名建议：

- `*Controller`：只在 `controller/http` 或 `controller/ws`
- `*ApplicationService`：只在 `application/service`
- `*DomainService`：只在 `domain/service`
- `*Repository`：只在 `domain/repository` 或 `*-api/service`
- `*AutoConfiguration`：只在配置相关包
- `*Properties`：只在配置相关包
- `*Client`：只在 `*-impl/client`

## 12. 禁止事项

禁止以下情况：

- 在 `chat-domain` 根下平铺 `controller/service/dto`
- 在 `infrastructure-basic` 放业务语义对象
- 在 `chat-domain` 直接依赖外部服务实现
- 在 `application-starter` 写核心业务逻辑
- 用 `support/shared/common/util` 长期承接无法归类的代码

## 13. 最终判断标准

一个包结构是否合格，以以下标准判断：

- 能否先看业务域，再看职责层
- 类的归属是否一眼明确
- 是否支持后续拆分某个 feature
- 是否维持了既定依赖方向
