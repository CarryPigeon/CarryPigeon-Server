# 项目注释规范

## 1. 规范目的

本规范用于统一项目中的注释写法与注释边界。

注释的目标不是重复代码表面含义，而是补充代码自身不容易直接表达的信息，重点包括：

- 设计意图
- 业务语义
- 边界条件
- 依赖约束
- 副作用与风险
- 长期维护约定

本规范同时服务于：

- 人类开发者阅读与协作
- AI 辅助理解、补全、重构
- 项目长期迭代中的知识沉淀

## 2. 总原则

### 2.1 注释写意图，代码写实现

优先写：

- 为什么这样设计
- 这里的业务语义是什么
- 这里依赖什么边界
- 这里有哪些约束和副作用

避免写：

- 代码表面动作的直译
- 一眼可见的语句复述
- 与当前实现无关的泛泛说明

### 2.2 注释必须和代码一起维护

只要代码语义发生变化，相关注释必须同步修改。

禁止保留已经失真的旧注释。

### 2.3 注释优先稳定

注释应尽量描述稳定语义，不依赖短期实现细节。

优先写长期成立的设计规则，不优先写临时实现过程。

### 2.4 注释必须精确、直接、可检索

推荐在关键注释中明确写出：

- 职责
- 边界
- 输入
- 输出
- 依赖
- 副作用
- 约束

## 3. 注释层级

### 3.1 模块级注释

适用于模块文档、`package-info.java`、模块配置入口。

必须说明：

- 模块职责
- 模块边界
- 可以依赖谁
- 不得依赖谁

### 3.2 类级注释

适用于配置类、服务类、适配器类、工具类、抽象基类、配置属性类。

至少说明一项：

- 该类负责什么
- 该类在架构中的位置
- 该类和其它对象如何协作
- 使用限制或替换约束

禁止只写“XX工具类”“XX服务类”这类无信息注释。

### 3.3 方法级注释

适用于公共方法、业务入口方法、有副作用的方法、边界条件复杂的方法。

重点说明：

- 方法的业务语义
- 参数的业务含义
- 返回值的业务含义
- 前置条件
- 副作用
- 异常或失败语义

简单 getter、setter、纯映射方法可以不写。

### 3.4 代码块注释

适用于复杂分支、兼容逻辑、补偿逻辑、顺序不可打乱的逻辑。

代码块注释应写在代码块上方，不写在行尾。

### 3.5 字段级注释

适用于配置项、业务常量、状态字段、约束性字段。

重点说明：

- 字段用途
- 允许值范围
- 默认值语义
- 是否允许为空

## 4. 面向 AI 的要求

关键注释建议显式包含以下信息：

- `职责：`
- `边界：`
- `依赖：`
- `输入：`
- `输出：`
- `副作用：`
- `约束：`
- `原因：`

要求：

- 同一概念保持术语一致
- 不写模糊表述
- 不依赖上下文猜测
- 能写全就写全，避免悬空指代

## 5. 面向长期迭代的要求

### 5.1 注释必须抗重构

优先描述稳定设计意图，不依赖临时实现细节。

### 5.2 临时方案必须显式标记

仅允许使用以下标记：

- `TODO:` 后续要完成但当前未完成
- `FIXME:` 当前存在已知问题
- `NOTE:` 重要说明
- `WARNING:` 明显风险提示

禁止使用随意标签，例如：

- `后续优化`
- `先这样`
- `以后再改`

### 5.3 待办注释必须可执行

`TODO` / `FIXME` 必须写清楚动作目标，不能只写现象。

推荐：

- `TODO: 将邮件发送能力迁移到 infrastructure-service/email-api 与 email-impl`

不推荐：

- `TODO: 后面优化一下`

### 5.4 注释不能替代设计文档

跨模块规则、长期边界、统一约定，应优先写入 `docs/`。

代码注释只保留与当前代码直接相关的局部说明。

## 6. 模块注释建议

### 6.1 `chat-domain`

重点写：

- 业务语义
- 领域规则
- 依赖边界
- 为什么只依赖 `*-api`

### 6.2 `infrastructure-basic`

重点写：

- 全局固定基础设施能力的职责
- 为什么这是固定能力而不是可替换能力
- 统一日志、序列化、时间、异常等规则

### 6.3 `infrastructure-service`

`*-api` 重点写：

- 抽象能力语义
- 输入输出契约
- 供 `chat-domain` 使用的边界

`*-impl` 重点写：

- 具体中间件适配说明
- 技术限制
- 一致性与性能注意事项

### 6.4 `application-starter`

重点写：

- 启动装配关系
- 默认实现为何在这里装配
- 配置加载与 Bean 选择逻辑

## 7. JavaDoc 规范

推荐对以下对象使用 JavaDoc：

- `public class`
- `public interface`
- `public method`
- 跨模块调用的抽象接口
- 配置属性类

JavaDoc 应优先说明业务语义，不应只复制参数名。

建议包含：

- 一句话职责
- 必要的边界说明
- `@param` 的业务含义
- `@return` 的业务结果语义
- `@throws` 的失败条件

## 8. 禁止事项

禁止以下类型的注释：

- 重复代码表面含义
- 已经过时但未更新
- 情绪化表达
- 模糊表达
- 与当前实现不一致
- 用注释掩盖坏命名或坏结构
- 把设计文档全文复制到代码里

## 9. 测试注释规范

测试代码同样遵守本规范。

测试类必须说明：

- 被测对象
- 验证的核心契约
- 不验证什么边界

测试方法必须说明：

- 当前方法测的是什么行为
- 输入条件是什么
- 期望结果是什么

当测试方法名已经表达场景时，注释仍需要补充该测试背后的契约意义。

```java
 * 职责：为整个应用提供 snake_case 序列化约定。
 * 边界：不负责业务字段映射规则，也不处理模块级定制序列化。
 */
```

```java

```

不推荐：

```java
// 给对象赋值
user.setName(name);
```


## 11. 维护要求

以下情况必须检查并更新注释：

- 方法签名变化
- 返回语义变化
- 模块依赖方向变化
- 配置项语义变化
- 异常行为变化
- 临时方案转正式方案
- 重构导致职责变化

如果改动涉及架构边界，同时更新：

- 代码注释
- `docs/` 中的对应文档

## 12. 最终判断标准

一段注释是否合格，以以下标准判断：

- 人第一次读能否快速理解意图
- AI 能否稳定提取职责、边界、约束
- 三个月后回看是否仍然成立
- 删除这段注释后，是否真的会损失关键信息
