# 任务单模板

## 1. 文档目的

本文用于统一 AI 辅助开发时的任务描述方式。

目标是让每次任务都具备明确边界、明确约束、明确验收标准，减少 AI 自由发挥带来的漂移风险。

## 2. 使用规则

- 每次交给 AI 的正式开发任务，建议尽量按本模板描述
- 任务不一定要写满全部字段，但核心边界必须明确
- 结构性任务、跨模块任务、依赖变更任务必须尽量完整填写
- 任务单文件应存放在 `ai-agent-workplace/`
- 任务单文件命名统一使用：
  - `{{time}}-{{author}}-{{task}}-{{state}}.md`
- 命名字段说明：
  - `time`：时间戳，例如 `20260420-132451`
  - `author`：作者标识，例如 `ai`
  - `task`：任务短名，使用 kebab-case
  - `state`：任务状态枚举
- `state` 取值固定为：
  - `current`
  - `done`
- 任务执行中使用 `current`，任务完成后应重命名为 `done`

## 3. 标准模板

```text
任务名称：

任务目标：

任务背景：

影响模块：

允许修改范围：

禁止修改范围：

依赖限制：

配置限制：

文档依据：

实现要求：

测试要求：

文档要求：

验收标准：

补充说明：
```

## 4. 字段说明

### 4.1 `任务名称`

用于简要标识本次任务。

要求：

- 简洁
- 可检索
- 能直接看出主题

### 4.2 `任务目标`

说明这次任务最终要达成什么结果。

要求：

- 写结果，不写空泛动作
- 能直接用于验收

### 4.3 `任务背景`

说明为什么要做这次任务。

要求：

- 交代必要上下文
- 说明当前问题或缺口

### 4.4 `影响模块`

明确本次任务涉及哪些模块。

示例：

- `application-starter`
- `infrastructure-basic`
- `infrastructure-service`

### 4.5 `允许修改范围`

明确 AI 可以改哪些内容。

示例：

- 只允许修改 `infrastructure-basic`
- 允许新增 `docs/` 文档
- 允许补充测试，不允许改业务逻辑

### 4.6 `禁止修改范围`

明确 AI 不得碰哪些内容。

示例：

- 不允许修改模块依赖方向
- 不允许新增第三方依赖
- 不允许调整 `chat-domain` 包结构

### 4.7 `依赖限制`

说明本次任务在依赖上的边界。

示例：

- 只能使用 Spring Boot 与 Lombok
- 如需新依赖，必须先申请

### 4.8 `配置限制`

说明配置方面的约束。

示例：

- 只允许保留最小配置
- 不允许新增未来占位配置

### 4.9 `文档依据`

明确本次任务应遵守哪些文档。

示例：

- `docs/架构文档.md`
- `docs/包结构规范.md`
- `docs/注释规范.md`

### 4.10 `实现要求`

说明本次任务的实现层要求。

示例：

- 必须遵守 `features` 分包
- repository 保留在 domain 层
- 只做最小可运行实现

### 4.11 `测试要求`

说明本次任务需要的测试范围。

示例：

- 至少补充成功路径和失败路径
- 必须补充测试注释

### 4.12 `文档要求`

说明本次任务是否需要同步更新文档。

示例：

- 若新增长期规则，必须回写 `docs/`

### 4.13 `验收标准`

明确什么情况下算完成。

要求：

- 可检查
- 可验证
- 不要写成空话

### 4.14 `补充说明`

用于写特殊限制、人工拍板点、待确认事项。

## 5. 面向 AI 的精简任务单

若任务较小，可使用以下精简版本：

```text
目标：
影响模块：
允许修改：
禁止修改：
遵守文档：
验收标准：
```

## 6. 推荐实践

- 结构性任务使用完整模板
- 小任务可用精简模板
- 每次任务都至少明确“允许修改”和“禁止修改”
- 每次任务都尽量绑定具体规范文档
- 对长期规则有影响的任务，必须要求同步更新文档

## 7. 与 `ai-agent-workplace` 的关系

任务执行过程中产生的以下内容，默认进入：

- `ai-agent-workplace/`

包括：

- 任务拆解草稿
- 方案对比稿
- 中间分析记录
- 待确认草案

正式代码与正式文档不应长期停留在该目录。
