# 任务单模板

## 1. 文档目的

本文用于统一 AI 辅助开发时的任务描述方式。

目标是让每次任务都具备明确边界、明确约束、明确验收标准，减少 AI 自由发挥带来的漂移风险。

## 2. 使用规则

- 每次交给 AI 的正式开发任务，建议尽量按本模板描述
- 任务不一定要写满全部字段，但核心边界必须明确
- 结构性任务、跨模块任务、依赖变更任务必须尽量完整填写
- 任务单文件应存放在 `ai-agent-workplace/`
- 任务单文件命名统一使用：
  - `{{time}}-{{author}}-{{task}}-{{state}}.md`
- 命名字段说明：
  - `time`：时间戳，例如 `20260420-132451`
  - `author`：作者标识，例如 `ai`
  - `task`：任务短名，使用 kebab-case
  - `state`：任务状态枚举
- `state` 取值固定为：
  - `current`
  - `done`
- 任务执行中使用 `current`，任务完成后应重命名为 `done`

```text

任务目标：

任务背景：

禁止修改范围：

配置限制：

文档依据：

任务分解 / 执行计划：

关键假设与依赖：

实现要求：

测试要求：

质量门禁：

复审要求：

```

## 4. 字段说明

### 4.1 `任务名称`

用于简要标识本次任务。

要求：

- 简洁
- 可检索
- 能直接看出主题

### 4.2 `任务目标`

说明这次任务最终要达成什么结果。

要求：

- 写结果，不写空泛动作
- 能直接用于验收

### 4.3 `任务背景`

说明为什么要做这次任务。

要求：

- 交代必要上下文
- 说明当前问题或缺口

### 4.4 `影响模块`

明确本次任务涉及哪些模块。

示例：

- `application-starter`
- `infrastructure-basic`
- `infrastructure-service`

### 4.5 `允许修改范围`

明确 AI 可以改哪些内容。

示例：

- 只允许修改 `infrastructure-basic`
- 允许新增 `docs/` 文档
- 允许补充测试，不允许改业务逻辑

### 4.6 `禁止修改范围`

明确 AI 不得碰哪些内容。

示例：

- 不允许修改模块依赖方向
- 不允许新增第三方依赖
- 不允许调整 `chat-domain` 包结构

### 4.7 `依赖限制`

说明本次任务在依赖上的边界。

示例：

- 只能使用 Spring Boot 与 Lombok
- 如需新依赖，必须先申请

### 4.8 `配置限制`

说明配置方面的约束。

示例：

- 只允许保留最小配置
- 不允许新增未来占位配置

### 4.9 `文档依据`

明确本次任务应遵守哪些文档。

示例：

- `docs/架构文档.md`
- `docs/包结构规范.md`
- `docs/注释规范.md`

### 4.10 `任务分解 / 执行计划`

说明 AI 对本任务的自主执行计划。

要求：

- 拆成可检查的步骤
- 说明执行顺序
- 标出哪些步骤依赖前置确认或前置发现
- 不把计划写成空泛口号

### 4.11 `关键假设与依赖`

说明当前实现依赖哪些判断、环境或外部条件。

要求：

- 明确哪些是假设，哪些是已确认事实
- 架构、依赖、配置、对外协议相关假设必须等待确认
- 若假设被推翻，应先更新任务单再继续

### 4.12 `实现要求`

说明本次任务的实现层要求。

示例：

- 必须遵守 `features` 分包
- repository 保留在 domain 层
- 只做最小可运行实现

### 4.13 `测试要求`

说明本次任务需要的测试范围。

示例：

- 至少补充成功路径和失败路径
- 必须补充测试注释

### 4.14 `质量门禁`

说明任务完成前必须通过哪些检查。

示例：

- LSP/编译诊断无新增错误
- 相关测试通过
- 受影响 Maven reactor 构建通过
- 对外接口响应码测试覆盖成功与失败路径
- 跨模块改动通过依赖边界检查

要求：

- 门禁必须能被命令、测试、文件检查或人工复审验证
- 不能只写“检查通过”这类不可追溯表述

### 4.15 `复审要求`

说明是否需要独立复审以及复审重点。

适用场景：

- 跨模块改动
- 架构、依赖、配置、对外协议改动
- 安全、认证、权限、数据持久化相关改动
- 用户明确要求深度自检或审核

### 4.16 `文档要求`

说明本次任务是否需要同步更新文档。

示例：

- 若新增长期规则，必须回写 `docs/`

### 4.17 `验收标准`

明确什么情况下算完成。

要求：

- 可检查
- 可验证
- 不要写成空话

### 4.18 `完成定义`

说明本任务何时可以从 `current` 改为 `done`。

要求至少包含：

- 验收标准已满足
- 质量门禁已执行并记录
- 阻塞问题已修复或明确标记
- 工作目录已清理或归档

### 4.19 `实际结果`

任务结束时填写实际完成内容。

要求：

- 说明最终改了什么
- 说明是否偏离原计划
- 若范围变化，说明是否已重新确认

### 4.20 `验证记录`

记录实际运行过的命令、检查和结果。

要求：

- 写明命令或检查项
- 写明通过、失败或无法运行
- 失败后若已修复，应记录最终通过结果

### 4.21 `残留风险`

说明仍存在但不阻塞完成的问题。

要求：

- 区分阻塞风险与非阻塞风险
- 不能用“暂无”掩盖未验证内容

### 4.22 `知识沉淀 / 是否回写 docs`

判断本任务是否产生长期规则或可复用经验。

要求：

- 若产生长期规则，应写明回写到哪个 `docs/` 文件
- 若不需要回写，应说明原因

### 4.23 `产物清理与保留说明`

说明 AI 工作目录中哪些产物保留、归档或删除。

要求：

- 当前任务单完成后应改为 `done`
- 历史完成任务单应归档
- 冗余草稿、占位文件或重复总结应删除或标明替代关系

### 4.24 `补充说明`

用于写特殊限制、人工拍板点、待确认事项。

## 5. 面向 AI 的精简任务单

若任务较小，可使用以下精简版本：

```text
目标：
影响模块：
允许修改：
禁止修改：
执行计划：
遵守文档：
验收标准：
质量门禁：
是否回写 docs：
```

## 6. 推荐实践

- 结构性任务使用完整模板
- 小任务可用精简模板
- 每次任务都至少明确“允许修改”和“禁止修改”
- 每次任务都尽量绑定具体规范文档
- 对长期规则有影响的任务，必须要求同步更新文档

## 7. 与 `ai-agent-workplace` 的关系

任务执行过程中产生的以下内容，默认进入：

- `ai-agent-workplace/`

包括：

- 任务拆解草稿
- 方案对比稿
- 中间分析记录
- 待确认草案

正式代码与正式文档不应长期停留在该目录。
