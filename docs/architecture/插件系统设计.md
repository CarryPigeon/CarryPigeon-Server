# 插件系统设计

## 1. 文档状态

- 状态：已审核确认并已落地的目标设计
- 适用版本：启动期插件运行时 v1
- 当前日期：2026-07-23
- 审核日期：2026-07-23
- 本文同时记录当前实现基线；实现细节以仓库代码、任务单和测试结果为准。

本文定义 CarryPigeon Backend 的启动期插件模型。插件作为独立 JAR 进入应用启动 classpath，由 JVM 默认应用 ClassLoader 加载，并由 Spring Boot 在主 ApplicationContext 创建期间完成 Bean 装配。

本文采用以下已确认决策：

1. 不把后端打成单一上帝包。
2. 分发物区分启动 JAR、核心 JAR、依赖 JAR 和插件 JAR；核心 JAR 与第三方依赖 JAR 共同放入 `lib/`，不设置独立 `core/`。
3. 插件只在应用启动时加载，不支持运行期安装、卸载、替换或热更新。
4. 不实现自定义插件 ClassLoader；插件与核心代码使用同一应用 classpath。
5. 插件进入主 Spring ApplicationContext，可以注入和访问宿主运行时资源。
6. 第三方插件可被授予最高权限；最高权限插件与核心代码处于同一应用内信任级别。
7. 插件可以访问应用进程有权访问的数据库、缓存、对象存储、配置、密钥、文件和网络等资源。

## 2. 设计目标

### 2.1 核心目标

- 保持启动模块、核心模块、第三方依赖和插件的物理产物边界清晰。
- 让插件能够使用 Spring Boot 的自动配置、依赖注入、事务、事件、HTTP、WebSocket 和定时任务能力。
- 让插件能够扩展消息 Domain，也能够扩展其它后端能力。
- 避免自定义 ClassLoader、跨 ClassLoader 类型转换和热卸载资源回收复杂度。
- 通过启动前校验、精确版本绑定、依赖冲突检查、健康状态和安全模式降低全权限插件的运维风险。
- 明确插件的最高权限边界，不把治理措施错误描述成安全沙箱。

### 2.2 非目标

本文不设计：

- 运行期热加载、热卸载或无重启升级。
- 自定义插件 ClassLoader、Java ModuleLayer 隔离或进程外插件运行器。
- 第三方代码安全沙箱。
- 插件市场、在线下载、在线安装或自动更新。
- 对最高权限插件实施 Capability 强制隔离。
- 插件间独立依赖命名空间。
- 插件故障下的进程级容错隔离。

## 4. 信任与权限模型

### 4.1 最高权限定义

本文目标插件统一按受信任的 `SYSTEM` 插件设计。插件被管理员放入启用目录并进入启动 classpath，即代表管理员接受其以核心代码级权限运行。

`SYSTEM` 插件可以：

- 获取主 `ApplicationContext` 中的 Bean。
- 注入 `DataSource`、事务管理器、缓存、对象存储和邮件客户端。
- 调用各 feature 的 `domain/api`。
- 在技术上引用核心模块的其它 public 类型。
- 注册 Controller、Filter、Interceptor、WebSocket Handler、事件监听器和定时任务。
- 执行 SQL、创建插件表，并在管理员接受后修改核心表。
- 读取 Spring `Environment`、运行配置和应用进程可读取的密钥。
- 使用应用进程允许的文件系统和网络权限。
- 创建线程、占用连接池、发布 Spring 事件和调用 JVM API。

### 4.2 权限的实际边界

“访问全部资源”限定在当前应用进程的实际权限范围内：

```text
操作系统用户 / 容器权限
  -> JVM 进程权限
      -> Spring ApplicationContext 与已配置外部服务
          -> 插件可访问范围
```

插件不会因为被标记为 `SYSTEM` 自动获得操作系统 root 权限，也不能访问操作系统、容器或网络策略已经禁止的资源。

### 4.3 不存在应用内安全隔离

全权限插件可以绕过业务 API、直接访问数据库或读取密钥，因此以下治理措施都不是安全边界：

- Manifest
- 插件签名
- 配置开关
- 插件目录
- Spring Bean 注册规范
- 日志和审计
- 推荐使用 `domain/api` 的约定

恶意或错误插件可以泄露数据、删除数据、阻塞线程、耗尽内存、破坏事务、调用 `System.exit` 或使应用无法启动。Java 默认应用 ClassLoader 和 Spring ApplicationContext 不能阻止这些行为。

因此，只有经过人工审核并愿意承担核心代码同等级风险的插件才应进入生产 `plugins/` 目录。

## 5. 目标分发结构

目标分发包保持分层 thin-jar，不使用 fat jar：

```text
carrypigeon/
├── app/
│   └── application-starter-<version>.jar
├── plugins/
│   ├── mc-bridge-<version>.jar
│   ├── audit-extension-<version>.jar
│   ├── 其它已物理加载的插件 JAR
│   └── disabled/
│       └── 不进入启动 classpath 的插件 JAR
├── config/
│   ├── application.yaml
│   └── log4j2-spring.xml
├── lib/
│   ├── chat-domain-<version>.jar
│   ├── infrastructure-basic-<version>.jar
│   ├── database-api-<version>.jar
│   ├── database-impl-<version>.jar
│   ├── cache-api-<version>.jar
│   ├── cache-impl-<version>.jar
│   ├── storage-api-<version>.jar
│   ├── storage-impl-<version>.jar
│   ├── mail-api-<version>.jar
│   ├── mail-impl-<version>.jar
│   ├── spring-*.jar
│   ├── jackson-*.jar
│   ├── mysql-connector-*.jar
│   └── 其它核心或第三方依赖 JAR
├── bin/
│   ├── start.sh
│   ├── start.bat
│   ├── start.ps1
│   └── verify.*
├── run/
└── service-logs/
```

目录职责：

| 目录 | 职责 |
| --- | --- |
| `app/` | 只存放 Spring Boot 启动入口 JAR |
| `plugins/` | 直接存放本次启动需要进入 classpath 的插件 JAR；`plugins/disabled/` 不进入 classpath |
| `config/` | 存放外部运行配置和日志配置 |
| `lib/` | 同时存放仓库核心模块 JAR 和宿主、插件共同使用的第三方依赖 JAR |
| `bin/` | 在 JVM 启动前完成校验和 classpath 组装 |

当前 `distribution` 已将核心模块和第三方依赖放入单数目录 `lib/`，并新增 `plugins/` 与 `plugins/disabled/`；启动脚本已支持普通模式和安全模式。

## 6. 启动与加载模型

### 6.1 “不使用 ClassLoader”的准确含义

Java 字节码必须由 ClassLoader 加载。本文所说“不使用 ClassLoader 加载插件”准确含义是：

- 不实现自定义插件 ClassLoader。
- 不在应用运行后调用 `URLClassLoader` 动态追加 JAR。
- 不为不同插件创建独立类命名空间。
- 插件与核心 JAR 一起由 JVM 默认应用 ClassLoader 从启动 classpath 加载。

### 6.2 JVM 启动前组装 classpath

Linux/macOS 目标启动模型：

```bash
APP_JAR="app/application-starter-1.0.0.jar"
java \
  -cp "$APP_JAR:lib/*:plugins/*" \
  team.carrypigeon.backend.starter.ApplicationStarter
```

Windows 目标启动模型：

```bat
set "APP_JAR=app\application-starter-1.0.0.jar"
java -cp "%APP_JAR%;lib\*;plugins\*" ^
  team.carrypigeon.backend.starter.ApplicationStarter
```

实际脚本应解析出唯一的启动 JAR，不能依赖应用 JAR 通配符恰好只匹配一个文件。

不能以普通方式使用：

```bash
java -jar app/application-starter.jar
```

原因是使用 `-jar` 时，命令行 `-cp` 不作为应用 classpath 生效，而 Manifest `Class-Path` 不适合在部署时动态枚举插件目录。

### 6.3 启动阶段

目标启动阶段为：

```text
阶段 0：启动脚本预检
  -> 定位唯一启动 JAR
  -> 确认 lib/plugins/config 目录
  -> 判断是否启用安全模式
  -> 组装完整 classpath

阶段 1：JVM 创建与插件描述校验
  -> ApplicationStarter.main
  -> 枚举 META-INF/carrypigeon/plugin.yaml
  -> 校验 plugin_id、版本、宿主版本、依赖和重复项
  -> 校验通过后才调用 SpringApplication.run

阶段 2：Spring Context 装配
  -> 发现插件 AutoConfiguration
  -> 注册插件 BeanDefinition
  -> 创建插件 Controller、Service、Repository 和扩展点 Bean
  -> 注入宿主 DataSource、事务和其它资源

阶段 3：插件初始化与就绪
  -> 执行插件数据库迁移
  -> 执行插件启动回调
  -> 执行插件健康检查
  -> 所有逻辑启用的插件成功后切换为 ACCEPTING_TRAFFIC

阶段 4：应用运行
  -> 插件与核心代码共同处理请求和事件

阶段 5：应用停止
  -> 按启动逆序调用插件停止回调
  -> 关闭 Spring Context
  -> JVM 退出
```

插件迁移必须在插件依赖的数据访问 Bean开始承载业务请求前完成。具体实现可以由启动协调器在 Spring Context 就绪后、Readiness 接受流量前执行；不要求把 SQL 执行塞入启动脚本。

插件不得在构造器、静态初始化块或迁移前的 Bean 初始化回调中访问尚未迁移的插件表。宿主还必须提供全局启动门禁：插件初始化结束前 Readiness 保持 `REFUSING_TRAFFIC`，直接到达应用端口的业务请求也应被启动门禁拒绝，避免仅依赖上游负载均衡器理解 Readiness。

### 6.4 不支持运行期变更

以下操作必须通过停止和重新启动应用完成：

- 新增插件。
- 删除插件。
- 启用或禁用插件。
- 升级或降级插件。
- 修改插件依赖 JAR。
- 修改插件需要的核心模块版本。
- 修改需要重新绑定的插件配置。

运行期管理接口最多展示状态或写入“下次启动”配置，不得声称已经加载、卸载或替换插件代码。

## 7. 插件包契约

### 7.1 插件 JAR 必需内容

每个插件 JAR 至少包含：

```text
META-INF/carrypigeon/plugin.yaml
META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
插件实现类
插件资源文件
```

插件通过 Spring Boot AutoConfiguration 进入主 ApplicationContext，不依赖任意包扫描。

### 7.2 Manifest 目标结构

```yaml
plugin_id: com-example-mc-bridge
name: MC Bridge
version: 1.0.0
publisher: example
description: Minecraft bridge integration

trust_level: SYSTEM
load_mode: STARTUP_CLASSPATH

host:
  version: 1.0.0
  build_hash: abc123
  java_version: 21
  spring_boot_version: 3.5.3

entrypoint:
  auto_configuration:
    - example.plugin.mcbridge.McBridgePluginAutoConfiguration

dependencies:
  required_plugins: []
  optional_plugins: []
  required_host_artifacts:
    - CarryPigeon:chat-domain:1.0.0

extensions:
  message_domains:
    - Example:McBridge
  http: true
  websocket: false
  scheduled_jobs: false
  database_migrations: true

configuration:
  prefix: cp.plugin.configs.com-example-mc-bridge
```

### 7.3 标识规则

- `plugin_id` 必须全局唯一、稳定，只允许小写字母、数字和连字符；推荐将反向域名归一化为连字符形式。
- `plugin_id` 一经发布不得因展示名称变化而改变。
- 同一启动 classpath 中不允许出现同一 `plugin_id` 的多个版本。
- `version` 使用语义化版本，但 `SYSTEM` 插件仍必须绑定精确宿主构建。
- `publisher` 只用于识别和审计，不构成安全证明。
- `trust_level` 在本文方案中固定为 `SYSTEM`。
- `load_mode` 在本文方案中固定为 `STARTUP_CLASSPATH`。
- `configuration.prefix` 必须等于 `cp.plugin.configs.<plugin_id>`，与现有插件配置规范保持一致。
- `dependencies.required_host_artifacts` 使用 `groupId:artifactId:version` 精确坐标；必须由 `app/` 或 `lib/` 中的非插件构件提供，不能由插件 JAR 自己伪造满足。

### 7.4 Spring AutoConfiguration

插件必须使用显式 AutoConfiguration 或 `@Import` 注册组件，不应对整个宿主包进行宽泛 `ComponentScan`。

AutoConfiguration 可以注册：

- 插件服务和配置属性。
- HTTP Controller、Filter 和 Interceptor。
- WebSocket Handler。
- Spring 事件监听器。
- 定时任务。
- MyBatis Mapper 扫描或其它数据访问组件。
- 消息 Domain 插件注册项。
- 插件生命周期 Bean。

Spring 默认的 Bean 命名冲突和循环依赖检查仍然生效。最高权限意味着插件能够访问全部资源，不意味着默认允许静默覆盖同名核心 Bean。需要替换核心 Bean 时必须使用明确、可审核的 Spring 装配策略。

## 8. 插件生命周期

### 8.1 目标接口

后续实现可以提供最小生命周期契约：

```java
public interface SystemPlugin {
    PluginDescriptor descriptor();

    void start(SystemPluginContext context);

    PluginHealth health();

    void stop();
}
```

该接口是推荐的统一管理入口，不是权限边界。插件即使不通过 `SystemPluginContext`，仍可通过 Spring 注入访问宿主资源。

### 8.2 状态模型

目标状态：

```text
DISCOVERED
  -> VERIFIED
  -> CONFIGURING
  -> STARTING
  -> ACTIVE
  -> DEGRADED
  -> STOPPING
  -> STOPPED

任意启动阶段失败 -> FAILED
被安全模式排除   -> SKIPPED
```

不设计 `UNLOADED` 或运行期 `DISABLED`，因为插件代码一旦进入本次 JVM classpath，就不会在运行期卸载。

### 8.3 启动顺序

- 插件依赖关系必须形成无环图。
- 强依赖插件按拓扑顺序启动。
- 相同依赖层级使用稳定的 `plugin_id` 排序，保证日志和测试可复现。
- 停止顺序为启动顺序的逆序。
- 不允许使用 classpath 中 JAR 的文件系统枚举顺序表达依赖关系。

## 9. 插件访问核心能力

### 9.1 统一 Spring Context

插件 Bean 与核心 Bean 位于同一个主 ApplicationContext：

```text
默认应用 ClassLoader
  -> application-starter JAR
  -> lib 中的核心 JAR
  -> lib 中的第三方依赖 JAR
  -> plugin JAR

Spring ApplicationContext
  -> 核心 Bean
  -> 基础设施 Bean
  -> 插件 Bean
```

因此插件可以使用构造器注入获得核心资源，不需要 RPC、序列化或跨 ClassLoader 适配。

### 9.2 推荐调用路径与实际权限

为了减少宿主升级造成的插件破坏，推荐路径仍是：

```text
插件
  -> 目标 feature 的 domain/api
  -> 核心领域服务
  -> infrastructure-service/*-api
  -> 对应 *-impl Bean
```

但 `SYSTEM` 插件在技术上可以引用其它 public 类型、获取具体 Spring Bean 或直接使用基础设施实现。项目包结构规范对仓库内核心模块仍然有效；外部全权限插件的越界访问属于允许但版本不稳定的特权行为。

这一区分必须明确：

- `domain/api` 是稳定、推荐的业务调用入口。
- 直接调用 `service`、`repository` 或 `*-impl` 是最高权限插件可执行的行为。
- 直接调用内部类型不享受兼容性承诺，宿主升级时插件必须重新测试和编译。

### 9.3 ApplicationContext 访问

插件可以直接注入 `ApplicationContext` 或通过目标生命周期 Context 获得：

```text
ApplicationContext
ConfigurableEnvironment
BeanFactory
ResourceLoader
ApplicationEventPublisher
TaskExecutor
ObjectMapper
DataSource
PlatformTransactionManager
InfrastructureBasics
各 feature domain/api
各基础设施 API 和已装配实现
```

不提供假装安全的万能代理来阻止 `SYSTEM` 插件访问宿主 Bean。

## 10. 数据库访问与事务

### 10.1 访问方式

插件通过主 Spring Context 注入宿主已有数据库资源：

```java
public PluginRepository(
        DataSource dataSource,
        JdbcTemplate jdbcTemplate,
        PlatformTransactionManager transactionManager) {
    // 构造器注入
}
```

插件无需获取数据库密码或自行创建连接池。注入宿主 `DataSource` 后即可使用已有连接池、驱动和事务设施。

最高权限插件还可以从 `Environment` 读取数据库配置，但不推荐把原始凭据作为常规调用方式。

### 10.2 事务

插件使用宿主 `DataSource` 和事务管理器时，可以参与宿主事务：

- 使用 `@Transactional`。
- 使用项目已有事务抽象。
- 调用核心 Repository 和插件 SQL时共享同一事务上下文。
- 注册事务提交后的事件或副作用时，应复用项目现有 after-commit 语义。

插件不得因为拥有最高权限就假定数据库事务可以覆盖 Redis、MinIO、邮件或外部 HTTP。跨资源一致性仍需显式补偿、幂等或持久化事件机制。

### 10.3 插件数据

推荐插件自有表使用稳定前缀：

```text
plugin_<normalized_plugin_id>_*
```

推荐规则：

- 插件自有表由插件自己解释和迁移。
- 插件表可以引用核心实体 ID，但默认不建立跨 feature 外键。
- 插件卸载不自动删除数据。
- 插件升级必须提供向前迁移方案。
- 插件可以在最高权限下修改核心表，但这属于高风险操作，必须在 Manifest 和审核记录中显式声明。

### 10.4 数据库迁移

插件迁移资源建议放置：

```text
db/plugin/<plugin-id>/migration/
  V1__create_plugin_tables.sql
  V2__add_index.sql
```

宿主目标运行时应维护独立迁移记录：

```text
plugin_schema_history
```

至少记录：

```text
plugin_id
plugin_version
migration_version
description
checksum
executed_at
success
```

迁移约束：

- 同一插件迁移版本不可重复。
- 已成功迁移的 checksum 变化必须拒绝启动。
- 强依赖插件按依赖顺序迁移。
- 迁移失败时不得进入可接流量状态。
- 不自动执行破坏性回滚。
- 修改核心表的迁移必须被标记并在部署前人工审核。

## 11. 消息 Domain 扩展

当前 `chat-domain/features/plugin` 已实现消息 Domain 扩展能力。通用插件可以通过 Spring AutoConfiguration 注册现有 `ChannelMessagePluginRegistration` Bean，将自己的消息 Domain 接入现有注册表。

消息 Domain 插件继续负责：

- 声明 `domain` 和 `domain_version`。
- 校验 `message.data`。
- 规范化 `message.data`。
- 派生或校验 `preview`。
- 声明公开插件标识和治理信息。

消息创建主链路继续由核心 message feature 负责：

- `message_id`、`sender_id`、`channel_id` 和时间生成。
- mentions 元数据处理。
- 消息状态与撤回。
- 权限检查。
- 持久化。
- HTTP/WS 协议输出。
- realtime 事件发布。

全权限插件虽然可以绕过这些边界，但标准消息扩展必须通过现有插件 SPI 接入，以保持消息统一模型和 HTTP Domain data 校验链路一致。

## 12. 其它扩展能力

插件可以借助 Spring 原生装配扩展：

| 扩展类型 | 推荐接入方式 |
| --- | --- |
| HTTP API | 插件 AutoConfiguration 显式注册 Controller |
| HTTP Filter / Interceptor | 注册 FilterRegistrationBean 或 WebMvcConfigurer |
| WebSocket | 注册项目约定的 Handler 或协议适配 Bean |
| 事件处理 | 注册 Spring 事件监听器或项目事件处理 Bean |
| 定时任务 | 注册受治理的 TaskScheduler 任务 |
| 数据访问 | 注入 DataSource、数据库 API 或 Repository |
| 消息 Domain | 注册 `ChannelMessagePluginRegistration` |
| 配置 | 绑定 `cp.plugin.configs.<plugin-id>` |
| JSON | 注册插件自有 ObjectMapper Module，避免全局破坏性覆盖 |
| 核心 Bean 定制 | 显式装饰、`@Primary` 或经审核的 BeanPostProcessor |

需要长期稳定和可诊断的扩展类型，后续应在 `features/plugin` 中增加显式注册表；不应仅依赖插件自行扫描或静态初始化。

## 13. 配置模型

沿用当前统一入口：

```yaml
cp:
  plugin:
    configs:
      com-example-mc-bridge:
        enabled: true
        options:
          endpoint: "https://example.invalid"
```

约束：

- `cp.plugin.configs.<plugin-id>` 只承载对应插件配置。
- `enabled` 是 Spring 装配和业务启用的协作性配置，不是安全边界。
- 插件配置由插件实现负责类型绑定和业务校验。
- 未实现读取逻辑前，不在默认配置中添加占位项。
- 插件密钥仍由部署 YAML 或部署侧密钥系统在启动前物化。
- 修改要求重新绑定的配置后，必须重启应用。

硬禁用插件的可靠方式是将 JAR 移出 `plugins/` 或以安全模式启动。仅设置 `enabled: false` 不能防止恶意全权限插件中的静态初始化代码执行。

需要区分两个状态：

- 物理加载：JAR 位于 `plugins/`，因此已经进入本次 JVM classpath，必须参与 Manifest、重复类和依赖冲突检查。
- 逻辑启用：`cp.plugin.configs.<plugin-id>.enabled=true`，插件 AutoConfiguration 和启动回调才应创建业务 Bean 并开始工作。

未配置插件沿用当前配置基建语义，默认逻辑不启用。为了避免无效 JAR 仍污染 classpath，生产部署应把长期禁用的插件移入 `plugins/disabled/` 或移出分发包。

## 14. 依赖与兼容性治理

### 14.1 单一类命名空间

由于不使用自定义 ClassLoader，所有核心模块、依赖和插件共享同一类命名空间。其结果是：

- 同一全限定类名只能有一个有效定义。
- 不同插件不能安全携带同一库的不同不兼容版本。
- 不能依赖 classpath 文件枚举顺序解决版本冲突。
- 插件与宿主必须共同完成依赖收敛。

### 14.2 插件构建规则

- Java、Spring Boot、Jackson、日志框架和数据库驱动由宿主提供，插件使用 `provided`。
- CarryPigeon 核心 JAR由宿主 `lib/` 提供，插件构建时可以依赖，但不得重复打入插件 JAR。
- 插件复用宿主已有第三方依赖时，必须使用宿主锁定版本。
- 插件独占的小型依赖可以 shade 并 relocation 到插件私有包名。
- 无法 relocation 的公共依赖必须进入统一 `lib/`，并通过全局依赖收敛审核。
- 禁止插件把另一版本的 Spring、Jackson、Log4j2、MySQL Driver 或 CarryPigeon 核心类打入插件 JAR。

### 14.3 宿主版本绑定

因为插件具有直接访问内部类型的权限，不能只依赖宽松的 API 语义版本。Manifest 至少绑定：

```text
host_version
host_build_hash
java_version
spring_boot_version
```

兼容策略：

- `host_build_hash` 精确匹配：允许启动。
- 只匹配 `host_version`：默认拒绝，除非部署审核显式放宽。
- Java 主版本不匹配：拒绝启动。
- Spring Boot 主/次版本不匹配：拒绝启动。
- 插件引用宿主内部类型时，宿主升级后必须重新编译并验证插件。

### 14.4 启动前冲突检查

目标 `verify` 流程至少检查：

- 重复 `plugin_id`。
- 重复插件版本。
- 缺失强依赖插件。
- 插件依赖环。
- 重复 Maven 坐标的不同版本。
- 核心类被插件 JAR重复打包。
- Spring 自动配置元数据是否存在。
- Manifest 读取与冲突扫描期间 JAR 文件 checksum 是否保持一致，防止预检过程中构件被替换。
- 已知高风险重复类和资源。

冲突检查只能降低意外风险，不能证明任意 Java 字节码安全。

当前实现还遵循以下判定：

- 插件必须以真实 JAR 提供，不接受展开目录作为生产插件包。
- Manifest 声明的 AutoConfiguration 类必须由同一插件 JAR 提供，并且列入该 JAR 的 Boot `AutoConfiguration.imports`。
- 插件不得重新打包 `team.carrypigeon.backend` 核心类，也不得携带 Spring、Jackson、Log4j2、SLF4J 或 MySQL Driver 等明确由宿主提供的类。
- 插件类与宿主或其它插件出现重复全限定名时拒绝启动；multi-release JAR 路径同样参与检查。
- 顶层 classpath 构件出现同 Maven 坐标不同版本时拒绝启动。已经 shade 并 relocation 的插件私有依赖不因保留嵌入式 Maven 元数据而被误判，但未 relocation 的重复类仍会被拒绝。
- 插件 JAR 的运行时 SHA-256 是宿主对实际字节计算的指纹和预检期一致性依据，不是代码签名，也不是独立于 JAR 的可信完整性锚点。发布真实性仍需分发包 checksum、签名或外部制品系统保证。

## 15. 启动失败与安全模式

### 15.1 失败原则

在统一 classpath 和主 ApplicationContext 模型下，插件装配失败可能污染整个 Spring Context，不能可靠地在同一进程内撤销已经注册的 Bean。

因此采用 fail-fast：

- Manifest 校验失败：拒绝启动。
- 插件依赖不满足：拒绝启动。
- 重复类或关键依赖冲突：拒绝启动。
- 逻辑启用插件的数据库迁移失败：拒绝进入 readiness。
- 逻辑启用插件的 Bean 创建失败：Spring Context 启动失败。
- 逻辑启用插件的启动回调失败：关闭 ApplicationContext 并退出。

直接放入 `plugins/` 的每个 JAR 都是本次 classpath 的组成部分；其 Manifest、重复类或依赖冲突校验失败时，本次启动必须失败。逻辑启用插件一旦参与主 Context 装配并发生异常，也不能尝试继续运行一个部分污染的上下文。需要可靠跳过的插件必须在 JVM 启动前移入 `plugins/disabled/` 或移出分发包。

### 15.2 安全模式

安全模式必须在 JVM 创建前生效：

```bash
CP_PLUGIN_SAFE_MODE=true bin/start.sh
```

或：

```bash
bin/start.sh --safe-mode
```

启动脚本在安全模式下使用：

```text
app/application-starter.jar:lib/*
```

而不是：

```text
app/application-starter.jar:lib/*:plugins/*
```

这样即使插件存在静态初始化错误、重复类或恶意 AutoConfiguration，也不会进入本次 JVM classpath。

仅在 `application.yaml` 中设置 `cp.plugin.safe-mode=true` 不足以形成可靠恢复边界，因为 Spring 读取配置时插件类可能已经可见。应用内配置可以用于展示状态，但硬安全模式必须由启动器排除插件 JAR。

## 16. 可观测性与审计

每次启动应记录：

```text
plugin_id
plugin_version
publisher
jar_path
jar_sha256
host_build_hash
manifest_validation_result
configuration_enabled
startup_order
migration_result
startup_duration
health_status
```

运行日志应包含稳定的 `plugin_id` 上下文。指标至少包括：

- 插件启动耗时。
- 插件健康状态。
- 插件任务执行次数和失败次数。
- 插件 HTTP 请求次数和耗时。
- 插件事件处理次数和失败次数。
- 插件数据库迁移状态。

由于插件可以绕过宿主封装，审计只能覆盖通过宿主治理入口发生的行为，不能完整捕获全权限插件的所有直接数据库、文件和网络操作。

## 17. 升级、回滚与停用

### 17.1 升级

```text
停止服务
  -> 备份数据库和插件目录
  -> 替换插件 JAR及必要依赖 JAR
  -> 执行 verify
  -> 启动服务
  -> 执行迁移和健康检查
  -> readiness 通过后接入流量
```

### 17.2 回滚

- 代码回滚通过停止服务并恢复旧 JAR完成。
- 数据库迁移默认只向前，不承诺自动反向迁移。
- 若新版本执行了不兼容数据库变更，回滚前必须使用插件提供的人工回退方案或恢复备份。
- 插件升级任务必须分别验证“新版本启动”和“旧版本在迁移后是否仍能启动”。

### 17.3 停用

```text
停止服务
  -> 将插件 JAR移至 plugins/disabled/
  -> 保留插件数据
  -> 重新启动
```

停用插件默认不删除数据库表、配置和历史数据。

## 18. 模块职责

后续实现必须继续遵守现有模块职责：

| 模块 | 插件相关职责 |
| --- | --- |
| `application-starter` | 启动前 Manifest 复检、Spring 最终装配、插件启动协调和 readiness 编排 |
| `chat-domain/features/plugin` | 插件目录业务语义、消息 Domain SPI、注册表、治理状态和对内 API |
| `infrastructure-basic` | 固定的插件配置绑定、checksum/基础描述解析等无业务语义能力 |
| `infrastructure-service/*-api` | 插件和业务可使用的外部服务抽象 |
| `infrastructure-service/*-impl` | 数据库、缓存、存储等具体实现 Bean |
| `distribution` | `app/plugins/config/lib` 主体产物与 `bin` 启动脚本组装 |
| 外部插件 JAR | 插件 AutoConfiguration、业务逻辑、资源和 Manifest |

边界补充：

- `application-starter` 只承担启动装配和协调，不承载插件业务规则。
- 通用插件业务语义归 `chat-domain/features/plugin`，不能散落到 message、server 等 feature。
- `infrastructure-basic.plugin` 不承载插件业务运行时。
- 全权限插件可以访问具体实现，不改变核心仓库自身的依赖方向规则。
- 当前已确认不新增独立 `plugin-sdk` Maven 模块；插件直接依赖 `lib/` 中的核心 JAR 和现有插件 SPI。未来若稳定契约确有独立版本价值，再另行审核。

## 19. 测试与质量门禁

### 19.1 分发测试

- 分发包存在唯一启动 JAR。

### 19.2 启动测试

- 无插件时正常启动。
- 有合法插件时自动配置进入主 Context。

## 21. 已确认设计决策

用户已在 2026-07-23 逐项确认：

1. 分发采用 `app + plugins + config + lib`；`lib/` 同时承载核心 JAR 和第三方依赖 JAR。
2. 插件 JAR在 JVM 启动前加入 classpath，运行期不追加、移除或替换。
3. 插件通过 Spring Boot AutoConfiguration 进入主 ApplicationContext，共享全部 Spring Bean。
4. 插件按 `SYSTEM` 全权限运行，不实施 Capability 权限隔离。
5. 插件绑定宿主精确 `host_build_hash`。
6. 全部 JAR共享同一依赖空间，依赖冲突通过全局收敛或 shade relocation 处理。
7. 已启用插件启动失败采用 fail-fast，不保留部分污染的 Spring Context。
8. 提供 JVM 启动前从 classpath 排除全部插件 JAR 的安全模式。
9. 插件可以访问和修改核心数据库；迁移必须记录、校验 checksum，并显式声明核心表变更。
10. 保留现有消息 Domain SPI 作为标准消息扩展入口。
11. 当前不新增独立 `plugin-sdk` Maven 模块。

## 22. 最终边界

该方案将插件定义为“随应用启动装配的受信任后端模块”，而不是“受限脚本”或“可热插拔沙箱”。

其灵活性来自：

- 独立插件 JAR。
- Spring Boot AutoConfiguration。
- 主 ApplicationContext 完整资源访问。
- 数据库和事务直接注入。
- 消息 Domain 与其它 Spring 扩展能力。
- 分层 thin-jar 交付。

其代价是：

- 插件与宿主版本高度绑定。
- 所有插件共享依赖命名空间。
- 插件故障可以影响整个应用。
- 插件拥有读取密钥和修改数据的能力。
- 安装、升级、停用都必须重启。
- 安全只能依赖发布审核、部署权限、操作系统隔离、备份和安全模式，不能依赖应用内沙箱。

上述代价已经在本轮设计审核中明确确认。后续代码实施仍必须按分阶段任务单分别获得实施授权，并完成对应构建、测试和启动验证。
