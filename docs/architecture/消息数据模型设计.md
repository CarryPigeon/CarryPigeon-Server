# 消息数据模型设计

## 1. 文档目的

本文定义 CarryPigeon Backend 重写阶段的统一消息目标模型，是后续消息领域、HTTP、WebSocket 与数据库持久化改造的设计依据。

本文描述的是**已落地的 canonical 消息模型**。领域、HTTP、realtime 与数据库必须共同遵守本文边界。

## 2. 设计结论

统一消息只由以下十个 canonical 字段组成：

```text
message_id
sender_id
channel_id
domain
domain_version
data
send_time
mentions
preview
status
```

核心原则：

- `domain + domain_version + data` 是消息内容唯一真源。
- `data` 是 JSON object，承载当前 domain 的全部专属字段。
- `mentions` 是所有消息共用的提醒元数据，不属于任何具体 domain。
- `preview` 是服务端派生的通用摘要，不是消息内容真源。
- `status` 只表达发送与撤回，不承载编辑版本语义。
- 消息不提供编辑能力。
- HTTP、历史查询和 realtime 必须投影同一份 canonical 消息。

## 3. Canonical 字段

| 领域字段 | Wire 字段 | 领域类型 | Wire 类型 | 说明 |
| --- | --- | --- | --- | --- |
| `messageId` | `mid` | `long` | 十进制字符串 | 消息 Snowflake ID |
| `senderId` | `uid` | `long` | 十进制字符串 | 发送者用户 ID |
| `channelId` | `cid` | `long` | 十进制字符串 | 所属频道 ID |
| `domain` | `domain` | `String` | `string` | 消息 domain，例如 `Core:Text` |
| `domainVersion` | `domain_version` | `String` | `string` | domain 语义版本，例如 `1.0.0` |
| `data` | `data` | `Map<String, Object>` | JSON object | domain 专属数据唯一容器 |
| `sendTime` | `send_time` | `Instant` | epoch 毫秒 | 服务端确认的发送时间 |
| `mentions` | `mentions` | `List<Long>` | 十进制字符串数组 | 需要提醒的用户 ID |
| `preview` | `preview` | `String` | `string` | 服务端生成的通用摘要 |
| `status` | `status` | `MessageStatus` | `sent` / `recalled` | 消息生命周期状态 |

Wire 中所有 Snowflake ID 必须使用十进制字符串。服务端内部领域模型和数据库主键使用 `long / BIGINT`。

## 4. 领域模型

目标领域模型：

```java
public record ChannelMessage(
        long messageId,
        long senderId,
        long channelId,
        String domain,
        String domainVersion,
        Map<String, Object> data,
        Instant sendTime,
        List<Long> mentions,
        String preview,
        MessageStatus status
) {
}
```

状态模型：

```java
public enum MessageStatus {
    SENT,
    RECALLED
}
```

领域模型约束：

- `messageId`、`senderId`、`channelId` 必须为正数。
- `domain`、`domainVersion` 不得为空。
- `data` 不得为 `null`，无专属内容时使用空 object。
- `mentions` 不得为 `null`，没有提醒目标时使用空列表。
- `mentions` 写入前去重，并保留每个用户 ID 第一次出现的顺序。
- `preview` 不得为 `null`；无可展示摘要时使用空字符串。
- 新建消息状态必须为 `SENT`。

以下旧字段不再属于目标领域模型：

- `serverId`
- `conversationId`
- `messageType`
- `body`
- `payload`
- `metadata`
- `forwardedFrom`
- `editedAt`
- `editVersion`

## 5. Wire 模型

通用消息响应固定使用以下结构：

```json
{
  "mid": "20001",
  "uid": "67890",
  "cid": "12345",
  "domain": "Core:Text",
  "domain_version": "1.0.0",
  "data": {
    "text": "hello"
  },
  "send_time": 1700000005000,
  "mentions": ["67891"],
  "preview": "hello",
  "status": "sent"
}
```

协议边界：

- 顶层不得增加只服务某个 domain 的字段。
- 发送者昵称、头像等用户资料不是 canonical 消息字段；需要展示时由用户资料投影或客户端缓存解析 `uid`。
- 客户端忽略未知字段的兼容规则不代表服务端可以把 domain 专属字段提升到顶层。
- 同一条消息经创建响应、历史查询和 realtime 下发时，canonical 字段和值必须一致。

## 6. Domain Data 设计

### 6.1 `Core:Text`

```json
{
  "text": "hello",
  "link_preview": {
    "url": "https://example.com"
  }
}
```

约束：

- `text` 为必填字符串。
- `link_preview` 是 `Core:Text` 专属字段，因此位于 `data`。
- mentions 不进入 `data`，统一使用消息顶层 `mentions`。

### 6.2 `Core:ReplyText`

```json
{
  "content": {
    "text": "确实如此"
  },
  "reply_to_mid": "10086",
  "reply_to": {
    "mid": "10086",
    "sender_name": "Bob",
    "preview": "今天 meeting 改到 3 点",
    "created_at": 1700000000000,
    "unavailable": false
  },
  "quote_reply": {
    "mid": "10087",
    "uid": "67890",
    "preview": "下午 3 点"
  },
  "link_preview": {
    "url": "https://example.com"
  }
}
```

约束：

- `content` 固定表达 `Core:Text` payload，目前为 `{ "text": string }`。
- `reply_to_mid`、`reply_to`、`quote_reply`、`link_preview` 均为 `Core:ReplyText` 专属字段，只能位于 `data`。
- `reply_to_mid` 与 `reply_to.mid` 同时存在时必须一致。
- mentions 不进入 `data`。
- 至少存在 `reply_to_mid` 或 `quote_reply`，允许仅使用 `quote_reply`。

### 6.3 `Core:Forward`

单条转发：

```json
{
  "domain": "Core:Text",
  "domain_version": "1.0.0",
  "content": {
    "text": "看看这条"
  },
  "forwarded_from": {
    "mid": "10086",
    "cid": "99999",
    "uid": "54321",
    "preview": "原消息",
    "send_time": 1699999999000
  }
}
```

合并转发：

```json
{
  "domain": "Core:Text",
  "domain_version": "1.0.0",
  "content": {
    "text": "三则重要通知"
  },
  "forwarded_messages": [
    {
      "mid": "10086",
      "cid": "99999",
      "uid": "54321",
      "preview": "msg a",
      "send_time": 1699999999000
    },
    {
      "mid": "10087",
      "cid": "99998",
      "uid": "54322",
      "preview": "msg b",
      "send_time": 1699999999100
    }
  ]
}
```

约束：

- `domain`、`domain_version`、`content` 描述可选转发附言；当前只要求支持 `Core:Text`。
- `forwarded_from` 与 `forwarded_messages` 都是 `Core:Forward` 专属字段，只能位于 `data`。
- `forwarded_from` 与 `forwarded_messages` 不能同时存在。
- `forwarded_messages` 顺序必须与客户端请求顺序一致。
- 服务端必须使用真实源消息生成可解析来源快照，不信任客户端提供的来源详情。
- 合并转发至少包含两个 mid；不存在的来源保留 `{ "mid": "...", "unavailable": true }`，不丢弃整体请求。
- 合并转发有附言时 preview 使用附言；无附言时使用 `转发 N 条消息`。
- 每个可解析来源都校验调用者具有源频道可见性。

### 6.4 其它 Domain

`Core:File`、`Core:Voice`、系统消息和插件消息同样遵守：

- domain 身份由顶层 `domain / domain_version` 表达。
- 文件引用、语音时长、插件载荷、系统事件等专属字段全部进入 `data`。
- 不再使用 `body / payload / metadata` 组成第二套消息内容模型。

### 6.5 Domain 插件校验边界

创建消息时，任何入口提供的 `data` 都不能作为通用 JSON 直接写入数据库。客户端输入是不可信数据；内部入口生成的数据也必须经过对应 domain 插件校验。处理顺序固定为：

```text
校验通用 envelope 与 mentions
  -> message 调用 plugin.domain.api.MessageDomainPluginApi
  -> plugin feature 按 domain 查找唯一 ChannelMessagePlugin
  -> 插件校验 domain_version 与 data schema
  -> 插件返回规范化 data 和派生 preview
  -> 构建 canonical ChannelMessage
  -> 保存消息并在事务提交后发布 realtime 事件
```

插件契约：

- 每个插件必须声明唯一的 `supportedDomain()`；同一运行时不得注册重复 domain。
- `validateCanonicalData(...)` 必须把客户端 `data` 视为不可信数据，严格校验必填字段、JSON 类型、ID 表达和 domain version。
- 插件返回的是校验、复制并规范化后的 `data`，持久化层不得继续使用原始请求 Map。
- 未注册 domain、版本不受支持或 data 不合法时，必须在消息保存和 realtime 发布前失败。
- 插件只拥有 domain schema、规范化与 preview 派生规则，不拥有频道权限、事务、仓储、ID、时间和 realtime 投递。
- 新增消息 domain 只需实现并注册插件，不新增消息表字段，也不修改 HTTP 发布服务的 domain 分支。
- `Core:System` 与 `Core:Forward` 由内部专用入口构造 data，但仍经过各自插件校验；两者拒绝客户端通过通用 HTTP 发送。
- 不保留 `ChannelMessageDraft` 或插件 `createMessage` 第二套构建入口；所有新消息统一从 `domain + domain_version + data + mentions` 构建。

## 7. Mentions 元数据

`mentions` 是消息级通用元数据，语义是“该消息创建后需要提醒的用户 ID 集合”。

Wire 示例：

```json
{
  "mentions": ["67890", "67891"]
}
```

规则：

- 只保存用户 ID，不保存昵称、展示名或 target type。
- ID 必须是正数 Snowflake；Wire 与 JSON 存储使用十进制字符串。
- 服务端去重并保留首次出现顺序。
- 服务端必须校验提醒目标处于当前消息的可见或接收范围。
- mentions 缺失与空数组统一规范化为 `[]`。
- domain data 内不得重复出现 mentions。
- 消息撤回时 canonical mentions 清空，不再产生新的提醒。

### 7.1 派生提醒索引

若需要支持 mention 收件箱、未读状态和批量已读，允许保留 `chat_mention` 作为派生查询索引。

边界：

- `chat_message.mentions` 是消息元数据真源。
- `chat_mention` 只能由消息写入事务根据 canonical mentions 生成。
- 读取完整消息时不得依赖 `chat_mention` 反向重建 mentions。
- canonical 消息与派生索引必须在同一数据库事务中写入或清理。
- realtime `mention.created` 只能在事务提交后发布。

## 8. Preview

`preview` 是所有消息共有的服务端摘要字段。

规则：

- preview 由对应 domain 规则从 canonical data 派生。
- preview 可以持久化，避免列表和通知反复解析复杂 data。
- preview 不是消息内容真源，不允许用 preview 反向恢复 data。
- preview 的最大长度由数据库和 API 契约统一限制。
- 撤回后 preview 使用固定提示，例如 `消息已撤回`。

## 9. 生命周期与撤回

消息不提供编辑能力，生命周期只有：

```text
sent -> recalled
```

撤回规则：

- 保留 `message_id`、`sender_id`、`channel_id`、`domain`、`domain_version` 和 `send_time`。
- 将 `status` 更新为 `recalled`。
- 将 `data` 更新为空 object，避免历史接口继续泄露原内容。
- 将 `mentions` 更新为空数组，并清理派生 mention 索引。
- 将 `preview` 更新为固定撤回提示。
- recalled 消息不得命中内容搜索。
- 事务提交后发布 `message.recalled`，不发布 `message.updated`。
- 不保留 `edited_at`、`edit_version`，不提供消息编辑端点。

本文不把数据库维护性物理删除定义为用户可见消息生命周期。对外业务只能使用撤回语义。

## 10. 持久化目标模型

目标 `chat_message` 主表只持久化 canonical 消息字段：

```sql
CREATE TABLE chat_message (
    message_id BIGINT NOT NULL,
    sender_id BIGINT NOT NULL,
    channel_id BIGINT NOT NULL,
    domain VARCHAR(128) NOT NULL,
    domain_version VARCHAR(32) NOT NULL,
    data JSON NOT NULL,
    send_time DATETIME(6) NOT NULL,
    mentions JSON NOT NULL,
    preview VARCHAR(255) NOT NULL DEFAULT '',
    status VARCHAR(32) NOT NULL,
    PRIMARY KEY (message_id),
    CONSTRAINT fk_chat_message_channel FOREIGN KEY (channel_id) REFERENCES chat_channel (id),
    CONSTRAINT fk_chat_message_sender FOREIGN KEY (sender_id) REFERENCES auth_account (id)
);
```

建议索引：

```sql
CREATE INDEX idx_chat_message_channel_message
    ON chat_message (channel_id, message_id DESC);

CREATE INDEX idx_chat_message_channel_sender_message
    ON chat_message (channel_id, sender_id, message_id DESC);

CREATE INDEX idx_chat_message_channel_domain_message
    ON chat_message (channel_id, domain, message_id DESC);
```

JSON 列规则：

- `data` 始终保存 JSON object，不使用 SQL `NULL` 表达空 data。
- `mentions` 始终保存 JSON array，不使用 SQL `NULL` 表达无 mentions。
- mentions 元素使用十进制字符串，保持与 Wire ID 表达一致。

### 10.1 消息写操作幂等投影

幂等状态不属于消息内容，也不能进入 canonical message 或 domain `data`。转发写操作使用独立的 `chat_message_idempotency` 表：

```sql
CREATE TABLE chat_message_idempotency (
    account_id BIGINT NOT NULL,
    operation VARCHAR(64) CHARACTER SET ascii COLLATE ascii_bin NOT NULL,
    idempotency_key VARCHAR(128) CHARACTER SET utf8mb4 COLLATE utf8mb4_bin NOT NULL,
    request_fingerprint CHAR(64) CHARACTER SET ascii COLLATE ascii_bin NOT NULL,
    message_id BIGINT NULL,
    created_at DATETIME(6) NOT NULL,
    completed_at DATETIME(6) NULL,
    PRIMARY KEY (account_id, operation, idempotency_key)
);
```

事务规则：

- `account_id + operation + idempotency_key` 是幂等身份，键按二进制排序规则区分大小写。
- 请求先以原子 upsert 建立占位，再用 `SELECT ... FOR UPDATE` 锁定当前记录；禁止使用存在先查后插竞争窗口的实现。
- 请求指纹由规范化后的源消息、目标频道、附言和有序 `merged_mids` 计算，不包含 canonical message 字段。
- 同键同指纹且已有 `message_id` 时读取并返回原消息，不再次写消息或发布 realtime 事件。
- 同键不同指纹返回 `409 idempotency_key_reused`。
- 占位、消息写入、mention 派生和结果绑定必须位于同一事务，任何一步失败时整体回滚。
- 当前不配置过期窗口；幂等记录持续保留，以提供稳定的重复请求结果。

以下旧列不进入目标主表：

- `server_id`
- `conversation_id`
- `message_type`
- `body`
- `preview_text`
- `searchable_text`
- `payload`
- `metadata`
- `forwarded_from`
- `created_at`
- `edited_at`
- `edit_version`

其中旧 `created_at` 的消息发送语义由 `send_time` 明确承接。

### 10.2 搜索读模型

搜索所需的规范化文本属于派生查询数据，不进入 canonical 消息模型。

实现可以在 database-impl 内选择：

- 从受支持 domain 的 data 查询；或
- 维护独立的派生搜索投影。

无论采用哪种方式，都必须满足：

- 搜索数据由 canonical data 派生。
- 搜索数据不能反向覆盖消息 data 或 preview。
- recalled 消息不命中内容搜索。
- 具体数据库查询能力留在 `database-api / database-impl` 边界，不泄漏到 Wire 模型。

## 11. HTTP 与 Realtime 一致性

消息创建流程：

```text
HTTP 请求
  -> domain/version 对应规则校验 data
  -> 规范化 mentions
  -> 生成 preview
  -> 保存 canonical message
  -> 生成派生 mention/search 投影
  -> transaction commit
  -> 发布 message.created / mention.created
```

要求：

- 创建 HTTP 响应、历史查询与 realtime 使用同一套 canonical 消息投影规则。
- realtime 不得单独根据 `body / payload / forwarded_from` 重建另一份消息。
- 当前 `TransactionRunner + afterCommit` 边界继续保留。
- WS event envelope 是否调整不影响 payload 内消息必须符合本文模型。

## 12. 旧模型迁移原则

| 旧字段 | 目标处理 |
| --- | --- |
| `message_type` | 转换为完整 `domain` |
| 固定补出的版本 | 显式保存为 `domain_version` |
| `body` | 按原消息类型迁入 `data` |
| `payload` | 解析并合并到 `data`，不得嵌套第二套 envelope |
| `metadata` | 按真实语义迁入对应 domain data；无语义数据丢弃 |
| `mentions` 对象数组 | 转换为用户 ID 字符串数组 |
| `forwarded_from` | 迁入 `Core:Forward.data.forwarded_from` |
| `preview_text` | 转换为 `preview` |
| `created_at` | 转换为 `send_time` |
| `edited_at / edit_version` | 删除，不进入目标模型 |

迁移要求：

- 不长期保留新旧字段双写。
- 不通过旧 body 或 payload 反向覆盖已经生成的 canonical data。
- 历史消息若无法可靠转换，应形成明确的迁移失败记录，不静默伪造 domain data。
- 本项目处于重写式重构阶段，不默认维护旧外部协议字段回显。

## 13. 模块边界

- `chat-domain/features/message/domain/model` 持有 canonical 消息领域模型。
- message domain 对 envelope、mentions、preview 投影和状态转换负责。
- `domain/repository` 定义消息业务语义仓储。
- `chat-domain/features/message/support/persistence` 只负责领域模型与 database-api 最小投影转换。
- `database-api` 只表达持久化所需字段，不复制 ReplyText、Forward 等业务协议对象。
- `database-impl` 负责 JSON、SQL、索引与派生读模型实现，不承载 domain 校验规则。
- controller 只处理 Wire 与领域命令转换，不直接决定 domain 业务语义。
- `chat-domain/features/plugin/domain/extension/ChannelMessagePlugin` 是所有消息 domain data 校验与消息类型扩展 SPI；它只在 plugin feature 内使用。
- `chat-domain/features/plugin/domain/api/MessageDomainPluginApi` 是 message 发布服务的唯一插件校验入口，API 命令与结果不暴露插件实例或注册表。
- `chat-domain/features/plugin/support/message` 承载 ReplyText、Forward、Text、File、Voice、System、Custom 等内建消息插件。
- 插件目录、domain 目录和 required-plugin gate 由 plugin feature 管理；server/auth 仅消费 `PluginCatalogApi`。
- 附件 key 构建与 scope 校验由 file feature 的 `FileReferenceApi` 提供，plugin 不依赖 message 内部服务。
- server feature 只负责通用 realtime envelope、缓存与投递，不依赖 message 内部模型。

本设计不引入通用 `domain/port`；继续遵守项目既有 `domain/api`、`domain/repository`、`domain/capability` 和 `domain/extension` 边界。

## 14. 协议决策

- `Core:ReplyText` 至少包含 `reply_to_mid` 或 `quote_reply`，允许仅提供 `quote_reply`；`reply_to` 存在时必须同时提供一致的 `reply_to_mid`。
- 校验失败沿用项目统一的 HTTP `422 Unprocessable Entity`。
- 合并转发至少包含两个 mid，单项 `merged_mids` 返回校验失败。
- 不存在的合并转发来源保留 mid 并标记 `unavailable = true`；可解析来源必须通过源频道可见性校验。
- 合并转发有附言时 preview 使用附言，无附言时使用 `转发 N 条消息`。
- `Idempotency-Key` header 与 body `idempotency_key` 同时存在时必须一致；同键同请求返回首次消息，同键不同请求返回 `409 idempotency_key_reused`。
- 幂等记录使用独立持久化投影，不进入 canonical message；唯一键占位、消息写入与结果绑定必须位于同一事务。
- realtime 保持当前 `type = event` envelope，payload 内消息使用 canonical 模型。
- 不保留旧消息字段或旧 WS 发消息入口的兼容链。

## 15. 验收标准

- 领域、Wire 和数据库主投影均使用本文十个 canonical 字段。
- `domain + domain_version + data` 可以无损往返。
- 所有 domain 专属字段只位于 `data`。
- 所有消息创建入口都经过已注册 domain 插件校验，不存在 draft 或直接构造消息的旁路。
- mentions 只表达提醒用户 ID 数组，并能生成一致的提醒索引。
- preview 和搜索数据均为单向派生数据。
- 消息只存在 `sent / recalled` 两态，不存在编辑 API 和编辑版本字段。
- HTTP 创建、历史查询和 realtime 对同一消息返回一致 canonical 内容。
- ReplyText、Forward、文件、语音和插件消息都遵守同一 envelope。
- HTTP 消息的 domain version 与 data 必须先经对应注册插件校验，失败时不写库、不发布事件。
- 新增插件 domain 不需要修改发布服务和数据库结构。
- 带幂等键的重复转发只创建一条消息并只发布一次 `message.created`，事务失败不残留幂等占位。
