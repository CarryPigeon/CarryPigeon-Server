# 项目异常与错误码规范

## 1. 文档目的

本文用于统一项目中的异常设计、错误码设计与响应码映射规则。

本文只固化当前重写阶段已经明确的异常边界，不扩展未确认的复杂异常体系。

## 2. 总原则

- 异常先表达语义，再表达技术细节
- 错误码先服务定位与检索，再服务展示
- 基础设施异常、业务异常、协议响应异常必须分层
- 低层异常不得直接污染上层语义
- 对外返回必须稳定，内部错误细节不应直接泄露

## 3. 分层规则

### 3.1 `infrastructure-basic`

`infrastructure-basic` 只负责固定基础设施异常。

当前已存在的基础设施异常能力：

- `InfrastructureException`
- `InfrastructureErrorCode`

职责：

- 表达日志、配置、JSON、ID、时间等固定基础设施执行失败
- 提供稳定、可检索的错误标识
- 为上层统一包装提供基础异常语义

边界：

- 不用于表达业务规则失败
- 不直接作为最终对外响应语义

### 3.2 `chat-domain`

`chat-domain` 负责表达业务失败语义。

职责：

- 表达参数错误、资源不存在、权限不足、业务规则不满足等业务问题
- 在业务入口处决定当前失败属于哪一类业务响应

边界：

- 业务异常不应直接承载底层中间件细节
- 业务异常不应绕过统一响应约定

### 3.3 协议响应层

协议响应层负责把内部异常语义映射为稳定的对外结果。

当前项目上下文中已经存在 `CPResponse.code` 约定，测试要求已明确需要覆盖：

- `100`
- `200`
- `300`
- `404`
- `500`

原则：

- 对外只暴露稳定响应码与必要消息
- 内部异常堆栈进入日志，不直接进入外部响应

## 4. 基础设施异常规范

### 4.1 使用场景

以下场景应优先使用 `InfrastructureException`：

- JSON 序列化失败
- JSON 反序列化失败
- 配置绑定或配置校验失败
- ID 生成失败
- 时间能力访问失败

### 4.2 错误码规则

`InfrastructureErrorCode` 用于表达固定基础设施错误类别。

当前已存在错误码：

- `JSON_SERIALIZE_FAILED`
- `JSON_DESERIALIZE_FAILED`
- `CONFIG_BIND_FAILED`
- `ID_GENERATE_FAILED`
- `TIME_ACCESS_FAILED`

规则：

- 错误码必须稳定、可搜索、可读
- 错误码名称统一使用全大写加下划线
- 一个错误码只表达一类稳定问题
- 不允许把错误消息本身当错误码

### 4.3 抛出规则

基础设施代码抛出异常时，应优先满足以下要求：

- 尽量携带 `InfrastructureErrorCode`
- 错误消息必须可读，能直接指出失败点
- 若存在原始异常，应保留 `cause`

推荐：

- `new InfrastructureException(InfrastructureErrorCode.CONFIG_BIND_FAILED, "workerId must be between 0 and 31")`

不推荐：

- `throw new RuntimeException("error")`
- `throw new Exception("json fail")`

## 5. 业务异常规范

### 5.1 业务失败类型

业务层失败至少应区分以下几类：

- 参数错误
- 权限不足
- 资源不存在
- 业务规则不满足
- 系统内部失败

### 5.2 业务异常边界

业务异常应表达业务语义，而不是技术语义。

推荐表达：

- 用户输入不合法
- 当前对象不存在
- 当前用户无权限执行操作
- 当前状态不允许继续流程

不推荐直接向上暴露：

- SQL 异常名
- JSON 库异常名
- 第三方 SDK 异常名

### 5.3 组件链路中的中断

结合当前项目已有说明，组件执行失败时应通过统一问题异常中断链路。

当前已明确的业务问题类型包括：

- 参数错误：`validationFailed()`
- 资源不存在：`notFound(message)`
- 权限不足：`forbidden(reason, message)`
- 其他业务错误：`fail(...)`

规则：

- 参数与上下文校验失败应尽早中断
- 中断原因必须与当前失败语义一致
- 不允许所有失败都统一粗暴映射为内部错误

## 6. 响应码映射规则

当前项目对外响应以 `CPResponse.code` 为稳定约定。

在当前重写阶段，先固化最小映射原则：

- `100`：成功
- `200`：参数或校验类失败
- `300`：权限、认证或拒绝类失败
- `404`：资源不存在
- `500`：系统内部错误

说明：

- 上述映射用于统一当前代码与测试的最小判断边界
- 后续若要细化响应码语义，应在此文档上增量补充，而不是推翻现有边界

## 7. 转换规则

### 7.1 基础设施异常向上转换

`InfrastructureException` 到上层时，不应直接原样暴露给外部。

推荐处理方式：

- 记录错误码、错误消息、原始异常
- 在业务入口或统一异常处理处转换为稳定对外响应
- 对外通常落为 `500`，除非当前异常已明确可映射到更具体的业务语义

### 7.2 业务异常向响应转换

业务异常应尽量在统一入口转换成稳定的 `CPResponse.code`。

要求：

- 相同业务问题返回相同类别响应码
- 不能同一类问题今天返回 `200`，明天返回 `500`
- 转换点应集中，不要散落在各处手写

## 8. 错误消息规则

- 面向日志的消息可以更技术化，但必须可检索
- 面向外部响应的消息应稳定、简洁、避免泄露内部细节
- 消息必须与错误码语义一致
- 不能用空字符串或无意义消息替代真实语义

推荐：

- `channel does not exist`
- `workerId must be between 0 and 31`

不推荐：

- `error`
- `fail`
- `system wrong`

## 9. 测试规则

异常与错误码相关测试至少应覆盖：

- 成功路径
- 对应失败路径
- 响应码是否符合约定
- 错误码是否符合预期
- 原始异常是否被保留

当前测试约束中，业务逻辑至少应覆盖：

- `CPResponse.code` 为 `100`
- `CPResponse.code` 为 `200`
- `CPResponse.code` 为 `300`
- `CPResponse.code` 为 `404`
- `CPResponse.code` 为 `500`

基础设施测试至少应覆盖：

- `InfrastructureErrorCode` 是否正确携带
- `InfrastructureException` 是否保留 `cause`

## 10. 禁止事项

- 禁止直接抛裸 `RuntimeException` 作为长期方案
- 禁止基础设施异常直接充当业务异常
- 禁止把外部中间件错误直接返回给客户端
- 禁止同类错误使用多套错误码或多套响应码
- 禁止只定义异常类，不定义其使用边界
- 禁止错误码名称含糊、不可检索

## 11. 当前阶段结论

当前项目在异常体系上已具备第一版基础骨架：

- 基础设施层已有统一异常与错误码
- 业务层已有明确的问题中断类型
- 对外层已有稳定的 `CPResponse.code` 约定

后续若继续重构业务域，应优先补齐：

- 统一业务异常类型
- 统一异常处理入口
- 统一业务错误码命名规则

但这些补充必须建立在本文当前边界之上渐进推进。
