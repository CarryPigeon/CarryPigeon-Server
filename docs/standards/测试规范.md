# 项目测试规范

## 1. 文档目的

本文用于统一项目中的测试分层、测试命名、测试注释与测试边界。

本文只固化当前重写阶段已经明确的测试约束，不扩展未确认的复杂测试体系。

## 2. 总原则

- 测试服务于稳定重构，不服务于形式化堆数量
- 测试优先覆盖稳定契约，不优先覆盖易碎实现细节
- 测试边界必须和模块边界一致
- 每个测试都应回答“当前在验证什么契约”
- 测试代码同样必须可读、可维护、可长期迭代

## 3. 测试目录规则

- 测试代码统一放在各模块自己的 `src/test/java`
- 测试文件应与被测包结构尽量保持邻近
- 不允许把一个模块的测试放到另一个模块中

当前示例：

- `infrastructure-basic/src/test/java/team/carrypigeon/backend/infrastructure/basic/...`

### 3.1 mock 测试与真实业务链路测试分包

当同一协议入口既需要 mock 协议测试，又需要真实业务链路测试时，必须明确分包：

- mock 协议测试放在被测入口邻近包的 `mock` 子包中
  - 示例：`chat-domain/src/test/java/team/carrypigeon/backend/chat/domain/features/auth/controller/http/mock/AuthControllerTests.java`
  - 目标：验证 HTTP 请求、响应、错误码、DTO 到 command/query 的映射
  - 边界：领域 API、外部服务或仓储使用 Mockito、假对象或最小替身
- 真实业务链路测试放在 feature 下的 `chain` 包中
  - 示例：`chat-domain/src/test/java/team/carrypigeon/backend/chat/domain/features/auth/chain/AuthUserBusinessChainTests.java`
  - 目标：验证 controller、真实领域 API、认证上下文、领域仓储抽象替身之间的最小业务闭环
  - 边界：不直接依赖 `*-impl`，不连接 MySQL、Redis、MinIO、SMTP 等真实外部服务

分包原则：

- 不为了分包移动生产代码。
- 不把 `chat-domain` 测试改成依赖 `infrastructure-service/*-impl`。
- `mock` 测试关注协议边界，`chain` 测试关注真实业务协作链路。
- 若需要验证真实数据库、Redis、MinIO 或 SMTP 行为，应放在对应 `*-impl` 或 `application-starter` 的集成/装配测试中，不放进 `chat-domain`。

## 4. 测试命名规则

### 4.1 测试类命名

测试类统一使用：

- `<Name>Tests`

当前示例：

- `JsonsTests`
- `InfrastructureBasicsTests`
- `SnowflakeIdGeneratorTests`

### 4.2 测试方法命名

测试方法统一使用：

- `methodName_condition_expected()`

要求：

- 方法名必须能直接看出被测行为、条件和预期
- 禁止使用无语义命名，例如 `test1`、`shouldWork`

当前示例：

- `toJson_map_returnsJsonString()`
- `fromJson_invalidJson_throwsInfrastructureException()`

## 5. 测试分层规则

### 5.1 `infrastructure-basic`

`infrastructure-basic` 重点验证固定基础设施契约。

至少应覆盖：

- 配置绑定与配置校验
- ID 生成能力
- JSON 序列化与反序列化
- 时间能力
- 日志上下文工具
- 基础异常与错误码
- 对外门面聚合行为

原则：

- 重点测“项目封装后的契约”
- 不对第三方框架内部实现做重复性测试

### 5.2 `chat-domain`

`chat-domain` 重点验证业务语义与业务边界。

优先覆盖：

- 领域服务业务方法
- 领域规则
- 仓储抽象交互
- 协议入口的基本输入输出约束
- 业务异常与 HTTP / 错误响应映射

原则：

- 测业务语义，不测无意义框架细节
- 测成功路径，也测失败路径

### 5.3 `application-starter`

`application-starter` 重点验证启动与装配。

适合覆盖：

- Spring 容器能否启动
- 自动配置是否生效
- 关键基础 Bean 是否可装配

不适合覆盖：

- 大量业务规则细节
- 重复验证下游模块已经覆盖的契约

### 5.4 `infrastructure-service`

后续外部服务模块建立后：

- `*-api` 优先验证契约稳定性
- `*-impl` 优先验证具体实现行为、失败处理与适配边界

## 6. 当前阶段推荐测试类型

当前重写阶段优先六类常规测试：

- 单元测试（`unit`）
- 模块内契约测试（`contract`）
- 最小启动测试（`smoke`）
- 高风险链路回归测试（`regression`）
- mock 协议测试（`mock`）
- 真实业务链路测试（`business`）

真实环境测试作为受控补充能力存在：

- 真实环境集成测试（`env` / `env-*`）

当前不建议过早铺开：

- 大量端到端测试
- 大量依赖真实外部服务的集成测试
- 为未来功能预写的大量占位测试

### 6.1 测试分级定义

#### `unit`

用于验证纯对象、纯规则、纯值对象、纯配置对象或固定基础能力的最小契约。

特征：

- 不依赖 Spring 容器启动
- 不依赖真实外部服务
- 执行快、定位直接、失败语义清晰

示例方向：

- JSON 工具
- 时间能力
- ID 生成能力
- 配置对象校验
- 值对象行为

#### `contract`

用于验证模块边界上的稳定契约。

特征：

- 重点关注输入输出、响应码、异常映射、抽象交互与适配边界
- 可使用最小替身、`MockMvcBuilders.standaloneSetup`、`ApplicationContextRunner` 或最小适配器装配
- 不以复刻框架内部行为为目标

示例方向：

- Controller 协议测试
- 全局异常映射测试
- 仓储抽象与适配器交互测试
- 自动配置条件装配测试

#### `smoke`

用于验证当前核心链路的最小可运行或最小可装配状态。

特征：

- 只测最关键的启动和装配稳定面
- 目标是快速发现“大面积不可用”问题
- 不扩展为细节业务回归集合

示例方向：

- starter 关键 Bean 是否可装配
- 关键初始化检查链路是否能建立

#### `regression`

用于验证历史问题、复杂链路或高风险修复点不会回退。

特征：

- 通常比 `unit` / `contract` 更长、更跨层
- 必须能明确说出“在防止什么回归”
- 不允许把普通测试随意命名成 regression

示例方向：

- 历史缺陷修复后的回归链路
- 附件、鉴权、消息编排等跨层关键路径

#### `mock`

用于验证协议入口在 mock 或替身隔离下的稳定边界。

特征：

- 通常使用 Mockito、手写假对象或 `MockMvcBuilders.standaloneSetup`
- 重点验证 HTTP 状态码、错误响应、DTO 字段映射、command/query 入参映射
- 不验证真实领域 API 的内部业务编排
- 不连接真实外部服务

示例方向：

- Controller 请求字段到 command/query 的映射
- Controller 统一错误响应
- 协议层参数校验
- 外部端口异常到 HTTP 响应的转换

#### `business`

用于验证不依赖真实外部服务的真实业务链路。

特征：

- 尽量使用真实 controller、真实领域 API、真实认证上下文和真实领域模型
- 外部服务、数据库和对象存储仍使用内存替身或确定性测试替身
- 重点验证跨层协作后的业务状态变化，而不只验证单点返回值
- 不替代 `application-starter` 的 Spring 容器装配测试，也不替代 `*-impl` 的真实适配实现测试

示例方向：

- `HTTP -> Controller -> Domain API -> Repository 替身` 的最小闭环
- 鉴权 token 签发后访问受保护资源
- 注册后资料初始化、默认频道成员关系建立
- refresh token 轮换与撤销状态变化

#### `env`

用于验证真实外部服务或完整运行环境下的适配行为。

特征：

- 必须连接真实 MySQL、Redis、MinIO、SMTP 或完整应用运行环境中的至少一种
- 必须在测试开始前主动注入本用例所需数据，不依赖人工预置数据
- 必须使用唯一命名空间隔离数据，并在测试结束后清理
- 默认开发命令不应隐式执行，必须通过 `test.groups=env` 或具体 `env-*` 标签显式选择

推荐子标签：

- `env-db`：真实数据库与 MyBatis/JDBC 适配行为
- `env-cache`：真实 Redis 缓存行为
- `env-storage`：真实 MinIO 或对象存储行为
- `env-mail`：真实 SMTP 或邮件服务行为
- `env-app`：真实 Spring Boot 应用装配与跨外部服务链路

示例方向：

- 数据库唯一键、事务、分页、时间字段与 SQL 条件真实行为
- Redis TTL、序列化、删除和连接失败恢复行为
- MinIO bucket、object key、上传、下载、删除与错误映射
- SMTP 健康检查与发送失败映射
- 应用启动后通过真实配置访问真实外部服务的最小闭环

### 6.2 分级使用规则

- 首选在测试类级别使用 `@Tag` 标记主分级
- 单个测试类默认只保留一个主分级，避免同时承担多个核心职责
- 若一个测试类无法清晰归入某一级，优先拆分，而不是叠加多个主标签
- 分级用于执行选择和审查归类，不替代测试命名、注释和断言质量
- 真实环境测试可以同时标记 `@Tag("env")` 和一个具体 `@Tag("env-*")`，其中 `env` 用于总体选择，`env-*` 用于按外部服务选择

推荐：

- `@Tag("unit")`
- `@Tag("contract")`
- `@Tag("smoke")`
- `@Tag("regression")`
- `@Tag("mock")`
- `@Tag("business")`
- `@Tag("env")`
- `@Tag("env-db")`
- `@Tag("env-cache")`
- `@Tag("env-storage")`
- `@Tag("env-mail")`
- `@Tag("env-app")`

不推荐：

- 自行发明未经项目确认的新标签
- 用标签掩盖测试边界不清的问题

### 6.3 三层测试模型

当前项目按三层模型组织和审查测试。三层模型是测试职责模型，底层执行仍使用 JUnit 5 `@Tag` 和 Maven `test.groups` / `test.excludedGroups`。

#### 单方法功能测试

目标：验证单个类、单个方法、单个边界适配或单个协议入口的稳定契约。

对应标签：

- `unit`
- `contract`
- `mock`
- 单点高风险补测可使用 `regression`

边界：

- 不连接真实 MySQL、Redis、MinIO、SMTP 等外部服务
- 可以使用 Mockito、手写假对象、内存替身、`ApplicationContextRunner` 或 `MockMvcBuilders.standaloneSetup`
- 适合放在被测类邻近包，协议 mock 测试放在 `mock` 子包

#### 调用链测试

目标：验证不连接真实外部服务时，业务入口到领域 API、领域模型、仓储抽象替身之间的真实业务协作是否正确。

对应标签：

- `business`

边界：

- 主要放在 `chat-domain/src/test/java/.../features/{feature}/chain`
- 使用真实 controller、真实领域 API、真实认证上下文和真实领域模型
- 数据库、缓存、对象存储、邮件等外部服务必须使用确定性内存替身
- 不允许让 `chat-domain` 测试依赖 `infrastructure-service/*-impl`

#### 真实环境测试

目标：验证真实外部服务、真实配置或完整应用装配下的适配行为，补足 mock 与业务链路无法证明的运行环境风险。

对应标签：

- `env`
- `env-db`
- `env-cache`
- `env-storage`
- `env-mail`
- `env-app`

边界：

- `infrastructure-service/*-impl` 负责各外部服务实现的真实环境测试
- `application-starter` 负责真实应用装配和跨服务最小链路测试
- `chat-domain` 不放真实环境测试
- 真实环境测试默认不作为快速开发命令的一部分，必须显式选择标签运行

## 7. 断言规则

- 断言必须直接对应测试目标
- 一个测试只聚焦一个主要契约
- 失败断言必须验证失败语义，而不只是“抛了异常”

推荐断言：

- 返回值是否符合预期
- 错误码是否正确
- 异常类型是否正确
- 原始 `cause` 是否保留
- HTTP 状态码与错误响应是否符合统一约定

不推荐：

- 只断言方法执行过
- 只断言对象非空但不验证语义

## 8. 响应码测试规则

业务测试必须覆盖成功与失败路径。

当前代码基线下，应按两类协议分别验证：

- 对成功接口，验证直接资源对象或 `204 No Content`
- 对失败接口，验证 `HTTP status + ApiErrorResponse`

要求：

- 成功场景验证成功响应是否符合当前控制器契约
- 参数错误场景验证 `422`
- 认证失败场景验证 `401`
- 权限失败场景验证 `403`
- 资源不存在场景验证 `404`
- 冲突场景验证 `409`
- 前置条件失败场景在适用时验证 `412`
- 系统异常场景验证 `500`
- 错误场景同时验证 `ApiErrorResponse.error.reason` 与关键消息语义

## 9. 异常测试规则

异常相关测试至少应验证：

- 抛出的异常类型
- 错误码是否正确
- 错误消息是否可读
- 原始 `cause` 是否保留

当前基础设施异常测试应重点覆盖：

- `InfrastructureException`
- `InfrastructureErrorCode`

## 10. 测试数据规则

- 测试数据应最小化，只保留当前场景需要的输入
- 测试数据命名应直接表达语义
- 不使用含糊的魔法值堆砌测试场景
- 能直接内联表达的简单数据，不必额外引入复杂夹具
- 真实环境测试必须在测试开始前主动创建所需数据，不能依赖人工预置数据

推荐：

- `Map.of("name", "carry")`
- `workerId = 1`

不推荐：

- 大量无语义字符串
- 与当前断言无关的大对象初始化

### 10.1 真实环境测试数据注入规则

真实环境测试必须按“准备数据 -> 执行行为 -> 断言结果 -> 清理数据”的生命周期组织。

数据准备要求：

- 每个测试用例必须创建自己的测试数据，不读取或复用人工维护的固定业务数据
- 测试数据必须带唯一命名空间，推荐格式为 `it_{时间戳或随机段}_{caseName}`
- 数据命名必须能从真实环境中识别来源，例如用户邮箱、缓存 key、对象 key、bucket 前缀均带命名空间
- 数据创建失败时应快速失败，不能跳过断言或静默降级为 mock 行为

清理要求：

- 每个测试创建的数据都必须注册清理动作
- 清理动作按创建依赖的反向顺序执行，例如先删子表或对象，再删主记录
- 清理失败应暴露为测试失败或在测试日志中明确标出，不允许吞掉真实环境污染风险
- 若真实环境本身不允许删除，应使用专用测试命名空间和 TTL，并在测试注释中说明限制
- 真实数据库环境测试可使用 `infrastructure-service/database-impl/src/test/java/.../testsupport/EnvTestDataScope` 生成命名空间并注册清理动作；其它 `*-impl` 或 `application-starter` 模块如需同类能力，应在本模块测试包内保留测试侧 helper，不新增生产依赖

环境要求：

- `env-*` 测试必须明确说明需要哪些环境变量、配置或 Docker 服务
- 环境变量缺失时允许使用 JUnit assumptions 明确跳过，但不能伪造真实外部服务继续执行
- 真实数据库测试应优先使用专用测试库或专用 schema，不使用生产库
- 真实对象存储、缓存、邮件测试必须使用专用测试 bucket、key 前缀、database index 或收件目标

当前真实环境测试开关：

- `env-db`
  - 开关：`CP_ENV_DB_TEST_ENABLED=true` 或 `-Dcp.env.db.test.enabled=true`
  - 默认连接：`127.0.0.1:3306/carrypigeon`
  - 可覆盖：`CP_ENV_DB_URL`、`CP_ENV_DB_HOST`、`CP_ENV_DB_PORT`、`CP_ENV_DB_DATABASE`、`CP_ENV_DB_USERNAME`、`CP_ENV_DB_PASSWORD`
  - 前置：先执行 `docs/sql/00-all-in-one.sql` 初始化表结构
- `env-cache`
  - 开关：`CP_ENV_CACHE_TEST_ENABLED=true` 或 `-Dcp.env.cache.test.enabled=true`
  - 默认连接：`127.0.0.1:6379`，密码 `carrypigeon123`，database `0`
  - 可覆盖：`CP_ENV_CACHE_HOST`、`CP_ENV_CACHE_PORT`、`CP_ENV_CACHE_PASSWORD`、`CP_ENV_CACHE_DATABASE`
- `env-storage`
  - 开关：`CP_ENV_STORAGE_TEST_ENABLED=true` 或 `-Dcp.env.storage.test.enabled=true`
  - 默认连接：`http://127.0.0.1:9000`，access key `carrypigeon`，secret key `carrypigeon123`，bucket `carrypigeon`
  - 可覆盖：`CP_ENV_STORAGE_ENDPOINT`、`CP_ENV_STORAGE_ACCESS_KEY`、`CP_ENV_STORAGE_SECRET_KEY`、`CP_ENV_STORAGE_BUCKET`
  - 前置：确保 bucket 已由 `minio-init` 或人工命令创建
- `env-mail`
  - 开关：`CP_ENV_MAIL_TEST_ENABLED=true` 或 `-Dcp.env.mail.test.enabled=true`
  - 必填：`CP_ENV_MAIL_HOST`
  - 可覆盖：`CP_ENV_MAIL_PORT`、`CP_ENV_MAIL_USERNAME`、`CP_ENV_MAIL_PASSWORD`、`CP_ENV_MAIL_AUTH`、`CP_ENV_MAIL_STARTTLS`
  - 当前只验证 SMTP 健康连接，不发送邮件
- `env-app`
  - 开关：`CP_ENV_APP_TEST_ENABLED=true` 或 `-Dcp.env.app.test.enabled=true`
  - 默认连接：MySQL `127.0.0.1:3306/carrypigeon`，Redis `127.0.0.1:6379`，MinIO `http://127.0.0.1:9000`
  - 可覆盖：`CP_ENV_APP_MYSQL_HOST`、`CP_ENV_APP_MYSQL_PORT`、`CP_ENV_APP_MYSQL_DATABASE`、`CP_ENV_APP_MYSQL_USERNAME`、`CP_ENV_APP_MYSQL_PASSWORD`
  - 可覆盖：`CP_ENV_APP_REDIS_HOST`、`CP_ENV_APP_REDIS_PORT`、`CP_ENV_APP_REDIS_PASSWORD`
  - 可覆盖：`CP_ENV_APP_MINIO_ENDPOINT`、`CP_ENV_APP_MINIO_ACCESS_KEY`、`CP_ENV_APP_MINIO_SECRET_KEY`、`CP_ENV_APP_MINIO_BUCKET`
  - 当前验证真实 Spring Boot 应用装配与外部服务健康 Bean 暴露；业务数据读写闭环由各 `infrastructure-service/*-impl` 的 `env-*` 测试负责

## 11. Mock 与替身规则

- 只有在隔离边界、降低外部依赖时才使用 Mock
- Mock 应服务于验证当前模块契约，不应用来掩盖设计问题
- 能使用简单假对象或最小真实对象时，优先不用复杂 Mock

原则：

- `chat-domain` 测仓储抽象与外部服务抽象时可以使用替身
- `infrastructure-basic` 多数场景优先使用真实基础对象

## 12. 注释规则

测试代码必须遵守 `docs/standards/注释规范.md`。

测试类必须说明：

- 被测对象
- 验证的核心契约
- 不验证什么

测试方法必须说明：

- 当前测的是什么
- 输入条件是什么
- 期望结果是什么

禁止：

- 只有测试代码没有注释说明
- 只有“测试某方法”这类无信息注释

## 13. 测试状态标记规则

### 13.1 `@Disabled`

只有在当前测试已知不可执行、存在外部阻塞或需要短期冻结时，才允许使用 `@Disabled`。

要求：

- 必须写清楚禁用原因
- 原因必须可读、可检索、可执行
- 应说明恢复条件或阻塞点
- 禁止无说明禁用测试

推荐：

- `@Disabled("blocked by missing MinIO test container setup; enable after storage integration harness is available")`

不推荐：

- `@Disabled`
- `@Disabled("暂时不用")`

### 13.2 测试状态注释

测试代码中的临时状态说明只允许使用以下标记：

- `TODO:` 后续计划补测、拆分或完善的具体动作
- `FIXME:` 当前已知失真、脆弱点或待修复问题
- `NOTE:` 关键边界说明或选择原因
- `WARNING:` 高风险依赖、环境限制或显著风险提示

要求：

- 必须写清楚动作目标或风险语义
- 不得使用空泛表述
- 不得把长期规则只留在测试注释中而不回写 `docs/`

## 14. 覆盖率与执行规则

### 14.1 覆盖率原则

- 覆盖率用于发现测试盲区，不用于替代契约判断
- 禁止为追求覆盖率编写无语义测试
- 覆盖率门禁应保守起步，并随着模块成熟度逐步收紧
- 覆盖率结果应结合模块职责、测试分级和失败风险一起解读

### 14.2 当前执行入口

默认执行：

- `mvn test -DskipTests=false`
- `mvn verify -DskipTests=false`

按分级筛选：

- `mvn test -DskipTests=false -Dtest.groups=unit`
- `mvn test -DskipTests=false -Dtest.groups=contract`
- `mvn test -DskipTests=false -Dtest.groups=smoke`
- `mvn test -DskipTests=false -Dtest.groups=regression`
- `mvn test -DskipTests=false -Dtest.groups=mock`
- `mvn test -DskipTests=false -Dtest.groups=business`
- `mvn test -DskipTests=false -Dtest.groups=env`
- `mvn test -DskipTests=false -Dtest.groups=env-db`
- `mvn test -DskipTests=false -Dtest.groups=env-cache`
- `mvn test -DskipTests=false -Dtest.groups=env-storage`
- `mvn test -DskipTests=false -Dtest.groups=env-mail`
- `mvn test -DskipTests=false -Dtest.groups=env-app`

按模块与分级组合：

- `mvn -pl application-starter -am test -DskipTests=false -Dtest.groups=smoke`
- `mvn -pl chat-domain -am test -DskipTests=false -Dtest.groups=mock`
- `mvn -pl chat-domain -am test -DskipTests=false -Dtest.groups=business`
- `mvn -pl infrastructure-service/database-impl -am test -DskipTests=false -Dtest='*EnvTests' -Dtest.groups=env-db -Dsurefire.failIfNoSpecifiedTests=false`
- `mvn -pl infrastructure-service/cache-impl -am test -DskipTests=false -Dtest='*EnvTests' -Dtest.groups=env-cache -Dsurefire.failIfNoSpecifiedTests=false`
- `mvn -pl infrastructure-service/storage-impl -am test -DskipTests=false -Dtest='*EnvTests' -Dtest.groups=env-storage -Dsurefire.failIfNoSpecifiedTests=false`
- `mvn -pl infrastructure-service/mail-impl -am test -DskipTests=false -Dtest='*EnvTests' -Dtest.groups=env-mail -Dsurefire.failIfNoSpecifiedTests=false`
- `mvn -pl application-starter -am test -DskipTests=false -Dtest='*EnvTests' -Dtest.groups=env-app -Dsurefire.failIfNoSpecifiedTests=false`

按三层模型筛选：

- 单方法功能测试：
  - `mvn test -DskipTests=false -Dtest.excludedGroups=business,env,env-db,env-cache,env-storage,env-mail,env-app`
  - 用途：运行默认不依赖真实外部服务的单点功能、契约、mock、smoke 与 regression 测试。
- 调用链测试：
  - `mvn -pl chat-domain -am test -DskipTests=false -Dtest.groups=business`
  - 用途：运行 `chat-domain` 中不连接真实外部服务的真实业务调用链测试。
- 真实环境测试：
  - `mvn test -DskipTests=false -Dtest.groups=env`
  - 用途：显式运行所有真实环境测试；执行前必须准备 `config/application.yaml`、Docker 服务或通过系统环境变量 / JVM 参数显式提供目标测试环境。

常用开发入口：

- `mvn -pl chat-domain -am test -DskipTests=false`
  - 用途：验证 `chat-domain` 及其依赖模块的完整测试集合。
- `mvn -pl chat-domain -am test -DskipTests=false -Dtest.groups=mock`
  - 用途：只跑 `chat-domain` 中按 `@Tag("mock")` 标记的协议/mock 测试。
- `mvn -pl chat-domain -am test -DskipTests=false -Dtest.groups=business`
  - 用途：只跑 `chat-domain` 中按 `@Tag("business")` 标记的真实业务链路测试。
- `mvn test -DskipTests=false -Dtest.excludedGroups=business,env,env-db,env-cache,env-storage,env-mail,env-app`
  - 用途：在需要快速排除较长业务链路测试和真实环境测试时运行其余测试。
- `mvn -pl infrastructure-service/database-impl -am test -DskipTests=false -Dtest='*EnvTests' -Dtest.groups=env-db -Dsurefire.failIfNoSpecifiedTests=false`
  - 用途：只运行真实数据库适配测试；执行前必须准备测试数据库，并由测试用例自行注入数据。
- `mvn -pl application-starter -am test -DskipTests=false -Dtest='*EnvTests' -Dtest.groups=env-app -Dsurefire.failIfNoSpecifiedTests=false`
  - 用途：只运行真实应用装配链路测试；执行前必须准备所有被测外部服务配置。

说明：

- 默认不传 `test.groups` 时，应执行全量测试
- `test.groups` 和 `test.excludedGroups` 只用于筛选，不改变测试命名与职责
- 覆盖率报告应通过 Maven 生命周期自动生成，而不是依赖人工临时拼接命令
- `env-*` 命令不是快速开发默认入口；没有准备真实环境时不要运行
- `env-*` 模块命令建议同时使用 `-Dtest='*EnvTests'`，避免依赖模块中无关测试参与真实环境标签筛选

## 15. 禁止事项

- 禁止为追求覆盖率编写无语义测试
- 禁止把多个核心场景硬塞进一个测试方法
- 禁止只测成功不测失败
- 禁止测试方法命名含糊
- 禁止测试代码不遵守注释规范
- 禁止把第三方框架内部行为当作项目核心契约反复测试

新增禁止：

- 禁止无说明使用 `@Disabled`
- 禁止随意创建未经确认的测试标签
- 禁止让文档中的测试分级与 Maven 执行入口不一致

## 16. 当前阶段结论

当前项目测试规范的目标不是一次性铺满所有层，而是先守住重写阶段最关键的稳定面：

- 固定基础设施契约稳定
- 业务核心路径可验证
- 响应码与异常语义不漂移
- 启动装配至少可最小验证

后续随着 `chat-domain` 和 `infrastructure-service` 逐步落地，再按本文规则继续增量补齐测试即可。
