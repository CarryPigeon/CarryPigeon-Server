# 项目架构文档

## 1. 文档目的

本文只固化当前重写阶段已经明确的模块职责与依赖规则。

本文不负责扩展新的架构思想，不替代后续的包结构设计、领域建模设计与服务子模块设计。

## 2. 模块职责

### 2.1 `application-starter`

- 项目启动模块。
- 负责 Spring Boot 启动、Bean 装配、配置加载、环境拼装。
- 不承载核心业务规则。

### 2.2 `chat-domain`

- 项目的核心业务域模块。
- 负责项目的核心业务逻辑。
- 项目总体采用整洁架构，但 Spring Boot、Lombok 等能力在本项目中视为内部能力，可以在本模块中合理使用。

### 2.3 `infrastructure-basic`

- 定义全局性的、固定的基础设施能力。
- 包括但不限于日志、统一配置约定、通用序列化约定、基础工具支持。
- 此模块不是后期拔插点。

### 2.4 `infrastructure-service`

- 定义与外部服务相关的基础设施行为。
- 这部分允许后期拔插替换。
- 典型对象包括数据库、缓存、邮箱服务等。
- 该模块下后续按服务维度拆分子模块：
  - `{service-name}-api`
  - `{service-name}-impl`

### 2.5 `distribution`

- 后期用于打包与分发。
- 当前不纳入核心设计讨论。

## 3. 总体依赖原则

### 3.1 允许的依赖方向

- `application-starter` 可以依赖 `chat-domain`
- `application-starter` 可以依赖 `infrastructure-basic`
- `application-starter` 可以依赖各个 `infrastructure-service` 下的 `*-impl`
- `application-starter` 可以依赖各个 `infrastructure-service` 下的 `*-api`
- `chat-domain` 可以依赖 `infrastructure-basic`
- `chat-domain` 可以依赖各个 `infrastructure-service` 下的 `*-api`
- `*-impl` 可以依赖所属服务的 `*-api`
- `*-impl` 可以依赖 `infrastructure-basic`

### 3.2 明确禁止的依赖方向

- `chat-domain` 不得依赖任何 `*-impl`
- `chat-domain` 不得依赖 `application-starter`
- `infrastructure-basic` 不得依赖 `chat-domain`
- `infrastructure-basic` 不得依赖任何 `*-impl`
- `*-api` 不得依赖 `chat-domain`
- `*-api` 不得依赖 `application-starter`
- `*-api` 不得依赖其它服务的 `*-impl`
- `*-impl` 不得反向依赖 `chat-domain`
- `distribution` 不参与业务依赖设计

## 4. 装配规则

- `chat-domain` 只面向抽象编程，只依赖服务型基础设施的 `*-api`
- 具体外部实现由 `application-starter` 在启动时装配
- 可替换能力的切换点必须位于 `application-starter` 与 `*-impl`
- `infrastructure-basic` 提供全局稳定能力，不作为业务可替换点

## 5. Maven 层面的落地规则

### 5.1 根工程

- 根 `pom.xml` 负责统一版本、插件与模块编排
- 根工程必须显式纳入：
  - `application-starter`
  - `chat-domain`
  - `infrastructure-basic`
  - `infrastructure-service`

### 5.2 `application-starter`

- 只保留启动装配所需依赖
- 不再依赖旧重构前的 `api`、`common`、`dao` 等历史模块
- 当前阶段允许直接依赖：
  - `chat-domain`
  - `infrastructure-basic`
- 当 `infrastructure-service` 形成正式子模块后，改为依赖具体 `*-impl`，而不是依赖服务容器模块本身

### 5.3 `chat-domain`

- 当前阶段只保留业务域直接需要的依赖
- 后续如需要外部服务抽象，应依赖具体 `*-api`
- 在 `*-api` 子模块未建立前，不引入面向未来的占位错误依赖

### 5.4 `infrastructure-service`

- 当前作为服务型基础设施的承载目录
- 后续以父模块或聚合模块形式管理各服务子模块
- 子模块命名规则固定为：
  - `{service-name}-api`
  - `{service-name}-impl`

### 5.5 `infrastructure-service` 子模块规范

- `infrastructure-service` 自身不承载具体服务实现代码
- `infrastructure-service` 作为父模块或聚合模块存在
- `infrastructure-service` 的 `packaging` 应为 `pom`
- 具体外部服务能力全部下沉到其子模块中

### 5.6 `*-api` 子模块规则

- `*-api` 用于定义某个外部服务能力的抽象接口、数据契约、必要的领域无关模型
- `*-api` 的设计目标是让 `chat-domain` 面向抽象编程
- `*-api` 不承载具体外部中间件接入逻辑
- `*-api` 可以被 `chat-domain` 依赖
- `*-api` 可以被所属 `*-impl` 依赖
- `*-api` 不得依赖任何 `*-impl`
- `*-api` 不得依赖 `application-starter`
- `*-api` 不得反向承载业务域逻辑

### 5.7 `*-impl` 子模块规则

- `*-impl` 用于实现所属 `*-api` 中定义的抽象能力
- `*-impl` 负责具体中间件、外部系统、客户端 SDK、数据库框架等接入
- `*-impl` 只能实现对应服务的能力，不承载核心业务规则
- `*-impl` 可以依赖所属 `*-api`
- `*-impl` 可以依赖 `infrastructure-basic`
- `*-impl` 可以依赖具体外部框架与第三方客户端
- `*-impl` 由 `application-starter` 负责装配进运行时
- `*-impl` 不得被 `chat-domain` 直接依赖

### 5.8 子模块命名与目录规则

- 每个可替换外部服务占用一个独立服务名
- 每个服务名下固定拆为两个 Maven 子模块：
  - `{service-name}-api`
  - `{service-name}-impl`
- 推荐同名服务的两个子模块保持相邻声明与相邻目录层级
- 未开始实现的服务不预创建空子模块

### 5.9 后续新增服务时的接入顺序

- 先定义 `{service-name}-api`
- 再实现 `{service-name}-impl`
- 再由 `application-starter` 引入并完成装配
- 最后由 `chat-domain` 只依赖对应的 `*-api`

## 6. 当前已确认的设计边界

- 这是一次重写式重构，不以兼容旧模块目录为目标
- 模块职责优先于旧代码遗留结构
- 不主动引入未获许可的新架构形态
- 若后续需要补充事件、仓储、适配器、端口等更细设计，必须在本规则之上继续展开，而不是修改本规则
