# 项目架构文档

## 1. 文档目的

本文只固化当前重写阶段已经明确的模块职责与依赖规则。

本文不负责扩展新的架构思想，不替代后续的包结构设计、领域建模设计与服务子模块设计。

## 2. 模块职责

### 2.1 `application-starter`

- 项目启动模块。
- 负责 Spring Boot 启动、Bean 装配、配置加载、环境拼装。
- 负责执行运行前初始化检查，但只负责编排，不直接承载具体外部服务检查实现。
- 不承载核心业务规则。
- 不应长期承载领域仓储适配实现、独立外部运行时宿主等可迁移职责。

### 2.2 `chat-domain`

- 项目的核心业务域模块。
- 负责项目的核心业务逻辑。
- 项目总体采用整洁架构，但 Spring Boot、Lombok 等能力在本项目中视为内部能力，可以在本模块中合理使用。
- 在当前阶段，若某个 feature 已直接依赖稳定的 `*-api` 契约，则可在 `chat-domain` 内通过 Spring 装配创建面向领域语义的仓储适配器或 feature 运行时装配 Bean，但不得直接依赖任何 `*-impl`。

### 2.3 `infrastructure-basic`

- 定义全局性的、固定的基础设施能力。
- 包括但不限于日志、统一配置约定、通用序列化约定、基础工具支持。
- 此模块不是后期拔插点。

### 2.4 `infrastructure-service`

- 定义与外部服务相关的基础设施行为。
- 这部分允许后期拔插替换。
- 典型对象包括数据库、缓存、邮箱服务等。
- 该模块下后续按服务维度拆分子模块：
  - `{service-name}-api`
  - `{service-name}-impl`

### 2.5 `distribution`

- 用于打包与分发。
- 不参与业务依赖设计，但属于正式交付链路的一部分。
- 当前交付模式采用 thin jar + libs 目录结构，由 `application-starter` 作为启动入口，内部模块与第三方依赖以独立 jar 形式分发。

## 3. 总体依赖原则

### 3.1 允许的依赖方向

- `application-starter` 可以依赖 `chat-domain`
- `application-starter` 可以依赖 `infrastructure-basic`
- `application-starter` 可以依赖各个 `infrastructure-service` 下的 `*-impl`
- `application-starter` 可以依赖各个 `infrastructure-service` 下的 `*-api`
- `chat-domain` 可以依赖 `infrastructure-basic`
- `chat-domain` 可以依赖各个 `infrastructure-service` 下的 `*-api`
- `*-impl` 可以依赖所属服务的 `*-api`
- `*-impl` 可以依赖 `infrastructure-basic`

### 3.2 明确禁止的依赖方向

- `chat-domain` 不得依赖任何 `*-impl`
- `chat-domain` 不得依赖 `application-starter`
- `infrastructure-basic` 不得依赖 `chat-domain`
- `infrastructure-basic` 不得依赖任何 `*-impl`
- `*-api` 不得依赖 `chat-domain`
- `*-api` 不得依赖 `application-starter`
- `*-api` 不得依赖其它服务的 `*-impl`
- `*-impl` 不得反向依赖 `chat-domain`
- `distribution` 不参与业务依赖设计

## 4. 装配规则

- `chat-domain` 只面向抽象编程，只依赖服务型基础设施的 `*-api`
- 具体外部实现由 `application-starter` 在启动时装配
- 可替换能力的切换点必须位于 `application-starter` 与 `*-impl`
- `infrastructure-basic` 提供全局稳定能力，不作为业务可替换点

补充说明：

- 启动前初始化检查属于固定运行时编排能力，可由 `application-starter` 统一执行。
- 初始化检查的共享契约与执行器可放在共享模块中，具体检查实现仍留在对应基础设施模块。
- 必需的初始化检查必须在任何 `SmartLifecycle` 类型的独立监听器启动前完成，避免在依赖未验证前提前对外接流量。
- `application-starter` 不应继续承载 feature 级仓储适配器与 feature 级运行时宿主；若 `chat-domain` 已依赖稳定的 `*-api`，则这类 feature 级装配可下沉到对应 feature 的 `config` 或 `support` 中。
- `application-starter` 仍负责最终启动、组件扫描和运行前初始化检查编排，但不再承担 auth/user 仓储装配与 realtime 运行时等 feature 内部装配职责。

## 5. Maven 层面的落地规则

### 5.1 根工程

- 根 `pom.xml` 负责统一版本、插件与模块编排
- 根工程必须显式纳入：
  - `application-starter`
  - `chat-domain`
  - `infrastructure-basic`
  - `infrastructure-service`
  - `distribution`

### 5.2 `application-starter`

- 只保留启动装配所需依赖
- 不再依赖旧重构前的 `api`、`common`、`dao` 等历史模块
- 当前阶段允许直接依赖：
  - `chat-domain`
  - `infrastructure-basic`
- 当 `infrastructure-service` 形成正式子模块后，改为依赖具体 `*-impl`，而不是依赖服务容器模块本身

### 5.3 `chat-domain`

- 当前阶段只保留业务域直接需要的依赖
- 后续如需要外部服务抽象，应依赖具体 `*-api`
- 在 `*-api` 子模块未建立前，不引入面向未来的占位错误依赖
- 若某个业务 feature 已稳定依赖某个 `*-api`，则允许在 `chat-domain` 内通过 Spring 装配创建面向业务语义的仓储适配 Bean；该适配仍必须通过 `*-api` 完成实际读写，不能直接接入 `*-impl`

### 5.4 `infrastructure-service`

- 当前作为服务型基础设施的承载目录
- 后续以父模块或聚合模块形式管理各服务子模块
- 子模块命名规则固定为：
  - `{service-name}-api`
  - `{service-name}-impl`

### 5.5 `infrastructure-service` 子模块规范

- `infrastructure-service` 自身不承载具体服务实现代码
- `infrastructure-service` 作为父模块或聚合模块存在
- `infrastructure-service` 的 `packaging` 应为 `pom`
- 具体外部服务能力全部下沉到其子模块中

### 5.6 `*-api` 子模块规则

- `*-api` 用于定义某个外部服务能力的抽象接口、数据契约、必要的领域无关模型
- `*-api` 的设计目标是让 `chat-domain` 面向抽象编程
- `*-api` 不承载具体外部中间件接入逻辑
- `*-api` 可以被 `chat-domain` 依赖
- `*-api` 可以被所属 `*-impl` 依赖
- `*-api` 不得依赖任何 `*-impl`
- `*-api` 不得依赖 `application-starter`
- `*-api` 不得反向承载业务域逻辑

补充约束：

- `database-api` 这类服务型 `*-api` 可以承载数据库交互所必需的最小 service 抽象、record/model、事务抽象和健康抽象。
- 但 `database-api` 不应继续膨胀成第二套 feature 领域模型或第二套协议模型。
- 若某项新增契约主要表达业务决策、协议展示语义、前端消费语义或与数据库交互无关的复杂场景概念，应优先留在 `chat-domain`，而不是继续放大 `database-api`。
- `database-api` 中如果出现 `default` 方法或静默 fallback，它们本身就属于 public contract，需要通过行为测试锁定其语义；不应仅靠文档约定来判断其安全性。

数据库契约拆分策略：

- 不要求所有 `database-api` 契约一律做读写分离。
- 只有当某个契约已经明显同时承担写侧、查询侧、搜索侧、列表侧或展示投影侧多种语义时，才优先考虑做读写分离或投影分离。
- 若某个契约当前只承载少量、单一、边界清晰的数据库交互能力（例如简单按 ID 查询、简单写入、简单撤销），则不应为了形式统一而强行拆分。
- 架构一致性优先于局部简洁，但“架构一致性”不等于机械地把所有契约都拆成相同层数；应以降低长期语义膨胀和边界漂移为准。

### 5.7 `*-impl` 子模块规则

- `*-impl` 用于实现所属 `*-api` 中定义的抽象能力
- `*-impl` 负责具体中间件、外部系统、客户端 SDK、数据库框架等接入
- `*-impl` 只能实现对应服务的能力，不承载核心业务规则
- `*-impl` 可以依赖所属 `*-api`
- `*-impl` 可以依赖 `infrastructure-basic`
- `*-impl` 可以依赖具体外部框架与第三方客户端
- `*-impl` 由 `application-starter` 负责装配进运行时
- `*-impl` 不得被 `chat-domain` 直接依赖
- `database-impl` 这类 `*-impl` 可按 `config` / `jdbc` / `health` / `startup` / `transaction` / `mybatis` 等职责进一步分层；对外可以保留一个稳定的聚合自动配置入口，但内部装配可以按 feature 拆分成更小的配置类。
- `database-impl` 中的 service 层应保持薄适配器职责，允许局部统一执行/异常包装模板，但不应为形式一致而强行引入共享父类或放大抽象层。

### 5.8 子模块命名与目录规则

- 每个可替换外部服务占用一个独立服务名
- 每个服务名下固定拆为两个 Maven 子模块：
  - `{service-name}-api`
  - `{service-name}-impl`
- 推荐同名服务的两个子模块保持相邻声明与相邻目录层级
- 未开始实现的服务不预创建空子模块

### 5.9 后续新增服务时的接入顺序

- 先定义 `{service-name}-api`
- 再实现 `{service-name}-impl`
- 再由 `application-starter` 引入并完成装配
- 最后由 `chat-domain` 只依赖对应的 `*-api`

## 6. 当前已确认的设计边界

- 这是一次重写式重构，不以兼容旧模块目录为目标
- 模块职责优先于旧代码遗留结构
- 不主动引入未获许可的新架构形态
- 若后续需要补充事件、仓储、适配器、端口等更细设计，必须在本规则之上继续展开，而不是修改本规则
- 当前 `application-starter` 中遗留的领域仓储适配类与 realtime 运行时代码不应被视为最终稳态设计；当前确认的收敛方向是：仓储适配与 feature 运行时装配优先回到 `chat-domain` 内部，在不引入 `*-impl` 依赖的前提下完成 starter 瘦身

## 7. 当前运行时边界补充

### 7.1 事务与实时广播边界

- 当前项目没有统一的 outbox 或持久化事件补偿系统。
- 当前项目已通过 `TransactionRunner` 提供最小的事务后副作用登记能力，供应用服务在事务体内登记“仅在提交成功后执行”的广播动作。
- `message` feature 中，与消息主链路直接相关的 realtime 广播当前已在事务成功之后执行，包括：
  - `message.created`
  - `message.updated`
  - `message.deleted`
  - `message.pinned`
  - `message.unpinned`
  - `mention.created`
- `channel` feature 中当前已有 realtime 广播也已接入事务后执行语义，包括：
  - `channels.changed`
  - `channel.changed`
  - `read_state.updated`
- `channel` 中的 invite / accept invite / create application / decide application 等治理链路，当前也已统一通过 `TransactionRunner.afterCommit` 登记刷新广播，不再在事务未提交时直接投递 realtime 事件。
- 因此，当前单进程内的主要 realtime 广播已具备“提交成功后才发送”的一致性边界，可避免回滚后幽灵事件。
- 但这仍不等于跨实例、跨进程崩溃恢复意义上的全局一致性；若事务提交成功后进程在广播前崩溃，当前实现仍缺少持久化补偿能力。

### 7.1.1 feature 适配结论

- `channel` 与 `message` 当前已作为 `数据库事务 + afterCommit 副作用` 的标准 feature 落地。
- 这两个 feature 的应用服务负责先完成数据库内状态变更，再在事务体内登记 realtime 广播，由 `TransactionRunner` 保证只在提交成功后执行。
- `auth` 与 `user` 当前同样适合继续统一使用 `TransactionRunner`，但目前只需要普通数据库事务，不需要强行迁移到 `afterCommit` 写法。
- 原因是 `auth` 与 `user` 当前写路径只涉及数据库内多写一致性，还没有必须依赖“提交成功后再发送”的 realtime、通知、审计投递或跨模块事件副作用。
- `file` feature 当前不应直接套用该 after-commit 模型。其核心副作用是对象存储写入与下载授权，不是单纯数据库提交后的本地副作用，不能误表述为与数据库事务具备同构原子性。
- `server` feature 当前主要承载 realtime 会话、事件投递与恢复能力，本身不拥有业务写事务边界。它应消费 `channel` / `message` 提交后的结果，而不是反向定义业务事务模型。
- 验证码、邮件、对象存储及其它外部服务调用，也不应因为接入 `afterCommit` 就被视为已获得全局一致性保证。

### 7.1.2 `auth` / `user` 迁移到 after-commit 的触发条件

- 只有当 `auth` / `user` 出现明确的“数据库提交成功后才允许执行”的副作用时，才应迁移到 `runInTransaction(afterCommit -> ...)` 形式。
- 当前可接受的典型触发条件包括：
  - 注册成功后需要投递 welcome realtime 事件
  - 用户资料更新后需要广播 profile changed realtime 事件
  - 账户状态或资料状态变化后需要写入异步审计投递、通知投递或缓存失效事件
  - 某项副作用若在事务回滚时执行会产生用户可见脏状态或幽灵事件
- 一旦触发迁移，应用服务应遵循与 `channel` / `message` 相同的模式：
  - 事务内只完成数据库状态变更与提交后所需快照采集
  - 提交后副作用统一通过 `afterCommit` 注册
  - 不在事务体内直接执行依赖“提交已成功”前提的 realtime、通知或对外投递逻辑
- 若未来副作用需要跨进程恢复、失败重放或多节点一致投递，则应另行设计 outbox / 补偿机制，而不是继续放大当前最小 after-commit 契约。

### 7.2 discovery 与 realtime 公开语义

- `/api/server` discovery 当前返回的 `wsUrl` 来自 `cp.chat.server.realtime.host`、`port`、`path` 配置拼装。
- discovery 当前固定声明最小 capability 布尔值，不根据 realtime 监听器是否启用、端口是否已绑定或当前节点是否可接入做动态降级。
- `cp.chat.server.realtime.enabled=false` 当前只控制 Netty realtime 运行时是否启动，不会抑制 discovery 中的 `wsUrl` 与 capability 输出。
- 当 realtime 未装配时，message / channel realtime 发布器会退化为空实现，保证主业务链路仍可运行。

### 7.3 system channel 持久化基线

- 当前业务语义假定存在单个 canonical `system` channel。
- `system` channel 的初始化责任当前由 Flyway 迁移承担，而不是由注册流程懒创建。
- 注册与 system 频道查询流程都会要求 `findSystemChannel()` 能命中记录；缺失时按内部问题中断。
- 当前数据库迁移只保证“缺失时插入一条 system 频道”，尚未建立数据库级唯一约束来硬性防止多条 `type='system'` 记录。
- 因此，system channel 的“唯一性”在当前阶段仍是业务语义与迁移基线约束，不是数据库强约束。
