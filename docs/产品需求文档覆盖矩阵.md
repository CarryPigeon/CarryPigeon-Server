# 产品需求文档覆盖矩阵

## 1. 文档目的

本文用于把 `docs/产品需求文档.md` 中与当前后端实现直接相关的需求，映射到仓库中的实际能力、测试与任务证据，帮助后续做排期、验收和差距分析。

## 2. 评估规则

- **Implemented**：当前仓库已有稳定后端能力、接口或规则落地，并有代码/测试证据支撑。
- **Partial**：已有部分实现或基础骨架，但还未达到 PRD 所描述的完整能力边界。
- **Not Implemented**：PRD 已明确要求，但当前仓库中没有对应的实际后端落地。
- **Out of Current Scope**：PRD 明确属于后续阶段、非当前范围，或当前阶段明确不做。

说明：

- 本矩阵只评估**当前后端仓库**，不把客户端、前端、插件生态、商业化能力等误判为已实现。
- “架构预留”“文档方向”“任务意图”不等于已实现；只有代码、测试和已完成任务单才能作为实现证据。

## 4. 差距摘要

### 4.1 已实现较高覆盖的能力

- 注册 / 登录 / JWT / refresh 主链路
- 默认公共频道 + `private channel` 小型群聊
- OWNER / ADMIN / MEMBER 群治理
- invite / accept / mute / kick / ban / unban / ownership transfer
- 通用消息模型
- 文本、文件、语音消息
- 历史查询、频道内搜索
- 软撤回、撤回审计、撤回实时更新
- MinIO 文件存储与临时访问 URL

### 4.2 部分实现的能力

- 当前 PRD 主体能力已完成；后续工作主要是已实现能力的继续增强，而非缺失主链路

### 4.3 明确未实现的能力

- 当前矩阵覆盖范围内的核心后端能力已完成

## 5. 建议的下一阶段规划候选

1. **后续增强项可按产品节奏继续扩展**
   - 如更强插件运行管理、更多场景插件、内容能力增强等

## 6. 使用建议

- 这份矩阵适合作为后续 roadmap 和验收的基线。
- 后续每次完成一个 PRD 大能力后，应同步更新本矩阵，而不是只更新任务单。
- 如果某项能力只是“有架构预留”，不要把它从 `Partial` 提前升级成 `Implemented`。
