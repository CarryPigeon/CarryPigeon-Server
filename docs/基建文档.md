# 项目基建文档

## 1. 文档目的

本文只固化当前项目已经落地的基础设施能力与使用边界。

本文不扩展未确认的新基建设计，不替代架构文档、包结构规范与依赖引入规范。

## 2. 当前基建范围

当前已落地的基建重点在 `infrastructure-basic`，用于提供全局固定、长期稳定的基础设施能力。

当前已提供的能力包括：

- 统一日志基建
- 统一 ID 基建
- 统一 JSON 基建
- 统一时间基建
- 统一异常基建
- 基础自动配置与统一门面

`infrastructure-service` 当前仍只作为可拔插外部服务基建的承载目录，不在本文展开具体实现。

## 3. 模块定位

### 3.1 `infrastructure-basic`

- 负责固定基础设施能力
- 不承载业务规则
- 不承载数据库、缓存、邮件等外部服务实现
- 对上层提供稳定、简洁、可长期复用的基础能力

### 3.2 `application-starter`

- 负责装配与启动
- 提供当前最小运行配置
- 承接基础设施配置项

## 4. 当前已落地的基建能力

### 4.1 日志基建

日志实现已统一为 `log4j2`。

当前日志配置文件：

- `infrastructure-basic/src/main/resources/log4j2-spring.xml`

当前日志目录规则：

- 默认输出到 `./service-logs`
- 可通过 JVM 参数覆盖：`-Dcp.log.home=/your/path`
- 可通过环境变量覆盖：`CP_LOG_HOME=/your/path`

当前日志文件包括：

- `all.log`
- `debug.log`
- `info.log`
- `warn.log`
- `error.log`
- `error-json.log`

当前统一的日志上下文字段包括：

- `trace_id`
- `request_id`
- `route`
- `uid`

约束：

- 日志基建属于固定基础设施能力
- 业务模块不应自行复制一套日志输出规则
- 日志上下文应优先复用 `LogContexts` 与 `LogKeys`

### 4.2 ID 基建

项目统一使用雪花 ID。

当前实现：

- 基于 Hutool 雪花算法
- 对外能力由 `IdGenerator`、`Ids`、`InfrastructureBasics.ids()` 提供

当前配置项：

- `cp.infrastructure.id.worker-id`
- `cp.infrastructure.id.datacenter-id`

当前最小配置位于：

- `application-starter/src/main/resources/application.yaml`

约束：

- 整个项目的 ID 统一走该基建
- 不允许在业务代码中各自实现独立 ID 方案

### 4.3 JSON 基建

当前 JSON 能力由 `JsonProvider`、`Jsons`、`InfrastructureBasics.json()` 提供。

职责：

- 提供统一序列化入口
- 统一全局 Jackson 行为
- 避免上层模块零散创建 ObjectMapper

约束：

- 业务代码优先复用统一 JSON 能力
- 若后续需要补充模块级序列化约定，应先评估是否仍属于全局固定能力

### 4.4 时间基建

当前时间能力由 `TimeProvider`、`TimeFormats`、`InfrastructureBasics.time()` 提供。

职责：

- 统一当前时间获取
- 统一时间格式化约定
- 为后续测试替换与时间语义收敛预留稳定入口

约束：

- 时间能力属于必要的基础设施能力
- 业务代码不应在各处散落时间格式规则

### 4.5 异常基建

当前基础异常能力包括：

- `InfrastructureException`
- `InfrastructureErrorCode`

职责：

- 表达基础设施层错误语义
- 为日志、配置、ID、JSON 等基础能力提供统一异常出口

约束：

- 基础异常只表达基础设施问题
- 不承载具体业务错误语义

### 4.6 自动配置与门面

当前基础自动配置包括：

- `BasicInfrastructureAutoConfiguration`
- `IdAutoConfiguration`
- `JacksonAutoConfiguration`
- `TimeAutoConfiguration`

当前统一门面：

- `InfrastructureBasics`

门面当前对外暴露：

- `ids()`
- `time()`
- `json()`

目标：

- 让上层模块使用固定基础设施时入口更清晰
- 避免直接依赖过多底层基础类

## 5. 当前配置项

当前已确认并实际使用的配置项如下：

```yaml
spring:
  application:
    name: CarryPigeonBackend

cp:
  infrastructure:
    id:
      worker-id: 1
      datacenter-id: 1
```

补充：

- `spring.application.name` 用于定义当前运行时应用名
- `worker-id` 与 `datacenter-id` 当前取值范围均为 `0..31`
- 多节点部署时必须保证节点取值不冲突

## 6. 当前包落点

`infrastructure-basic` 当前包结构如下：

```text
team.carrypigeon.backend.infrastructure.basic
├── config
├── exception
├── id
├── json
├── logging
├── time
└── InfrastructureBasics
```

说明：

- `config`：基础自动配置与配置前缀支持
- `exception`：基础设施异常与错误码
- `id`：统一 ID 能力
- `json`：统一 JSON 能力
- `logging`：日志上下文与日志键
- `time`：统一时间能力

## 7. 使用约束

- 固定基础设施能力优先从 `infrastructure-basic` 获取
- 可替换外部服务能力不得塞入 `infrastructure-basic`
- `chat-domain` 可以依赖固定基础设施能力，但不得依赖任何外部服务实现
- `application-starter` 负责最终装配，不负责承载基础规则实现

## 8. 当前状态结论

当前基建已经形成第一版可运行骨架，重点能力已覆盖：

- 日志
- ID
- JSON
- 时间
- 异常
- 自动配置

但当前仍属于重写阶段的基建第一版，后续若新增数据库、缓存、邮件等能力，应进入 `infrastructure-service`，并继续遵守既定模块边界与依赖规则。
