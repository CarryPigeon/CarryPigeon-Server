# 项目基建文档

## 1. 文档目的

本文只固化当前项目已经落地的基础设施能力与使用边界。

本文不扩展未确认的新基建设计，不替代架构文档、包结构规范与依赖引入规范。

## 2. 当前基建范围

当前已落地的基建重点在 `infrastructure-basic`，用于提供全局固定、长期稳定的基础设施能力。

当前已提供的能力包括：

- 统一日志基建
- 统一 ID 基建
- 统一 JSON 基建
- 统一时间基建
- 统一异常基建
- 统一启动检查契约
- 基础自动配置与统一门面

`infrastructure-service` 当前仍只作为可拔插外部服务基建的承载目录，不在本文展开具体实现。

## 3. 模块定位

### 3.1 `infrastructure-basic`

- 负责固定基础设施能力
- 不承载业务规则
- 不承载数据库、缓存、邮件等外部服务实现
- 对上层提供稳定、简洁、可长期复用的基础能力

### 3.2 `application-starter`

- 负责装配与启动
- 提供当前最小运行配置
- 承接基础设施配置项
- 执行共享初始化检查
- 不承载 auth/user 仓储装配与 realtime 运行时等 feature 内部装配职责

## 4. 当前已落地的基建能力

### 4.0 AI 首轮阅读建议

如果阅读者是 AI 或第一次接触本仓库的开发者，建议优先建立以下心智模型：

1. `infrastructure-basic` 只承载**固定、全局、长期稳定**的基础设施能力。
2. 它不是数据库、缓存、对象存储等外部服务实现的容器；这些能力属于 `infrastructure-service/*-impl`。
3. 常用入口优先看：
   - `InfrastructureBasics`
   - `IdGenerator`
   - `JsonProvider`
   - `TimeProvider`
4. 启动阶段共享的初始化检查契约也属于固定编排基础设施，因此可放在 `infrastructure-basic`；但具体检查逻辑仍应留在相应 impl 模块。
5. 实际业务代码中，更常见的是直接注入具体 provider，而不是总是通过 `InfrastructureBasics` 门面访问。

### 4.1 日志基建

日志实现已统一为 `log4j2`。

当前日志配置文件：

- `infrastructure-basic/src/main/resources/log4j2-spring.xml`

当前日志目录规则：

- 默认输出到 `./service-logs`
- 可通过 JVM 参数覆盖：`-Dcp.log.home=/your/path`
- 可通过环境变量覆盖：`CP_LOG_HOME=/your/path`

当前日志文件包括：

- `all.log`
- `debug.log`
- `info.log`
- `warn.log`
- `error.log`
- `error-json.log`

当前统一的日志上下文字段包括：

- `trace_id`
- `request_id`
- `route`
- `uid`

约束：

- 日志基建属于固定基础设施能力
- 业务模块不应自行复制一套日志输出规则
- 日志上下文应优先复用 `LogContexts` 与 `LogKeys`

### 4.2 ID 基建

项目统一使用雪花 ID。

当前实现：

- 基于 Hutool 雪花算法
- 对外能力由 `IdGenerator`、`Ids`、`InfrastructureBasics.ids()` 提供

当前配置项：

- `cp.infrastructure.id.worker-id`
- `cp.infrastructure.id.datacenter-id`

当前最小配置位于：

- `application-starter/src/main/resources/application.yaml`

约束：

- 整个项目的 ID 统一走该基建
- 不允许在业务代码中各自实现独立 ID 方案

### 4.3 JSON 基建

当前 JSON 能力由 `JsonProvider`、`Jsons`、`InfrastructureBasics.json()` 提供。

职责：

- 提供统一序列化入口
- 统一全局 Jackson 行为
- 避免上层模块零散创建 ObjectMapper

约束：

- 业务代码优先复用统一 JSON 能力
- 若后续需要补充模块级序列化约定，应先评估是否仍属于全局固定能力

### 4.4 时间基建

当前时间能力由 `TimeProvider`、`TimeFormats`、`InfrastructureBasics.time()` 提供。

职责：

- 统一当前时间获取
- 统一时间格式化约定
- 为后续测试替换与时间语义收敛预留稳定入口

约束：

- 时间能力属于必要的基础设施能力
- 业务代码不应在各处散落时间格式规则

### 4.5 异常基建

当前基础异常能力包括：

- `InfrastructureException`
- `InfrastructureErrorCode`

职责：

- 表达基础设施层错误语义
- 为日志、配置、ID、JSON 等基础能力提供统一异常出口

约束：

- 基础异常只表达基础设施问题
- 不承载具体业务错误语义

### 4.6 启动检查契约

当前启动检查相关的固定共享能力包括：

- `InitializationCheck`
- `InitializationCheckResult`
- `InitializationCheckFailureException`

职责：

- 为启动层提供统一的初始化检查契约
- 让具体 impl 模块可以在不泄露自身实现细节的前提下参与启动检查
- 让 `application-starter` 只负责发现与执行检查，而不是直接耦合到数据库、缓存、存储等具体健康检查接口
- 必需检查应在独立监听器、长连接宿主或其他 `SmartLifecycle` 运行时启动前完成

边界：

- 这里只定义共享契约与失败语义
- 不在 `infrastructure-basic` 中实现数据库、缓存、对象存储等具体检查
- 具体检查实现应位于 `infrastructure-service/*-impl`

### 4.7 自动配置与门面

当前基础自动配置包括：

- `BasicInfrastructureAutoConfiguration`
- `IdAutoConfiguration`
- `JacksonAutoConfiguration`
- `TimeAutoConfiguration`

当前统一门面：

- `InfrastructureBasics`

门面当前对外暴露：

- `ids()`
- `time()`
- `json()`

目标：

- 让上层模块使用固定基础设施时入口更清晰
- 避免直接依赖过多底层基础类

补充说明：

- `InfrastructureBasics` 适合表达“当前逻辑需要一组固定基建能力”的场景。
- 若某个类只需要单一能力，例如时间、JSON 或 ID，直接注入 `TimeProvider`、`JsonProvider`、`IdGenerator` 更清晰。

## 5. 当前配置项

当前已确认并实际使用的配置项如下：

```yaml
spring:
  application:
    name: CarryPigeonBackend

cp:
  infrastructure:
    id:
      worker-id: 1
      datacenter-id: 1
```

补充：

- `spring.application.name` 用于定义当前运行时应用名
- `worker-id` 与 `datacenter-id` 当前取值范围均为 `0..31`
- 多节点部署时必须保证节点取值不冲突

## 6. 当前包落点

`infrastructure-basic` 当前包结构如下：

```text
team.carrypigeon.backend.infrastructure.basic
├── config
├── exception
├── id
├── json
├── logging
├── startup
├── time
└── InfrastructureBasics
```

说明：

- `config`：基础自动配置与配置前缀支持
- `exception`：基础设施异常与错误码
- `id`：统一 ID 能力
- `json`：统一 JSON 能力
- `logging`：日志上下文与日志键
- `startup`：启动检查共享契约与结果模型
- `time`：统一时间能力

## 7. 使用约束

- 固定基础设施能力优先从 `infrastructure-basic` 获取
- 可替换外部服务能力不得塞入 `infrastructure-basic`
- `chat-domain` 可以依赖固定基础设施能力，但不得依赖任何外部服务实现
- `application-starter` 负责最终装配，不负责承载基础规则实现
- `application-starter` 可执行共享初始化检查，但不应直接承载数据库、缓存、对象存储等检查实现
- 具体健康检查实现位于 `infrastructure-service/*-impl`，而不是 `infrastructure-basic`
- 若存在 WebSocket、Netty 等独立监听器，其默认暴露策略应保守，且握手阶段需显式鉴权，不能仅依赖 HTTP MVC 拦截器

## 8. 当前状态结论

当前基建已经形成第一版可运行骨架，重点能力已覆盖：

- 日志
- ID
- JSON
- 时间
- 异常
- 自动配置

但当前仍属于重写阶段的基建第一版，后续若新增数据库、缓存、邮件等能力，应进入 `infrastructure-service`，并继续遵守既定模块边界与依赖规则。
