# 项目部署手册

## 1. 文档目的

本文用于说明当前 CarryPigeon Backend 的交付与部署方式。

当前项目采用：

- **thin jar + libs** 的应用分发模式
- **Docker Compose** 提供外部依赖服务（MySQL / Redis / MinIO）
- **应用本身不在当前阶段容器化**，而是通过分发包或本地 Java 命令启动

本文聚焦当前已实现、已验证的运行路径，不描述未来容器化方案。

数据库初始化、迁移顺序和 MySQL 约束说明请参考：

- `docs/数据库部署手册.md`
- `docs/sql/README.md`

---

## 2. 当前部署体系概览

当前部署体系分为两层：

### 2.1 外部依赖层

由 Docker Compose 提供：

- MySQL
- Redis
- MinIO
- MinIO bucket 初始化辅助容器 `minio-init`

作用：为 `database-impl`、`cache-impl`、`storage-impl` 提供运行所需的外部服务。

### 2.2 应用运行层

由 `distribution` 模块提供 thin-jar 分发产物：

- `app/`：应用 thin jar
- `libs/`：内部模块 jar 与第三方依赖 jar
- `config/`：运行配置
- `bin/`：前台启动、后台启动、停止脚本

应用启动入口仍然是：

- `team.carrypigeon.backend.starter.ApplicationStarter`

只是运行方式从单一 fat jar 交付，转成了 **classpath 方式启动 thin jar**。

---

## 3. 适用边界

当前手册适用于：

- 本地开发环境部署
- 自建测试环境部署
- 不需要把应用本身容器化的部署场景
- 需要保留内部模块 jar 可见性与可替换性的分发场景

当前手册**不包含**：

- Kubernetes 部署
- 应用镜像构建与发布
- 多节点高可用部署编排
- 生产级进程托管方案（如 systemd、supervisor、容器编排平台）

---

## 4. 前置条件

在执行当前部署流程前，应确保：

1. 已安装 Java 21
2. 已安装 Maven（若需要本地打包）
3. 已安装 Docker 与 Docker Compose（若需要启动外部依赖）
4. 已具备当前项目运行所需的最小配置值，特别是：
   - `cp.chat.auth.jwt.secret`
   - `cp.chat.server.id`
5. 若数据库尚未初始化，应同时阅读 `docs/数据库部署手册.md`

---

## 5. 外部依赖服务启动

### 5.1 准备环境变量文件

首次使用时：

```bash
cp .env.example .env
```

或直接使用项目脚本：

```bash
bash bin/linux/docker-up.sh
```

该脚本会在 `.env` 不存在时自动从 `.env.example` 复制。

### 5.2 启动外部依赖

直接命令：

```bash
docker compose up -d
```

推荐命令：

```bash
bash bin/linux/docker-up.sh
```

MySQL 初始化细节、领域 SQL 执行方式和测试数据说明请参考：

- `docs/数据库部署手册.md`

### 5.3 停止外部依赖

```bash
bash bin/docker-down.sh
```

### 5.4 重置外部依赖与数据卷

```bash
bash bin/docker-reset.sh
```

### 5.5 查看外部依赖日志

```bash
bash bin/docker-logs.sh
```

### 5.6 当前 Docker 体系说明

当前 Docker 体系只负责**外部服务层**，不负责应用本身。

也就是说：

- Docker 启动 MySQL / Redis / MinIO
- Java 进程通过 thin-jar 分发包单独启动应用

这是一种“外部依赖容器化、应用非容器化”的混合模式。

## 6. 分发包构建

### 6.1 构建 thin-jar 分发包

推荐命令：

```bash
bash bin/linux/dist-package.sh
```

等价命令：

```bash
mvn -pl distribution -am package
```

### 6.2 产物位置

构建完成后，分发目录位于：

```text
distribution/target/full-distribution/full-distribution/
```

核心结构如下：

```text
app/
libs/
config/
bin/
```

其中：

- `app/application-starter-<version>.jar`：应用 thin jar
- `libs/*.jar`：内部模块和第三方依赖
- `config/application.yaml`：运行时主配置
- `config/log4j2-spring.xml`：日志配置
- `bin/start.sh` / `start.bat` / `start-background.sh` / `stop.sh`：运行脚本

注意：当前主分发结构以 `app/ + libs/ + config/ + bin/` 为准，thin jar 启动方式是推荐路径。

## 8. 运行参数说明

最关键的运行参数包括：

- `--cp.chat.auth.jwt.secret=...`
  - 必填
  - 当前 JWT 签名密钥，不能为空，且长度应满足项目约束

- `--cp.chat.server.id=...`
  - 推荐显式传入
  - 表示当前服务实例在消息模型中的稳定标识

你也可以通过环境变量或外部配置文件提供这些值，但当前最直接稳定的做法，是在启动命令中显式传入。

## 10. 当前 Docker / thin-jar 体系的特点

### 优点

- 外部依赖与应用运行职责清晰分离
- thin-jar 分发更利于观察与替换内部模块 jar
- 应用交付结构比 fat jar 更透明
- 本地部署与调试更灵活

### 当前限制

- 仍不是应用容器化部署体系
- 仍需要手动管理应用进程
- thin-jar 已验证最小启动链路，但完整生产依赖链路仍应在具备 MySQL / Redis / MinIO 的环境中进一步确认

---

## 11. 当前已知注意事项

1. 当前环境若没有 Docker CLI，则无法执行 `docker compose config` 或真正拉起外部依赖。
2. thin-jar 分发模式下，classpath 顺序与脚本一致性很重要，不应随意调整 `app/`、`libs/`、`bin/` 结构。
3. 若使用 `bin/linux/dist-start-background.sh` 启动，PID 与 stdout 日志位于 `distribution/target/full-distribution/full-distribution/` 下，而不是仓库根目录。
