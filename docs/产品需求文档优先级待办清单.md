# 产品需求文档优先级待办清单

## 1. 文档目的

本文基于 `docs/产品需求文档覆盖矩阵.md`，把尚未完全覆盖的 PRD 后端能力转化为可执行的优先级待办清单，用于后续迭代排期。

## 2. 使用规则

- 只纳入**当前后端仓库**相关的待办项。
- 只从覆盖矩阵中的 `Partial`、`Not Implemented` 中提炼待办。
- `Out of Current Scope` 项不进入当前待办池。
- 排序原则：**缺失核心契约 > 对外可见能力补齐 > 后续增强能力**。

## 3. 优先级分组

### P0：缺失核心契约与基础边界补齐

这些事项优先级最高，因为它们对应 PRD 中已经明确要求、但当前仍缺失或未正式化的后端契约。

#### 1. 建立正式的 public capabilities / public plugins 返回模型

- **PRD 映射**：`8.3-3`
- **当前状态**：`Not Implemented`
- **建议范围**：
  - 在公开源信息接口中正式表达 `public_plugins` / `public_capabilities`
  - 明确字段最小集合与输出边界
- **依赖关系**：依赖已完成的 `/.well-known/carrypigeon-server`
- **为什么是 P0**：
  - 公开源接口即使建出来，如果没有正式能力模型，仍然无法满足 PRD 的可发现性要求

#### 2. 完善插件治理基线

- **PRD 映射**：`7.1-7`、`10.1-3`、`10.2-5`、`13-1`、`13-2`、`13-3`
- **当前状态**：`Partial` / `Not Implemented`
- **建议范围**：
  - 从当前“消息类型插件扩展点”升级到更完整的插件边界
  - 补齐插件事件、插件权限声明、插件配置模型
- **依赖关系**：建议在事项 1 之后推进
- **为什么是 P0**：
  - 当前插件体系只有 message plugin 级别
  - 后续 `plugin/custom` 消息能力若继续推进，必须建立更稳定的插件治理边界

### P1：对外可见能力补齐

这些事项已经有部分基础，但还没有达到 PRD 所要求的完整后端能力。

#### 3. 把在线状态从内部 registry 提升为正式后端能力

- **PRD 映射**：`7.1-5`
- **当前状态**：`Partial`
- **建议范围**：
  - 基于现有 `RealtimeSessionRegistry` 提供更明确的 presence 语义
  - 视实际需求补充查询或聚合能力
- **依赖关系**：无强依赖，可在 P0 后推进
- **为什么是 P1**：
  - 内部基础已经存在
  - 当前缺口主要在“对外能力未正式化”

#### 4. 补齐“服务端隔离与公开源信息”的契约闭环

- **PRD 映射**：`8.1-2`、`8.3-2`、`9.1-2`、`15-1`
- **当前状态**：`Partial`
- **建议范围**：
  - 把现有匿名 `server summary` 能力收敛到 PRD 约定的公开源契约中
  - 明确哪些信息可公开、哪些必须隐藏
- **依赖关系**：依赖已完成的 `/.well-known/carrypigeon-server` 和事项 1
- **为什么是 P1**：
  - 当前已有匿名摘要接口，属于“能力不完整”，不是从零开始

### P2：后续增强能力

这些事项属于合理的下一阶段增强，但不应优先于上面的核心契约与边界补齐。

#### 5. 完成 `system` 频道业务流

- **PRD 映射**：`11.1-1`
- **当前状态**：`Partial`
- **建议范围**：
  - 明确 `system` 频道的正式消息与治理规则
  - 完成实际业务链路，而不只停留在类型存在
- **依赖关系**：建议在 P0 完成后推进
- **为什么是 P2**：
  - 当前 `public/private` 已经构成主业务闭环
  - `system` 频道更适合作为后续增强项

#### 6. 补齐 `system / plugin / custom` 消息链路

- **PRD 映射**：`11.4-3`
- **当前状态**：`Not Implemented`
- **建议范围**：
  - 完成这些消息类型的发送、存储、查询、实时消费与测试闭环
- **依赖关系**：
  - 依赖事项 3（插件治理基线）
  - `system` 消息建议在事项 6 明确后推进
- **为什么是 P2**：
  - 这是能力增强项
  - 若在插件治理规则不稳定前推进，容易把临时实现固化成长期包袱

## 4. 建议执行顺序

建议按以下顺序推进：

1. `/.well-known/carrypigeon-server`
2. public capabilities / public plugins 正式模型
3. 插件治理基线（事件、权限声明、配置模型）
4. 在线状态 presence 能力正式化
5. 服务端公开源与隔离契约闭环
6. `system` 频道业务流
7. `system / plugin / custom` 消息链路

## 5. 排序理由

- **P0 优先**：这些是 PRD 中已经明确要求、但当前仍缺失的核心后端契约。
- `/.well-known/carrypigeon-server` 已完成，因此当前 P0 起点前移到公开能力模型与插件治理边界。
- **先补契约，再补增强**：否则后续功能会建立在不完整的基础上，导致返工成本更高。
- **插件相关不应抢跑**：在插件权限和事件边界未稳定前，不宜继续扩展更多插件型消息能力。
- **system/custom 类能力靠后**：它们重要，但不如公开源信息和插件治理基线那样阻塞整体产品边界清晰度。

## 6. 使用建议

- 这份文档适合作为下一阶段排期输入，而不是替代覆盖矩阵。
- 覆盖矩阵解决“现状是什么”，本清单解决“接下来先做什么”。
- 后续每完成一个 P0/P1/P2 事项，应同步更新：
  - `docs/产品需求文档覆盖矩阵.md`
  - 本文档的优先级状态和顺序
