## 1. 文档目的

本文用于定义 CarryPigeon 的产品定位、目标用户、核心能力、典型应用场景、阶段目标与非目标。

本文当前作为产品草案，服务于后续架构拆解、模块设计、插件体系设计与开发排期。

## 2. 产品名称

- 产品名：CarryPigeon
- 产品形态：面向特定场景的分布式聊天服务平台

## 3. 产品定位

CarryPigeon 不是一个以“通用社交聊天”为目标、直接与 QQ / 微信竞争的即时通讯产品。

CarryPigeon 的定位是：

- 提供分布式、可扩展、可定制的聊天服务基础平台
- 为特定场景的聊天服务提供者提供开箱可用的聊天底座
- 通过多层插件系统让服务提供者专注于业务能力，而不是重复实现聊天内核
- 降低从 0 自研聊天系统，或大规模改造开源 IM 项目的成本与复杂度
- 支持一个客户端同时连接多个独立服务端，让用户在同一客户端中访问多个垂直聊天服务

一句话定义：

> CarryPigeon 是一个面向垂直场景、支持插件化扩展的分布式聊天服务平台。

## 4. 核心价值主张

CarryPigeon 要解决的问题不是“让更多人聊天”，而是：

- 让特定场景快速拥有自己的聊天能力
- 让聊天能力和业务能力深度融合
- 让服务提供方以插件方式扩展能力，而不是改内核
- 让部署、权限、安全、接入和扩展成本更低

核心价值包括：

1. 轻量接入
- 服务提供者无需从头实现用户、会话、消息、权限、存储、通知等基础能力

2. 快速扩展
- 通过插件市场、插件运行时、插件 API、配置体系快速挂载业务能力

3. 安全可控
- 通过权限模型、插件隔离、审计日志、限流与资源约束降低扩展失控风险

4. 面向垂直场景优化
- 能够围绕具体行业或群体做深度能力，而不是只停留在“发消息”

5. 分布式可演进
- 能支持后续多节点部署、服务拆分、插件分层与外部系统集成

6. 多服务端聚合体验
- 用户不应因为加入多个垂直聊天服务而安装多个客户端
- 一个客户端应能同时连接多个独立服务端，并在客户端侧完成服务切换、消息聚合与通知归并

## 5. 目标用户

### 5.1 直接用户

- 最终聊天参与者
- 群成员
- 社区成员
- 企业内部成员
- 特定项目协作者

### 5.2 购买者 / 决策者

- 社区运营者
- 服务器管理员
- 企业内部系统负责人
- 学校 / 班级管理方
- 开发团队负责人

### 5.3 平台构建者

- 聊天服务提供者
- 插件开发者
- 平台管理员
- 部署运维人员

## 6. 典型应用场景

### 6.1 MC 服务器社区

目标：

- 让 Minecraft 服务器管理和群聊深度融合

典型能力：

- 侧边栏嵌入服务器状态
- 实时展示在线玩家信息
- 游戏群聊与平台群聊双向转发
- 转发消息保留真实用户标识，而不是以机器人身份发言
- 提供按钮式管理，而不是让用户输入文本命令
- 区分服务器 OP 标记和平台管理员标记
- 提供服务器事件通知，如开服、崩溃、白名单变更、玩家上线/下线

### 6.2 诗词歌赋 / 长文讨论社区

目标：

- 让高质量内容在群聊中可沉淀、可检索、可互动

典型能力：

- 对诗歌、散文、长文提供格式化预览和展开渲染
- 对文章进行永久保存
- 提供文章快捷搜索和标签检索
- 支持评论、回复、引用
- 支持评分、推荐、收藏
- 支持作者主页、作品归档、活动专题

### 6.3 学校 / 班级管理类企业内部场景

目标：

- 把班级沟通与教学管理自动化整合

典型能力：

- 老师上传文件后自动按学科、班级、时间分类
- 自动整理作业、试卷、教案
- 允许老师在聊天界面远程触发下载、分发、通知操作
- 为家长、学生、老师区分不同权限和视图
- 提供值班提醒、作业提醒、考试通知

### 6.4 开发团队协作群

目标：

- 把项目状态、代码平台、进度提醒整合进聊天协作流

典型能力：

- 接入 GitHub / GitLab / CI 信息
- 提供项目进度实时提醒
- 重要 issue、PR、发布状态推送
- 支持按按钮快速处理常见协作动作
- 支持知识条目沉淀、搜索和引用

### 6.5 更多垂直场景

- 游戏公会
- 小型社区论坛群
- 企业项目群
- 社团活动群
- 客服协作群
- 内容创作协作群

结论：

CarryPigeon 的核心不是某一个特定行业，而是“聊天 + 垂直业务扩展”这一能力模型。
