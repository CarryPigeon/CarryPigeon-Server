任务名称：

测试治理第二阶段：全仓测试分级扩展

任务目标：

在第一阶段已建立测试治理规则、Maven 标签筛选与覆盖率门禁的基础上，继续推进全仓现有测试的分级落地，使仓库内绝大多数可明确归类的测试类都具备稳定的 `@Tag` 主分级，并形成明确的暂缓边界与验证证据。

任务背景：

第一阶段已经完成测试分级规则、状态标记规则、执行入口与覆盖率门禁的正式落地，也完成了四个代表性试点测试的标签接入与验证。当前剩余的主要问题不在于“规则不存在”，而在于“全仓大多数测试尚未完成分级补齐”，这会削弱标签筛选、测试治理统计与后续分层执行价值，因此需要进入第二阶段做仓库级扩展。

影响模块：

- 各模块 `src/test/java/**/*Tests.java`
- `ai-agent-workplace/`

允许修改范围：

- 允许为现有测试类补充 `@Tag("unit"|"contract"|"smoke"|"regression")`。
- 允许在必要时补充少量说明注释，使分级与测试边界更一致。
- 允许维护本轮实施任务单，并在完成后关闭为 `done`。

禁止修改范围：

- 不修改业务逻辑与模块架构。
- 不为了便于分级而重构测试行为本身。
- 不新增新的测试分级标签。
- 不把边界明显模糊的测试强行打错标签，只能记录暂缓理由。

依赖限制：

- 继续使用现有 JUnit 5 / Surefire / JaCoCo 方案。
- 不新增测试框架和测试依赖。

配置限制：

- 本阶段默认不继续改 `pom.xml`，除非发现本阶段分级扩展的真实阻塞。

文档依据：

- `AGENTS.md`
- `docs/AI协作开发规范.md`
- `docs/变更审核清单.md`
- `docs/任务单模板.md`
- `docs/测试规范.md`
- `docs/注释规范.md`

任务分解 / 执行计划：

1. 盘点全仓现有测试类，给出分级映射和模糊边界列表。
2. 对可明确归类的测试类批量补充类级 `@Tag`。
3. 对暂缓纳入的测试类记录原因，避免误分级。
4. 运行各分级命令与全量 `verify`，确认扩展后仍稳定。
5. 进行深度自检与 Oracle 复审。
6. 更新任务单实际结果、验证证据和残留风险，并改为 `done`。

关键假设与依赖：

- 现有测试的命名、注释与结构已足够支撑大多数测试的直接分级。
- 少量复杂跨层或边界混合测试可能需要暂缓，不应为追求“全量标签化”牺牲分级准确性。

实现要求：

- 优先保证标签准确性，而不是标签覆盖率表面最大化。
- 每个测试类只保留一个主分级。
- 不改变既有测试注释、命名和断言风格。

测试要求：

- 分级扩展后，各主分级应至少有多类真实测试可被筛选执行。
- 新增标签不得引入编译错误或测试行为漂移。

质量门禁：

- 本轮纳入范围的测试类均完成准确分级或明确暂缓说明。
- 分级命令可实际筛选并运行测试。
- `mvn verify -DskipTests=false` 通过。

复审要求：

- 完成后进行深度自检与 Oracle 复审。

文档要求：

- 本阶段原则上不新增长期规则；若发现第一阶段文档存在缺漏，再补充。

验收标准：

- 全仓现有测试类已完成大范围分级扩展。
- 模糊边界测试类有明确暂缓理由。
- 分级命令与全量 `verify` 通过。
- 任务单记录完整证据并关闭为 `done`。

完成定义：

- 所有计划项完成并复验通过。
- 任务单改名为 `done`。

实际结果：

- 已完成全仓 `*Tests.java` 盘点，共识别 54 个测试类。
- 已在第一阶段 4 个试点标签基础上，为第二阶段额外 46 个可明确归类的测试类补充主分级 `@Tag`。
- 当前全仓仍保留 4 个未打标签测试类，作为本阶段明确暂缓项：
  - `chat-domain/.../features/message/application/service/MessageApplicationServiceTests.java`
  - `chat-domain/.../features/message/controller/http/ChannelMessageControllerTests.java`
  - `chat-domain/.../features/server/config/RealtimeServerConfigurationTests.java`
  - `chat-domain/.../features/server/controller/ws/RealtimeChannelHandlerTests.java`
- 暂缓原因：这些类分别混合了跨用例应用服务编排、controller 协议与附件链路、配置装配与运行时内部断言、以及 WebSocket/Netty 实时编排；若强行打单一主标签，误导风险高于收益。

验证记录：

- `mvn test -DskipTests=false -Dtest.groups=unit` -> 通过。
- `mvn test -DskipTests=false -Dtest.groups=contract` -> 通过。
- `mvn test -DskipTests=false -Dtest.groups=smoke` -> 通过。
- `mvn test -DskipTests=false -Dtest.groups=regression` -> 通过。
- `mvn verify -DskipTests=false` -> 通过。
- 第二阶段中途曾因错误批量改写引入测试类损坏；已定位为我方变更造成的问题，并已先恢复相关文件，再采用安全脚本重新注入 `import Tag` 与类级 `@Tag`，之后上述全部命令重新通过。
- 复核统计：全仓 54 个 `*Tests.java` 中，50 个已完成主分级标签，4 个明确暂缓；不存在隐性漏标类。
- Oracle 复审结论：第二阶段从测试治理角度可以安全关闭为 `done`，4 个暂缓类属于显式边界决策而非隐藏未完成项。

残留风险：

- 暂缓类若后续继续膨胀，将进一步降低测试分级准确性；更合适的长期方案是按边界拆分测试类后再打标签。
- `ChannelMessageControllerTests` 相比其余 3 个暂缓类更接近 `contract`，可作为后续最优先复核对象，但不构成第二阶段阻塞。
