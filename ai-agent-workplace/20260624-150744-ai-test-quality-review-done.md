任务名称：
测试方法质量逐模块审查

任务目标：
以只读审查方式评估项目各模块测试方法的质量，判断当前测试是否达到成熟项目标准，并输出逐模块的结论、问题清单、证据与改进优先级。

任务背景：
用户要求下一步对测试方法质量进行逐模块审查，并判断是否达到成熟项目标准。本轮任务不改代码，重点审查测试命名、断言强度、边界覆盖、失败路径、脆弱性与可维护性。

任务类型：
只读探索 / 审查任务

影响模块：
- `application-starter`
- `chat-domain`
- `infrastructure-basic`
- `infrastructure-service/*`
- `docs/测试规范.md`
- `ai-agent-workplace/`

允许修改范围：
- 允许新增本次审查任务单
- 默认不修改正式代码、测试、配置、文档

禁止修改范围：
- 不在本轮直接修复测试问题
- 不把只读审查结论表述为“已完成整改”

依赖限制：
- 不新增依赖

配置限制：
- 不新增配置

文档依据：
- `AGENTS.md`
- `docs/AI协作开发规范.md`
- `docs/变更审核清单.md`
- `docs/任务单模板.md`
- `docs/测试规范.md`
- `docs/注释规范.md`

任务分解 / 执行计划：
1. 统计各模块测试文件分布与分级形态。
2. 按模块抽样阅读核心测试类与测试方法。
3. 从成熟项目视角审查测试方法质量：
   - 命名是否可读、稳定、可检索
   - 断言是否验证真实契约
   - 成功/失败路径覆盖是否完整
   - 是否过度依赖实现细节
   - 是否存在脆弱、冗余或低信号测试
4. 形成逐模块审查结论与阻塞/非阻塞问题。

关键假设与依赖：
- “成熟项目标准”按本仓库测试规范与常规工程质量标准综合判断，而不是按覆盖率单一指标判断。
- 本轮为只读审查，不运行测试也可形成结论，但若引用未运行的判断会明确说明依据来自源码证据。

实现要求：
- 审查结论必须引用仓库内测试文件作为证据。
- 若某模块无明显问题，也要说明剩余风险与盲区。

测试要求：
- 本任务为只读探索 / 审查任务，本次无代码改动，因此不适用实现类测试门禁。

质量门禁：
- 已完成必要测试代码阅读
- 已形成逐模块结论
- 已明确本次无正式代码改动

复审要求：
- 无

文档要求：
- 本轮不改 `docs/`

验收标准：
- 给出逐模块测试质量结论
- 明确是否达到成熟项目标准
- 给出主要问题与优先级

完成定义：
- 用户收到可直接用于后续整改排期的测试审查结论

实际结果：
- 已完成逐模块测试方法质量审查
- 已按模块给出成熟度判断、主要问题与整改优先级

残留风险：
