任务名称：

测试规范深度审查与治理任务单

任务目标：

基于当前项目文档、Maven 配置与既有测试代码，形成一份可执行的测试治理任务单，明确当前测试规范现状、主要缺口、后续优化方向与分阶段落地顺序，使项目测试体系在重写阶段更加规范、有序、可审查。

任务背景：

当前项目已经建立了基础测试规范，并在多个模块中补充了较为成型的测试代码，但测试治理仍主要停留在“命名、注释、基本契约覆盖”层面。随着 `chat-domain` 与 `infrastructure-service` 持续推进，测试体系需要进一步明确分级、状态标记、覆盖率门禁、执行入口与治理节奏，避免后续测试数量增长后出现规范漂移、执行混乱、选择性跳测和质量门禁失真。

影响模块：

- `docs/测试规范.md`
- `docs/注释规范.md`
- 根 `pom.xml`
- 各模块 `src/test/java`
- `ai-agent-workplace/`

允许修改范围：

- 允许对现有测试规范进行审查并形成治理方案。
- 允许在 `ai-agent-workplace/` 中沉淀任务单与审查结论。
- 允许后续按确认方案补充测试分级、覆盖率门禁、测试注释和状态标记规范。
- 允许后续在不改变架构边界的前提下，对根 `pom.xml` 与测试文档做测试治理增强。

禁止修改范围：

- 不允许借测试治理之名修改模块职责、依赖方向与架构边界。
- 不允许为覆盖率目标编写无语义测试。
- 不允许默认引入重量级测试基础设施或真实外部服务依赖，除非后续单独获批。
- 不允许把测试支持代码散落到错误模块或跨模块安放测试。

依赖限制：

- 继续以 Spring Boot Test 与 JUnit 5 为当前测试基础。
- 若需引入额外测试依赖、覆盖率插件扩展或集成测试中间件，必须先论证必要性与归属模块。

配置限制：

- 不新增未来占位型测试配置。
- 测试执行与覆盖率门禁配置应尽量集中在根 `pom.xml` 或明确模块级归属，不制造重复配置。

文档依据：

- `AGENTS.md`

- `docs/变更审核清单.md`

任务分解 / 执行计划：

1. 阅读测试规范、注释规范、AI 协作规范、变更审核清单与任务单模板。
2. 审查根 `pom.xml` 与模块 `pom.xml` 中的测试基础设施配置。
3. 盘点各模块现有测试代码，识别测试层次、命名、注释、执行方式与明显缺口。
4. 归纳当前测试体系的已有优点、结构性问题与治理盲区。
5. 输出一份面向后续执行的测试治理优化方案，包含建议优先级与阶段顺序。
6. 将以上结论沉淀为正式任务单，供后续确认后实施。

关键假设与依赖：

- 当前项目仍处于重写阶段，测试体系应优先服务稳定契约与关键边界，而非追求形式化大而全。
- 当前仓库暂无独立 CI 工作流配置，测试门禁主要依赖本地 Maven 约束与人工执行。
- 当前不默认引入大量真实外部服务集成测试，后续以轻量、可控、可分级为原则推进。

实现要求：

- 测试治理必须匹配项目当前模块边界与重写阶段节奏。
- 优先增强“可识别、可筛选、可验证、可审查”的测试治理能力。
- 优化方案应区分短期可直接落地项与中期治理项，避免一次性扩张过度。

测试要求：

- 后续补测必须继续覆盖成功路径与失败路径。
- 测试类与测试方法继续遵守 `*Tests` 与 `methodName_condition_expected()` 约定。
- 测试注释继续明确职责、边界、输入和期望。
- 需要建立测试分级与状态标记规则，使后续测试可被机器筛选与人工审查。

质量门禁：

- 文档与代码审查结论必须有实际仓库证据支撑。
- 任务单必须明确当前现状、问题、优化方案与建议验收标准。
- 后续若落地治理改动，应至少验证 Maven 测试命令、覆盖率报告生成、测试分级筛选和相关文档同步。

复审要求：

- 后续若修改 `docs/测试规范.md`、根 `pom.xml` 或测试执行门禁，应再次做独立复审，重点关注是否引入过度治理、是否破坏当前重写阶段节奏。

补充说明：

建议的后续优化方案如下。

一、建立测试分级体系，但保持当前项目节奏不过度扩张。

- 建议先固化四级：`unit`、`contract`、`smoke`、`regression`。
- `unit`：纯对象、纯规则、纯值对象测试；默认应最快、最稳定。
- `contract`：模块内协议、控制器、异常映射、配置绑定、仓储抽象交互测试。
- `smoke`：最小启动与关键装配验证，保证核心 Bean 与关键链路不塌。
- `regression`：针对历史缺陷或高风险链路的回归验证。
- 暂不默认铺开大规模 `integration` / `e2e`，避免超出当前重写阶段边界。

二、引入测试标记规则，使测试状态可筛选、可治理。

- 对分级测试统一引入 `@Tag`，例如 `@Tag("unit")`、`@Tag("contract")`、`@Tag("smoke")`、`@Tag("regression")`。
- 对已知暂缓执行的测试，允许 `@Disabled("原因")`，但必须写清原因、恢复条件与责任边界，禁止无说明禁用。
- 对测试代码中的状态注释建立专门约束：
  - `TODO:` 表示计划补测或拆分夹具；
  - `FIXME:` 表示当前已知失真或脆弱点；
  - `NOTE:` 表示重要测试边界说明；
  - `WARNING:` 表示高风险场景或环境依赖。
- 这些标记建议仅在确有治理价值时使用，避免注释泛滥。

三、引入覆盖率报告，但不要把覆盖率当目标本身。

- 根 `pom.xml` 中已有 JaCoCo `prepare-agent`，建议下一步补齐 `report` 与 `verify`。
- 覆盖率门禁建议分两阶段：
  - 第一阶段只生成报告，不设硬阈值，用于看清盲区；
  - 第二阶段只对关键模块或关键包设最低阈值，且阈值保守。
- 阈值不建议一开始用全仓高比例统一压住，优先按模块成熟度与业务关键性区分。
- 覆盖率文档中必须明确：覆盖率只作为风险探测，不作为“测试质量已足够”的替代判断。

四、建立统一测试执行入口，让测试更有秩序。

- 建议在文档中定义最少四类执行命令：
  - 全量测试；
  - smoke 测试；
  - contract 测试；
  - regression 测试。
- 后续可通过 Surefire/Failsafe + `@Tag` 做筛选，而不是只靠类名习惯。
- 若后续确有独立集成测试需求，再引入 Failsafe 与 `*IT` / `*ITests` 规则，不要提前空转。

五、收紧测试注释与命名一致性要求。

- 当前复杂测试的类级/方法级注释已较好，但简单测试存在颗粒不一问题，建议统一：
  - 所有测试类必须有类级注释；
  - 所有测试方法至少说明当前契约意义，而不只写表面动作；
  - `@DisplayName` 在 controller、realtime、regression、smoke 这类可读性要求高的测试中建议强制；
  - 简单值对象测试可不强制 `@DisplayName`，但命名与注释必须完整。

六、对测试夹具和替身做轻量治理。

- 当前不少测试使用内联假对象与局部 fixture，这符合当前轻量策略，但在重复度明显上升的 feature 中，建议抽取模块内测试支持类，避免同类替身反复散落。
- 该抽取应服务于可读性，不应演化为难以理解的复杂测试基类体系。

七、建议的推进顺序。

- 第一优先级：回写测试分级、状态标记、覆盖率原则到 `docs/测试规范.md`。
- 第二优先级：在根 `pom.xml` 补齐 JaCoCo report/verify，并先生成报告不拦截。
- 第三优先级：选择 1 到 2 个成熟模块试点 `@Tag` 分级，例如 `infrastructure-basic` 与 `chat-domain` 的 controller / application tests。
- 第四优先级：根据试点结果再决定是否引入 Failsafe 或更细的测试层次。
