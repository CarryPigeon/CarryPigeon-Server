任务名称：
优化 swagger 示例与排序

任务目标：
进一步提升 Swagger UI 的可读性与可交付性，为关键接口补充更贴近业务的成功响应示例，并统一接口展示顺序/分组体验。

任务背景：
当前 Swagger 已具备分组、摘要、字段级 schema、共享错误示例，但成功响应示例仍偏通用，且接口展示顺序尚未显式优化。用户愿意继续把 Swagger 打磨成更像正式 API 门户的样子。

影响模块：
- `application-starter`
- `chat-domain`

允许修改范围：
- 允许微调 `OpenApiConfiguration`
- 允许为关键控制器操作补充成功响应示例或更明确的响应说明
- 允许必要时调整注解以优化接口排序/分组体验
- 允许更新任务单和必要文档说明

禁止修改范围：
- 不允许改变业务行为
- 不允许改变 HTTP 路径或响应结构
- 不允许引入新依赖

依赖限制：
- 仅使用当前已引入的 springdoc / swagger 注解能力

配置限制：
- 优先不新增运行时配置

文档依据：
- `AGENTS.md`
- `docs/API.md`
- `docs/AI协作开发规范.md`
- `docs/变更审核清单.md`

任务分解 / 执行计划：
1. 识别最值得补成功示例的关键接口。
2. 为这些接口补贴近真实业务的成功响应示例。
3. 视需要优化标签/顺序展示，让 Swagger UI 更易浏览。
4. 运行诊断与完整回归测试。
5. Oracle 复审并收尾。

关键假设与依赖：
- 当前字段级 schema 与错误示例已稳定。
- 本轮目标是“展示更好”，不是再扩协议边界。

实现要求：
- 成功示例要贴近实际返回值，不写空泛 `{}`
- 优先覆盖前端联调最常打开的接口

测试要求：
- 受影响 reactor 测试通过

质量门禁：
- Maven 完整 reactor 测试通过
- 不破坏现有 OpenAPI 文档语义

复审要求：
- 完成后需做只读复审，重点看示例是否真实、排序是否有提升

文档要求：
- 如有必要，可同步更新 `docs/API.md`

验收标准：
- Swagger 中成功示例更贴近真实业务
- 接口浏览顺序更清晰
- 验证通过

完成定义：
- 代码与必要文档修改完成
- 验证通过
- 任务单改为 `done`

实际结果：
- 在 `OpenApiConfiguration` 中为高价值接口补充了更贴近真实返回值的成功响应示例
- 成功示例覆盖认证、用户资料、频道基础查询/创建、服务基础回显与 presence 等高频联调接口
- 成功示例已对齐全局 snake_case 序列化约定，并修正 `/api/users` 列表接口与 `/api/users/page` 分页接口的差异
- 当前阶段未引入额外 OpenAPI 排序 hint，继续使用现有 `Tag` 分组与控制器内定义顺序
- 额外完成了一轮展示层润色：统一首页标题与描述、Tag 命名风格、Operation 摘要语气，使 Swagger 更接近正式 API 门户

验证记录：
- `mvn -pl "application-starter" -am -DskipTests=false test`：通过
- 优先级探索结果：认证、频道、频道消息、用户资料是最值得补成功示例的高价值操作，当前覆盖与该结论一致
- Oracle 只读复审：指出初版成功示例中 `/api/users` 被误写为分页对象，而 `presence` 的 snake_case 示例实际上正确；随后已修正并复验通过
- 最终文案润色后再次执行 `mvn -pl "application-starter" -am -DskipTests=false test`：通过
- Java LSP 初始化超时，未返回稳定结果；以 Maven 完整编译与完整 reactor 测试结果作为最终验证依据

残留风险：
- 当前成功示例主要覆盖高价值接口，尚未为所有频道治理细分操作逐个补独立成功示例
- 当前接口排序仍主要依赖 `Tag` 分组与控制器定义顺序，未额外引入 vendor-specific UI 排序配置

知识沉淀 / 是否回写 docs：
