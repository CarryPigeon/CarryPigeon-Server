任务名称：
排除邮件配置后的上线可行性深度审查

任务目标：
在不把“邮件配置缺失”视为阻断项的前提下，重新深度审查当前项目是否已经达到可直接上线的聊天服务后端基线，并给出明确的阻断项、风险项和结论。

任务背景：
用户要求基于当前最新仓库状态重新判断“是否能直接上线”，且明确排除邮件配置本身作为阻断条件。因此本轮审查重点应回到核心服务链路、鉴权安全、数据一致性、部署与可运维性，而不是 SMTP 环境细节。

影响范围：
- 全仓审查，不默认改代码
- `ai-agent-workplace` 审查记录

允许修改范围：
- 允许新增审查任务单和审查记录
- 默认不改业务代码，除非用户后续要求修复

禁止修改范围：
- 不在本轮直接实施架构调整
- 不把邮件配置缺失本身计入阻断项

审查重点：
1. 启动与部署链路是否可直接运行
2. 核心 HTTP / realtime / storage / cache / database 是否存在上线阻断
3. 认证、鉴权、文件访问、频道治理等是否还有高风险缺陷
4. 测试、回归、发布与观测能力是否满足直接上线最低要求

验收标准：
- 给出明确上线结论：可直接上线 / 不建议直接上线
- 若不可直接上线，按严重级别列出阻断项
- 说明已核查内容、验证证据与残留盲区
